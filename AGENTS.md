# AGENTS.md

本文件用于约束所有在 `D:\liuliu-minimap` 中工作的 AI 编码代理。目标不是让模型“多写点好看的 UI”，而是让它在重构时稳定遵守本项目的产品主线、微信小程序技术边界、视觉语言和验收标准。

如果用户没有明确要求另一种角色、技能或输出形式，按本文档执行。

## 项目定位

`liuliu-minimap` 是 `liuliu` City Walk 项目的原生微信小程序端。

核心体验是：

1. 探索页设置探索点。
2. 生成主题任务。
3. 单人或同行开始漫步。
4. 在记录页完成任务打卡、素材记录和轨迹追踪。
5. 在足迹页回看纪念卡册和成就。

这不是通用地图工具、社区信息流、后台管理系统，也不是营销落地页。所有前端改造都必须服务“轻松出门、观察城市、留下纪念”这个体验。

## 技术边界

- 这是原生微信小程序项目，主要代码在 `miniprogram/`。
- 页面由 `.wxml`、`.wxss`、`.js`、`.json` 组成。
- 不要把项目改成 React、Vue、uni-app、Taro、Tailwind 或其他前端框架。
- 不要引入重型 UI 库、动画库、状态管理库或构建链。
- 优先使用现有小程序能力：`wxss transition`、`wxss animation`、条件 class、已有组件拆分、现有 service/utils。
- 后端当前兼容两条链路：云开发兜底模式和 Web 后端共用接口模式。不要在前端重构时破坏 `miniprogram/utils/config.js` 控制的后端切换逻辑。
- `cloudfunctions/shared/*-runtime.js` 是共享运行时源文件。修改成就或主题生成共享逻辑后，必须同步对应云函数 runtime，不能只改某个云函数里的副本。

## 前端重构硬规则

- 不要只做视觉换皮。每次 UI 改动都必须回答：用户下一步更清楚了吗？操作更少了吗？状态更明确了吗？
- 不要新增无业务意义的装饰层、营销文案、宣传模块或解释型大段文字。
- 不要把每个区块都做成卡片。页面可以有卡片，但不能卡片套卡片、满屏漂浮卡片。
- 不要把所有按钮都做成胶囊。胶囊只适合标签、chip、状态或轻量选择项。
- 不要为每个页面、每个组件临时发明按钮样式。按钮必须先归类，再复用同类视觉规则。
- 不要用固定宽度硬塞按钮、标签或横向操作组。不同手机宽度下必须能自适应。
- 不要让相邻容器距离忽大忽小。页面、区块、卡片、按钮组、列表项必须遵守统一间距层级。
- 不要让同类页面各自排列。探索、记录、足迹、个人页中的标题区、状态区、内容区、底部行动区应保持同一套布局逻辑。
- 不要使用大面积紫色、蓝紫渐变、廉价霓虹、玻璃拟态、黑色重阴影或通用 AI 风背景。
- 不要使用纯黑大块、过亮渐变、无来源图标、随机 emoji 堆砌来制造“设计感”。
- 不要让文案替代交互。页面不应靠说明书教用户怎么用。
- 不要牺牲微信小程序真机性能。地图、列表、滚动页优先保证顺滑和稳定。
- 不要为了“代码整洁”改动主流程状态机、接口字段或数据结构，除非用户明确要求并已完成回归检查。
- 不要删除或弱化隐私、定位授权、登录、分享、删除确认等关键保护流程。

## 视觉语言

当前产品视觉是温暖、轻量、城市漫步感，不是科技后台，也不是高饱和潮流 App。

默认基调：

- 页面背景：`#f5f2ed` 或现有接近色 `#f4efe6`
- 卡片背景：`#fffdf8`
- 浅底块：`#faf4ea`、`#efe7da`
- 主文字：`#241f19`
- 次级文字：`#746b5f`
- 弱提示文字：`#9a9082`
- 主行动色：`#626247`、`#3f402f`
- 成功/进行中：`#66704a`
- 风险/错误：`#a55449`

样式规则：

- 优先复用 `miniprogram/app.wxss` 中已有的 `.page-shell`、`.section-title`、`.section-subtitle`、`.primary-btn`、`.secondary-btn`、`.ghost-btn`、`.danger-btn`、`.pill`、`.meta-pill` 等全局样式。
- 新增颜色前，先确认现有色板不能满足。新增颜色必须与温暖低饱和基调一致。
- 圆角应有层级：大容器约 `28rpx` 到 `32rpx`，内部控件约 `18rpx` 到 `26rpx`，标签可用 `999rpx`。
- 阴影必须偏暖、轻，不使用大面积黑色阴影。优先用 `rgba(77, 62, 38, 0.06-0.12)`。
- 边线使用 `1rpx solid rgba(126, 111, 89, 0.12-0.18)` 一类的低对比边界。
- 字号不要随意跳跃。标题、正文、标签应沿用现有页面层级。
- 中文长文本、地点名、任务标题、昵称、搜索结果必须处理换行或截断，不能溢出按钮或卡片。

## 按钮系统

按钮是当前前端质量最容易失控的地方。任何按钮改动都必须先判断它属于哪一类，再选择对应尺寸、颜色、状态和布局。

### 按钮类型

主按钮：

- 用途：页面最重要的下一步，例如生成主题、开始漫步、完成保存、创建房间。
- 每个可视区域原则上只允许一个最强主按钮。
- 颜色优先使用 `#626247` 或 `#3f402f`，文字使用 `#fffdf8`。
- 高度建议 `88rpx-108rpx`，不可低于 `80rpx`。
- 圆角建议 `24rpx-28rpx`。
- 宽度优先 `width: 100%`，底部行动区、表单提交、生成按钮都不应使用不稳定的固定宽度。
- 文案允许两行结构时，主文案和副文案必须分别设定行高，不得挤压。

次级按钮：

- 用途：更换地点、重新生成、查看详情、恢复登录等非最终提交动作。
- 颜色使用浅底深字，例如 `#faf4ea` + `#3f402f`，或白底内描边。
- 高度建议 `72rpx-88rpx`。
- 不要和主按钮使用同等重量阴影。

幽灵/轻按钮：

- 用途：展开、收起、刷新、复制、切换小状态。
- 应轻量，不抢主按钮注意力。
- 可以使用浅色底、低对比边线、较小圆角。
- 不要把轻按钮做成大面积实心色。

危险按钮：

- 用途：删除、退出、放弃、清空。
- 使用 `#a55449` 或红色低饱和浅底。
- 必须有确认或明确语义，不能和普通按钮只靠位置区分。

图标按钮：

- 用途：拍照、录像、录音、删除素材等短操作。
- 图标和文字排列必须固定：图标在左或上，不能同一页面混用。
- 图标尺寸建议 `28rpx-34rpx`；删除类圆形按钮建议 `44rpx-56rpx`。
- 图标按钮不能因为文字变化导致宽高跳动。

### 按钮适配规则

- 所有按钮必须设置 `box-sizing: border-box`。
- 横向按钮组优先使用 `display: flex`；每个按钮使用 `flex: 1; min-width: 0;`。
- 横向按钮组中的按钮间距用统一间隔，推荐 `12rpx`、`14rpx`、`16rpx`，不要一处 `8rpx` 一处 `48rpx`。
- 输入框 + 按钮这类复合操作行，不要把间距只写在原生 `button` 的 `margin-left` 上。微信小程序 `button` 有默认盒模型和局部样式叠加，容易出现视觉间距失效或不稳定。应由父级布局控制间距，或使用显式 spacer / grid column。
- 两个并排按钮文字较长时，优先让按钮等宽、文字换行或省略；不要把某个按钮挤到过窄。
- 三个及以上操作按钮在小屏上必须检查。若每个按钮文案超过 2 到 3 个汉字，应考虑图标 + 短文案、横向滚动或换成两行网格。
- 按钮文字默认 `line-height: 1.2-1.35`，不要使用过大的行高导致垂直不居中。
- 按钮文字不得依赖 `font-size` 自动缩放。需要短文案、换行、省略或重排。
- `button` 内部如果有 `view/text/image`，内部容器必须承担居中和溢出控制。
- 微信 `button[disabled]` 需要单独样式，不能只依赖系统默认灰色。
- Loading 状态按钮要保持原尺寸，不得因为文字从“生成”变成“正在生成”而改变宽高。
- Active 反馈只做轻微 `scale`、`translateY` 或 `opacity`，不要改变布局尺寸。

### 按钮视觉禁区

- 禁止按钮之间同色同重却代表不同危险级别。
- 禁止一个页面同时出现多个同样醒目的实心主按钮。
- 禁止按钮文字贴边，左右内边距不足。
- 禁止按钮高度在同一操作组里不一致。
- 禁止因为单个按钮文案长，就给它写死一个很大的 `margin-left` 或固定宽度。
- 禁止把标签、状态 pill、导航 tab 和提交按钮做成完全相同视觉。

## 间距与排列

这个项目必须用清晰的空间层级，不允许凭感觉给每个组件临时加 margin。

### 间距层级

推荐间距基准：

- 页面外边距：`32rpx`
- 大区块之间：`22rpx-28rpx`
- 卡片内部 padding：`24rpx-32rpx`
- 标题和说明之间：`8rpx-14rpx`
- 标题区和内容区之间：`18rpx-24rpx`
- 列表项上下 padding：`18rpx-26rpx`
- 按钮组内部间距：`12rpx-16rpx`
- chip/tag 内部 padding：横向 `14rpx-22rpx`，纵向 `8rpx-14rpx`

执行规则：

- 同一页面中，同级容器之间的垂直间距必须一致。
- 相邻卡片之间不要小于 `18rpx`，否则会显得挤。
- 一个容器内部的标题、说明、内容、按钮必须有清楚层级，不要全部平均堆叠。
- 地图、主题卡、任务卡、底部保存栏这类核心区域需要比普通列表项拥有更稳定的边界和间距。
- 不要用大 margin 修补局部错位。先检查父容器布局、flex 对齐、padding 和宽度。
- 不要同时在父容器和子容器上叠加同方向大 margin，容易造成不同页面间距失控。

### 排列规则

- 页面结构优先分为：顶部状态/标题区、核心内容区、辅助信息区、主行动区。
- 同类页面的结构顺序要一致。不要一个页面先按钮后说明，另一个页面先说明后按钮，除非业务动作不同。
- `section-title + section-subtitle + content` 是基础组合，应复用语义和间距。
- 卡片标题区如果左侧是文字、右侧是状态或按钮，左侧必须 `min-width: 0; flex: 1;`，右侧必须 `flex-shrink: 0;`。
- 列表卡片里不要把所有信息横向塞满。主信息优先纵向，状态和短 meta 放侧边或顶部。
- 横向滚动只用于媒体缩略图、少量 chip 或卡片预览，不要把核心流程按钮藏进横向滚动。
- 底部 sticky 行动栏必须预留页面底部 spacer，不能遮住最后一项内容。

## 一致性规范

每次重构都要先找同类组件，而不是直接写新样式。

### 同类组件必须统一

以下组件类型必须拥有一致的结构和视觉：

- 页面主按钮：生成、开始、保存、创建房间、完成。
- 次级操作按钮：更换、重新生成、查看、恢复、刷新。
- 选择 chip：主题方向、状态筛选、记录类型筛选、模式选择。
- 状态 pill：待出发、进行中、已完成、生成中、录音中、错误。
- 列表卡片：单人记录卡、同行记录卡、任务行、搜索结果、周边地点。
- 空状态：无记录、无搜索结果、未登录、未设置探索点。
- Loading 状态：生成中、搜索中、保存中、上传中。

### 一致性执行方法

- 新增样式前，先全局搜索已有同类 class。优先扩展已有 class，而不是复制一份改名字。
- 同类按钮必须使用同一套高度、圆角、字号、阴影和 active 反馈。
- 同类卡片必须使用同一套背景、边线、圆角和阴影。
- 同类列表项必须使用同一套 padding、分割线、标题字号和 meta 字号。
- 同类状态必须使用同一套颜色语义。不要同一个“进行中”在不同页面有不同颜色。
- 如果确实需要变体，命名要表达语义，例如 `primary-btn-large`、`status-pill-active`，不要用 `btn2`、`new-card`、`nice-style`。
- 修改一个全局组件或基础样式时，必须检查使用它的所有页面，避免局部修好、其他页面变坏。

### 页面之间的一致性

- 探索页、足迹页、个人页主背景必须统一，不要各自换背景。
- 页面标题、说明、主卡片、底部行动的垂直节奏要一致。
- 同样的“筛选区”在足迹页和其他页面出现时，chip 尺寸、active 样式、间距必须一致。
- 同样的“任务/记录卡片”在列表和详情中可以信息密度不同，但颜色、圆角、标题层级应保持亲缘关系。
- 同样的“错误/空状态”不要一处用弹窗、一处用红字、一处用大卡片，除非交互严重程度不同。

## 多机型适配

微信小程序在不同手机宽度、字体设置和系统渲染下容易出现按钮错位、文字溢出、容器过挤。任何前端重构必须主动处理。

### 必须考虑的屏幕

- 小屏窄机型：约 `320px-375px` 宽，例如 iPhone SE、旧安卓机。
- 主流宽度：约 `390px-430px` 宽，例如常见 iPhone 和安卓。
- 高屏设备：底部 sticky 区域和滚动内容要自然。
- 开启较大系统字体时，按钮和卡片不能直接崩坏。

### 适配规则

- 页面级横向 padding 使用 `32rpx` 为基准，小组件内部不要再叠加过多横向 padding 导致内容区过窄。
- 不要使用固定 `px` 宽度做主要布局。小程序页面布局优先使用 `rpx`、`%`、`flex`。
- 需要固定视觉比例的区域使用稳定高度或 `aspect-ratio` 的替代方案；地图、缩略图、印章、状态点不能被内容撑变形。
- 并排列超过两个时，必须检查窄屏。窄屏无法容纳时应换行、纵向堆叠或缩短文案。
- 文案来自用户、地点、AI 生成、昵称、接口返回时，一律视为不可信长度。
- 标题允许换行；按钮短操作优先不换行并省略；说明文案允许自然换行。
- 对齐不要依赖某个文案刚好是两个字。文案变成四个字、六个字时仍要成立。
- 不要用负 margin 调整适配问题。

## 交互与动效

- 主要点击元素必须有 active 反馈。
- 动画只优先使用 `transform` 和 `opacity`。
- 避免动画 `top`、`left`、`width`、`height`、`margin` 等容易触发布局重排的属性。
- 页面入场、面板展开、列表出现可以有轻微位移和透明度变化，但持续时间应控制在 `180ms-340ms`。
- Loading、保存中、录音中、生成中必须有明确状态，不要只让按钮“卡住”。
- 空状态必须给下一步行动，例如去设置探索点、重新搜索、恢复登录、继续记录。
- 地图区域不要叠加过多装饰层，避免影响拖动和真机性能。
- 底部主行动栏必须考虑安全区域、滚动遮挡和长文案。

## 页面级约束

### 探索页 `miniprogram/pages/index`

- 探索页必须保持“两步式”主流程：先设置探索点，再生成主题。
- 未设置探索点时，不允许生成主题或开始漫步。
- 设置探索点的方式包括当前定位、地点搜索、附近地点推荐。地图可以保留滚动、缩放、旋转浏览能力，但不要恢复“地图选点 / 确认这里”入口，除非用户明确要求。
- 第 1 步的当前定位按钮应和探索点信息同组，放在探索点信息右侧，宽度约占该行四分之一；避免单独增加“要从哪里出发”等说明行。搜索行必须和探索点信息行使用同一套右侧行动列：搜索按钮左边界要对齐当前定位按钮左边界，输入框约占操作行三分之二，间距由父级布局或显式 spacer 控制。
- 设定探索点后不要自动跳到第 2 步；用户可以继续改地点。但“下一步”按钮和顶部“生成任务”步骤都应可进入第 2 步。
- 纯粹模式主题选择必须收敛为单选。
- 进阶模式主题方向支持 1 到 2 个选择。
- 纯粹模式生成 1 个任务，进阶模式生成 3 个任务。
- 进阶模式的天气、季节、状态、偏好等个性化选项应和纯粹模式主题选择使用同一套标签与 chip 样式；不要额外包一层摘要卡片，也不要默认做展开/收起。
- 主题方向统一为：形状、色彩、声音、数字、气味。
- “随机”只负责随机挑主题方向，之后仍走单主题生成链路，不恢复旧随机云函数。
- 生成中状态要有过程感，但不能遮挡用户理解当前阶段。
- 生成结果应像一张可以带走的任务票据，而不是普通表单结果。

### 主题卡 `miniprogram/components/theme-card`

- 主题卡是探索页的情绪核心，要突出主题、任务和开始行动。
- 任务文本必须真实可执行，避免泛泛而谈的“观察城市之美”。
- 错误状态、生成中状态、成功状态要视觉区分。
- 不要把调试信息暴露给普通用户，除非当前已有显式 debug 开关。

### 记录页 `miniprogram/pages/record`

- 记录页首屏必须让用户快速知道：当前主题、当前位置/轨迹状态、任务进度、保存动作。
- 任务打卡不是表单填报，应像“一站一站收集线索”。
- 拍照、录像、录音、文字均为任务素材能力，按钮要稳定、可触达、状态明确。
- 录音中必须有明显状态提示，并能停止。
- 保存按钮必须容易触达，不能被内容挤没。
- 已开始的单人记录遵循“开始即创建记录、保存即结束记录”的状态型链路，不要改回纯本地临时记录。

### 足迹页 `miniprogram/pages/history`

- 足迹页应像“纪念卡册 / 成就”，不要设计成管理后台列表。
- 保留“状态 + 记录类型”组合筛选。
- 单人和同行记录要能在卡片上被清楚区分。
- 进行中记录要能继续恢复任务。
- 已结束单人记录支持详情页分享给微信朋友。
- 已结束同行记录的删除只影响当前用户视角，不联动删除其他成员数据。

### 个人页 `miniprogram/pages/profile`

- 个人页优先表达账号状态、授权状态、数据入口和反馈入口。
- 登录、退出、恢复登录的文案必须谨慎，不要误导用户。
- 状态切换不能造成布局大跳动。

### 同行相关页面

- 同行主链路是：建房 -> 等待 -> 房主开始 -> 成员提交 -> 房主结束 -> 团队结果。
- 房主和成员权限要明确，不要把房主操作暴露给普通成员。
- 未加入用户只能看基础预览信息，不能暴露成员、贡献和动态流。
- 成员昵称、贡献内容、任务标题都要处理长文本。

## 代码风格

- 遵循现有 CommonJS 写法：`require` / `module.exports`。
- 小程序页面继续使用 `Page({ ... })`，组件继续使用 `Component({ ... })`。
- 不要引入 TypeScript、Babel、ESM 构建链或额外编译步骤。
- 优先在 `services/` 做接口适配，在 `utils/` 做纯工具逻辑，页面层只承担状态编排和视图事件。
- 接口响应必须通过现有 normalize 思路兼容云函数和 Web 后端。
- `setData` 要收敛，避免在高频事件中大量更新大对象。
- 地图、录音、上传、生成等异步流程必须有 `try/catch/finally`，并恢复 loading 状态。
- 用户可见错误用 `wx.showToast` 或 `wx.showModal` 给下一步，不要只 `console.error`。
- 不要把测试数据、临时 mock、调试开关默认暴露给线上用户。

## WXSS 质量规则

- 固定格式 UI 要有稳定尺寸或响应约束，例如地图、任务按钮、媒体缩略图、底部操作栏。
- `flex` 子元素承载长文本时必须设置 `min-width: 0`。
- 横向布局中，图标、状态、按钮这类固定元素必须 `flex-shrink: 0`。
- 横向布局中，承载标题、地点名、任务说明的容器必须 `flex: 1; min-width: 0;`。
- 长文本常用 `word-break: break-word`；按钮短文案可用 `white-space: nowrap; overflow: hidden; text-overflow: ellipsis;`。
- 横向按钮组要考虑小屏宽度，不要让文字挤出按钮。
- 同一按钮组内不要混用不同 `line-height`、不同高度和不同圆角。
- 不要把局部组件的 margin 写得过大来制造分隔感。优先让父级 section 控制间距。
- 重复出现 3 次以上的颜色、按钮、卡片、标签样式，应考虑抽到全局或组件基础 class。
- 避免大面积 `backdrop-filter`、复杂多层阴影和高频渐变动画。
- `z-index` 只在弹层、底部栏、地图浮层等确有层级需求时使用。
- 新增 keyframes 前先确认是否可以复用已有入场动画。

## 内容与文案

- 文案要自然、轻，但不幼稚。
- 可以使用“66”作为产品人格提示，但不要滥用。
- 错误文案要说明下一步，例如“请先设置探索点”“可以换个关键词再试”。
- 不要承诺不确定结果，例如“马上生成完美路线”“一定能识别成功”。
- 隐私、定位、登录、删除、分享相关文案必须清楚说明影响。

## 验收清单

前端重构完成前，必须自查：

- 页面背景、卡片、按钮、chip 色彩是否统一。
- 每页是否只有一个最强主行动。
- 胶囊是否只用于标签、chip 或轻量选择。
- 同类按钮的高度、圆角、字号、阴影、active 反馈是否一致。
- 并排按钮在窄屏上是否不溢出、不挤压、不换成难看的多行。
- Loading、disabled、active、error 状态下按钮尺寸是否不跳动。
- 相邻容器间距是否遵守统一层级，是否存在局部过近或过远。
- 同类页面的标题区、内容区、主行动区排列是否一致。
- 同类卡片、筛选 chip、状态 pill、空状态是否保持同一视觉语言。
- 是否出现卡片套卡片、满屏漂浮卡片、重黑阴影、廉价渐变。
- 主要点击项是否都有 active 反馈。
- 生成中、保存中、录音中、上传中、错误、空状态是否明确。
- 长地点名、长主题名、长任务、长昵称是否不会溢出。
- 小屏下按钮文字是否仍能放下。
- 大字体或较长中文文案下，容器是否不会互相覆盖。
- 地图拖动、列表滚动、任务展开是否不明显卡顿。
- 探索页是否仍满足“先探索点、后生成”的流程约束。
- 单人记录和同行记录的状态链路是否没有被破坏。
- 云开发模式和 Web 后端模式是否都没有被写死。

## 验证方式

如果只是文案或小样式改动：

- 至少检查相关 `.wxml` / `.wxss` 是否存在长文本、按钮宽度、active 状态问题。
- 至少检查一个窄屏宽度下按钮和容器间距是否正常。

如果涉及页面结构、主流程或组件复用：

- 使用微信开发者工具预览相关页面。
- 检查探索页、记录页、足迹页三个主页面。
- 检查至少一种空状态、一种 loading 状态、一种错误状态。
- 检查至少三类按钮状态：默认、按压、禁用或 loading。
- 检查至少两种宽度：窄屏和主流宽度。
- 对比同类页面或同类组件，确认排列、间距、圆角、阴影没有明显漂移。

如果涉及接口、生成、记录、上传、同行：

- 分别检查云开发兜底模式和 Web 后端接口模式是否仍有清晰路径。
- 不确定时不要假设接口字段，先读 `miniprogram/services/api.js` 和对应 service。

## 输出要求

当用户要求前端重构时，回复或提交内容必须包含：

- 本次改了哪些页面/组件。
- 解决了什么具体体验问题。
- 是否影响主流程、接口字段、数据结构。
- 已做的验证；如果无法真机验证，要明确说明未验证项。

不要只说“优化了 UI”“提升了体验”。必须具体到用户动作和页面状态。

## 客户沟通与 relay mode

如果用户粘贴客户消息，并以 `客户说：`、`客户：` 或类似格式开头，默认进入 relay mode：

- 输出 ready-to-send 区块，标题为 `发给客户：`。
- 保持自然、简洁、商业现实。
- 推进范围、约束、时间、预算、验收标准和下一步。
- 不要过度承诺交付时间、价格或不确定功能。
- 有必要时补充 `内部判断：`，供用户自己参考。
