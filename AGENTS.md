# AGENTS.md

本文件约束在 `D:\liuliu-minimap` 中工作的 AI 编码代理。默认按本文执行；只有当用户明确指定其他角色、技能或输出方式时，才切换到对应规则。

## 项目定位

`liuliu-minimap` 是 `liuliu` City Walk 项目的原生微信小程序端。

核心体验是：

1. 探索页设置探索点。
2. 生成主题任务。
3. 单人或同行开始漫步。
4. 在记录页完成任务打卡、素材记录和轨迹追踪。
5. 在足迹页回看纪念卡册和成就。

这不是通用地图工具、社区信息流、后台管理系统，也不是营销落地页。所有改动都必须服务“轻松出门、观察城市、留下纪念”这条主线。

## 技术边界

- 这是原生微信小程序项目，主要代码在 `miniprogram/`，页面由 `.wxml`、`.wxss`、`.js`、`.json` 组成。
- 不要把项目改成 React、Vue、uni-app、Taro、Tailwind 或其他框架。
- 不要引入重型 UI 库、动画库、状态管理库或新的构建链。
- 优先使用现有小程序能力：`wxss transition`、`wxss animation`、条件 class、已有组件、现有 `services/` 和 `utils/`。
- 后端同时兼容云开发兜底模式和 Web 后端共用接口模式。不要破坏 `miniprogram/utils/config.js` 控制的后端切换逻辑。
- `cloudfunctions/shared/*-runtime.js` 是共享运行时源文件。修改成就或主题生成共享逻辑后，必须同步对应云函数 runtime。
- 代码继续遵循现有 CommonJS 写法：`require` / `module.exports`。页面使用 `Page({ ... })`，组件使用 `Component({ ... })`。

## 前端改造核心规则

### 产品与流程

- 不要只做视觉换皮。每次 UI 改动都要让用户下一步更清楚、操作更少或状态更明确。
- 不要新增无业务意义的装饰层、营销文案、宣传模块或解释型大段文字。
- 不要为了“代码整洁”改动主流程状态机、接口字段或数据结构，除非用户明确要求并完成回归检查。
- 不要删除或弱化隐私、定位授权、登录、分享、删除确认等关键保护流程。
- 用户可见错误必须说明下一步，例如“请先设置探索点”“可以换个关键词再试”。

### 视觉语言

- 产品视觉应保持温暖、轻量、城市漫步感，不做科技后台、高饱和潮流 App 或通用 AI 风。
- 优先使用现有色板：背景 `#f5f2ed` / `#f4efe6`，卡片 `#fffdf8`，浅底块 `#faf4ea` / `#efe7da`，主文字 `#241f19`，次级文字 `#746b5f`，弱提示 `#9a9082`，主行动 `#626247` / `#3f402f`，风险 `#a55449`。
- 优先复用 `miniprogram/app.wxss` 中的 `.page-shell`、`.section-title`、`.section-subtitle`、`.primary-btn`、`.secondary-btn`、`.ghost-btn`、`.danger-btn`、`.pill`、`.meta-pill`。
- 不要使用大面积紫色、蓝紫渐变、霓虹、玻璃拟态、黑色重阴影、随机 emoji 或无来源图标。
- 页面可以有卡片，但不要卡片套卡片，也不要把每个区块都做成漂浮卡片。

### 按钮、间距与适配

- 每个可视区域原则上只保留一个最强主按钮。生成、开始、保存、创建房间、完成等主行动应复用同一套视觉规则。
- 次级按钮、幽灵按钮、危险按钮、状态 pill、选择 chip 必须有清楚区分，不要同色同重表达不同风险。
- 横向按钮组使用 `display: flex`，子项使用 `flex: 1; min-width: 0;`，间距由父级控制。
- `button[disabled]`、loading、active 状态要单独处理，按钮尺寸不能因为文案变化而跳动。
- 页面外边距、大区块、卡片内边距、按钮组间距要保持统一层级，不要用大 margin 或负 margin 修补局部问题。
- 长地点名、长主题、长任务、长昵称、接口返回文本都要按不可信长度处理，避免溢出按钮或卡片。
- 底部 sticky 行动栏必须预留页面底部空间，不能遮住最后一项内容。

### 交互与状态

- 主要点击元素必须有 active 反馈。
- 动画优先使用 `transform` 和 `opacity`，避免用 `top`、`left`、`width`、`height`、`margin` 做高频动画。
- 生成中、保存中、录音中、上传中、错误、空状态都必须有明确反馈和下一步行动。
- 地图、列表、任务展开、媒体操作要优先保证真机顺滑和稳定。

## 页面级约束

### 探索页 `miniprogram/pages/index`

- 保持“两步式”主流程：先设置探索点，再生成主题。
- 未设置探索点时，不允许生成主题或开始漫步。
- 设置探索点方式包括当前定位、地点搜索、附近地点推荐。不要恢复“地图选点 / 确认这里”入口，除非用户明确要求。
- 设定探索点后不要自动跳到第 2 步；用户可以继续改地点。
- 纯粹模式主题选择为单选，生成 1 个任务。
- 进阶模式主题方向支持 1 到 2 个选择，生成 3 个任务。
- 主题方向统一为：形状、色彩、声音、数字、气味。
- “随机”只负责随机挑主题方向，之后仍走单主题生成链路。

### 主题卡 `miniprogram/components/theme-card`

- 主题卡要突出主题、任务和开始行动。
- 任务文本必须真实可执行，避免泛泛而谈。
- 错误、生成中、成功状态要视觉区分。
- 不要向普通用户暴露调试信息，除非已有显式 debug 开关。

### 记录页 `miniprogram/pages/record`

- 首屏要让用户快速知道当前主题、位置 / 轨迹状态、任务进度和保存动作。
- 任务打卡应像“一站一站收集线索”，不要做成表单填报。
- 拍照、录像、录音、文字都是任务素材能力，按钮要稳定、可触达、状态明确。
- 已开始的单人记录遵循“开始即创建记录、保存即结束记录”的状态型链路，不要改回纯本地临时记录。

### 足迹页 `miniprogram/pages/history`

- 足迹页应像“纪念卡册 / 成就”，不要做成管理后台列表。
- 保留“状态 + 记录类型”组合筛选。
- 单人和同行记录要能在卡片上被清楚区分。
- 进行中记录要能继续恢复任务。
- 已结束单人记录支持详情页分享给微信朋友。
- 已结束同行记录的删除只影响当前用户视角，不联动删除其他成员数据。

### 个人页 `miniprogram/pages/profile`

- 优先表达账号状态、授权状态、数据入口和反馈入口。
- 登录、退出、恢复登录的文案必须谨慎，不要误导用户。
- 状态切换不能造成布局大跳动。

### 同行相关页面

- 主链路是：建房 -> 等待 -> 房主开始 -> 成员提交 -> 房主结束 -> 团队结果。
- 房主和成员权限要明确，不要把房主操作暴露给普通成员。
- 未加入用户只能看基础预览信息，不能暴露成员、贡献和动态流。
- 成员昵称、贡献内容、任务标题都要处理长文本。

## 代码与 WXSS 质量

- 优先在 `services/` 做接口适配，在 `utils/` 做纯工具逻辑，页面层只承担状态编排和视图事件。
- 接口响应必须通过现有 normalize 思路兼容云函数和 Web 后端。
- `setData` 要收敛，避免在高频事件中大量更新大对象。
- 地图、录音、上传、生成等异步流程必须有 `try/catch/finally`，并恢复 loading 状态。
- 不要把测试数据、临时 mock、调试开关默认暴露给线上用户。
- 固定格式 UI 要有稳定尺寸或响应约束，例如地图、任务按钮、媒体缩略图、底部操作栏。
- `flex` 子元素承载长文本时必须设置 `min-width: 0`；固定元素如图标、状态、按钮必须 `flex-shrink: 0`。
- 重复出现 3 次以上的颜色、按钮、卡片、标签样式，应考虑抽到全局或组件基础 class。

## 文档撰写规则

- 文档必须和代码当前状态一致。写说明前先读相关代码，不确定时不要猜字段、接口、文件名或页面行为。
- 不要把计划中的能力写成已经实现。需要表达规划时，明确分成“已实现”“暂不做”“下一步建议”。
- 功能说明文档优先写清楚：功能范围、用户入口、核心流程、关键代码位置、数据 / 接口影响、异常状态、已知限制。
- 方案 / PRD 文档在功能落地后要更新口径：先说明当前已实现哪个方案，再说明推荐下一步，而不是继续保留旧版待选方案的语气。
- 删除过期口径。不要让同一份文档同时出现互相矛盾的新旧描述；历史背景必要时放到“背景”或“变更记录”。
- 文风要直接、自然、可执行。少用空泛词，例如“赋能”“沉淀”“打造闭环”“全面提升”“智能化体验升级”。
- 不要用长句堆概念。优先使用短段落、清晰小标题和列表，让维护者能快速找到结论。
- 说明用户体验时要具体到页面状态和用户动作，不要只写“优化体验”“提升稳定性”。
- 引用代码路径、接口名、字段名、云函数名时必须准确，并使用反引号标记。
- 涉及验证时，写清楚已经验证什么、没有验证什么，以及未验证的原因。
- 面向客户或非技术读者的文档，减少实现细节；面向开发者的文档，保留必要路径、状态和数据结构。

## 验证与输出

如果只是文案或小样式改动：

- 至少检查相关 `.wxml` / `.wxss` 是否存在长文本、按钮宽度、active 状态问题。

如果涉及页面结构、主流程或组件复用：

- 检查探索页、记录页、足迹页三个主页面。
- 检查至少一种空状态、一种 loading 状态、一种错误状态。
- 检查至少三类按钮状态：默认、按压、禁用或 loading。
- 检查窄屏和主流宽度下的按钮、卡片、底部行动区是否正常。

如果涉及接口、生成、记录、上传、同行：

- 分别检查云开发兜底模式和 Web 后端接口模式是否仍有清晰路径。
- 不确定接口字段时，先读 `miniprogram/services/api.js` 和对应 service。

回复或提交内容必须说明：

- 本次改了哪些页面、组件或文档。
- 解决了什么具体问题。
- 是否影响主流程、接口字段、数据结构。
- 已做的验证；无法真机验证时，要明确说明未验证项。

不要只说“优化了 UI”“提升了体验”。

## 客户沟通与 relay mode

如果用户粘贴客户消息，并以 `客户说：`、`客户：` 或类似格式开头，默认进入 relay mode：

- 输出 ready-to-send 区块，标题为 `发给客户：`。
- 保持自然、简洁、商业现实。
- 推进范围、约束、时间、预算、验收标准和下一步。
- 不要过度承诺交付时间、价格或不确定功能。
- 有必要时补充 `内部判断：`，供用户自己参考。
