# 云函数登录系统说明

## 1. 文档目的

本文说明当前小程序在“仅使用微信云开发，不依赖 Web 后端”的前提下，登录系统已经实现到什么程度、采用了哪些技术方案、目前存在哪些不足，以及后续如何继续加固到更稳健、可用于生产环境的状态。

适用范围：

- 微信小程序端 `miniprogram/`
- 云函数端 `cloudfunctions/syncUser`
- 基于微信云开发身份上下文 `OPENID` 的个人历史记录隔离

---

## 2. 当前已经实现的功能

### 2.1 基础身份识别

当前云函数登录系统已经能够基于微信云开发运行时上下文识别当前用户身份：

- 云函数通过 `cloud.getWXContext()` 获取当前访问用户的 `OPENID`
- `OPENID` 作为当前系统在云函数模式下的真实用户唯一标识
- 不依赖自建 Web session，也不依赖服务端 JWT

当前关键实现：

- `cloudfunctions/syncUser/index.js`
- `cloudfunctions/createWalk/index.js`
- `cloudfunctions/listMyWalks/index.js`
- `cloudfunctions/getWalkDetail/index.js`
- `cloudfunctions/deleteWalk/index.js`
- `cloudfunctions/publishWalkShare/index.js`

### 2.2 用户资料同步

当前已经支持在“小程序个人页”中同步用户资料到云数据库 `users` 集合：

- 用户首次使用时可填写昵称
- 用户首次使用时可选择微信头像
- 点击“创建资料并登录”后，将资料写入 `users` 集合
- 如用户文档已存在，则更新资料和最近登录时间
- 如用户文档不存在，则创建新用户文档
- 已登录用户可在个人页单独修改头像和昵称，并点击“保存资料”同步

当前已写入字段包括：

- `openid`
- `nickName`
- `avatarUrl`
- `role`
- `createdAt`
- `lastLoginAt`
- `updatedAt`

### 2.3 登录态恢复

当前已经支持小程序启动后的用户态恢复机制：

- 本地缓存用户资料 `citywalk_user`
- 本地缓存登录标记 `citywalk_login_status`
- 本地缓存“用户是否手动退出过”的标记 `citywalk_manual_logout`
- 小程序启动时先读本地缓存，提升首屏体验
- 启动后默认调用云函数查询当前 `OPENID` 对应的用户资料
- 若云端存在用户资料，则自动恢复登录态
- 若云端不存在用户资料，则清理本地登录态，避免假登录
- 若用户刚刚手动退出，则本次启动不自动恢复，直到用户手动点一次登录

### 2.4 页面登录态接入

目前已接入的页面包括：

- 个人页：展示当前用户资料，区分“直接登录 / 首次创建资料 / 修改资料 / 退出登录”
- 历史页：优先吃到自动登录；未进入账号时区分“首次完善资料”和“恢复登录”
- 记录页：保存漫步记录前强制检查登录状态，未进入账号时区分“首次完善资料”和“恢复登录”

### 2.5 个人历史记录隔离

当前已经可以在云函数模式下正确区分不同用户的历史信息：

- 保存记录时，将 `userId` 写为当前 `OPENID`
- 查询“我的历史”时，使用 `where({ userId: wxContext.OPENID })`
- 查询详情时，未公开记录仅本人可见
- 删除记录时，仅本人可删
- 发布分享时，仅本人可操作

这意味着：

- 即使没有 Web 后端账号体系
- 也已经具备“每个微信用户只能看到自己的私有历史”的能力

## 3. 技术说明

## 3.1 总体架构

当前登录系统是“微信云函数身份识别 + 云数据库用户资料 + 小程序本地缓存”的组合方案。

链路如下：

1. 用户打开小程序
2. `app.js` 初始化云环境
3. 小程序读取本地缓存中的用户资料和登录偏好
4. 如果用户没有手动退出，则调用 `syncUser` 云函数执行 `action=get`
5. 云函数通过 `OPENID` 读取 `users` 集合中的用户文档
6. 小程序根据返回结果恢复或清理当前登录态
7. 若云端已有资料，用户可直接进入登录态
8. 若云端没有资料，用户在个人页首次填写昵称/头像后调用 `syncUser` 执行 `action=sync`
9. 已登录用户后续修改昵称/头像时，同样调用 `syncUser` 执行 `action=sync`
10. 云函数按 `OPENID` 更新或创建用户资料

### 3.2 当前登录态模型

当前并没有实现传统意义上的账号密码、短信验证码、JWT、Refresh Token 体系。

现阶段的登录态定义是：

- “用户是否已经在当前小程序内完成资料同步”
- “系统是否能在当前微信环境下识别到该用户的 `OPENID`”

因此，本系统本质上是：

- 微信原生身份上下文登录
- 云函数内以 `OPENID` 识别用户
- 本地只保存展示资料和用户操作偏好

### 3.3 为什么当前方案可行

在纯微信云开发链路中，云函数天然可以拿到当前调用者的 `OPENID`，因此具备如下优点：

- 身份来源稳定
- 不需要自己交换 code 换 session
- 不需要引入独立鉴权服务
- 对“按人区分个人数据”这一目标足够直接

对于当前项目阶段，这是一种实现复杂度较低、落地速度较快的方案。

## 4. 当前实现到了什么程度

可以把当前成熟度理解为：

- 已经完成“可用的最小登录系统”
- 已经能支撑个人资料同步和个人历史记录隔离
- 已经能满足当前云函数模式下的核心业务流程
- 但还没有达到“多端统一账号体系”和“生产级安全治理”的完整程度

更具体地说，当前已达到：

- 可以识别用户
- 可以保存用户资料
- 可以恢复登录态
- 可以限制个人历史访问范围
- 可以在关键写入动作前做登录校验

当前尚未达到：

- 与 Web 后端统一用户体系
- 真正的 Access Token / Refresh Token 生命周期治理
- 细粒度权限体系
- 审计日志与风控
- 更严格的数据结构校验与异常分层
- 完整的数据迁移与用户合并策略

结论：

当前已经适合“测试环境、小范围试用、内部演示、MVP 验证”。

如果要用于正式生产环境，还需要继续完善。

## 5. 本次过程中遇到的问题

### 5.1 云函数入参与前端不一致

之前个人页登录时，前端向 `syncUser` 传的是：

- `code`
- `nickName`
- `avatarUrl`

但云函数中读取的是：

- `event.profile`

这会导致用户点击“登录成功”后，前端看似成功，但云函数不一定真正拿到了用户资料。

已修复方式：

- 将 `syncUser` 调整为兼容 `action`
- 将用户资料统一放在 `profile` 字段中传递
- 云函数兼容读取 `event.profile`，并在必要时兜底读取 `event`

### 5.2 之前没有真正的“登录态恢复”

旧实现更像“把用户资料存到本地”，而不是“每次启动都重新确认当前用户是否仍然有效”。

问题表现：

- 本地缓存可能还在
- 但云端用户文档可能不存在
- 页面依然会误认为处于登录状态

已修复方式：

- 启动时默认调用云函数读取当前用户
- 如云端有资料，则自动恢复到已登录状态
- 如云端无资料，主动清理本地状态
- 如用户主动退出，则通过 `citywalk_manual_logout` 暂停自动恢复

### 5.3 核心业务动作没有统一要求登录

之前历史页和个人页有登录概念，但保存漫步记录这类核心写入动作没有统一接入登录约束。

风险：

- 用户以为“自己没登录”
- 但实际上依然可以通过 `OPENID` 直接写入私有记录
- 产品心智不统一

已修复方式：

- 在记录页保存前增加登录检查
- 未进入账号时明确提示，并引导跳转到个人页
- 提示文案会区分“首次完善资料”和“恢复登录”

### 5.4 当前“退出登录”只是暂停自动恢复

现阶段退出登录的本质是：

- 清理本地缓存
- 清理小程序内存中的当前用户对象
- 写入 `citywalk_manual_logout`

它并不会让微信侧身份失效，因为 `OPENID` 仍然由微信云环境提供。

这不是 bug，而是当前技术方案下的自然结果。

意味着：

- 当前“退出登录”更接近“暂停本机自动登录”
- 不是传统互联网产品里的“服务端 session 销毁”

## 6. 建议进一步改进的地方

### 6.1 把登录态判断收口成统一能力

当前页面里已经开始使用统一的 `ensureUserReady()`，但仍建议继续收口：

- 做一个统一的 `requireLogin()` 方法
- 由该方法负责：
  - 等待用户态初始化
  - 判断是否已登录
  - 决定是否弹窗
  - 决定是否跳转到个人页

这样能避免每个页面重复写一套登录判断。

### 6.2 用户数据结构做显式校验

建议在云函数中增加更严格的数据验证：

- 昵称长度限制
- 头像 URL 类型检查
- 非法空对象过滤
- 字段白名单写入

目前虽然已经足够用，但生产环境下应该避免随意将前端对象直接写入数据库。

### 6.3 补齐用户集合索引与数据规范

建议在云数据库中明确约束：

- `users` 以 `openid` 为主键文档 ID
- `walkRecords.userId` 建索引
- `createdAt` / `lastLoginAt` / `updatedAt` 保持统一时间格式

这样后续查询性能和数据一致性会更稳定。

### 6.4 增加登录相关埋点和错误日志

建议记录以下事件：

- 登录点击
- 登录成功
- 登录失败
- 用户恢复成功
- 用户恢复失败
- 保存记录时因未登录被拦截

如果未来线上出现“我明明登录了为什么看不到历史”的问题，会更容易定位。

### 6.5 做首次使用、自动恢复、手动退出三种规则统一

后续要明确一个产品决策：

方案 A：未登录禁止保存任何个人记录  
方案 B：未登录允许按 `OPENID` 暂存，但历史页仍要求显式登录后查看

当前实现已经进一步细化为：

- 云端已有资料时，默认自动恢复
- 云端无资料时，首次保存或查看历史前引导先完善资料
- 用户主动退出后，不再自动恢复，需手动点一次登录

建议继续坚持一个统一规则，不要同时保留两套互相打架的行为。

### 6.6 与 Web 后端账号体系打通

如果后续要切到 Web 后端统一鉴权，则需要新增：

- 小程序 code 换取服务端登录凭证
- 服务端 token / refresh token 颁发
- token 过期与刷新逻辑
- 云函数与 Web 用户主键映射
- `openid` / `unionid` / 平台账号 ID 的绑定关系

否则未来会出现：

- 云函数体系是一套用户
- Web 后端又是另一套用户

后续整合成本会变高。

## 7. 后续如何修改，才能更稳健适用生产环境

建议按下面顺序推进。

### 第一阶段：把当前云函数登录链路做稳

目标：

- 继续使用云函数模式
- 把微信小程序端的登录体验做稳定

建议动作：

- 增加统一 `requireLogin()` 能力
- 在更多写操作前接入登录校验
- 增加云函数参数校验和错误码规范
- 增加用户态初始化失败重试机制
- 补测试用例与操作文档

### 第二阶段：把数据模型和监控补齐

目标：

- 降低后续扩展风险

建议动作：

- 统一用户表字段规范
- 为历史记录表建立必要索引
- 接入基础埋点和错误监控
- 增加关键云函数调用日志

### 第三阶段：与生产后端账号体系统一

目标：

- 支持正式商业化和跨端扩展

建议动作：

- 服务端接管登录态颁发
- 前端使用 token 调用 Web API
- 云函数只承担微信特有能力或兜底能力
- 通过服务端建立微信身份与平台用户主账号映射

到这一步，才能算真正意义上的“生产级统一登录体系”。

## 9. 测试方法

以下测试建议在微信开发者工具中完成，并确保当前配置为云函数模式：

- `miniprogram/utils/config.js` 中 `apiBaseUrl` 为空
- `useCloudWalkStorage: true`

### 9.1 基础准备

1. 上传并部署最新的 `syncUser`、`createWalk`、`listMyWalks`、`getWalkDetail`、`deleteWalk`、`publishWalkShare` 云函数
2. 打开微信开发者工具并清缓存
3. 确认当前测试账号可正常使用云开发环境

### 9.2 登录流程测试

1. 进入“我的账户”页
2. 若是新微信号，输入昵称并选择头像
3. 点击“创建资料并登录”
4. 预期结果：
   - 出现“登录成功”提示
   - 页面展示昵称和头像
   - 云数据库 `users` 集合中出现以当前 `openid` 为文档 ID 的记录

### 9.3 启动恢复测试

1. 登录成功后关闭并重新打开小程序
2. 进入“我的账户”页
3. 预期结果：
   - 无需重新创建资料
   - 当前用户资料自动恢复
   - 个人页直接显示已登录态

### 9.4 已有资料直接登录测试

1. 使用已在 `users` 集合中存在资料的微信号进入小程序
2. 进入“我的账户”页
3. 预期结果：
   - 页面应提示“可以直接登录当前账号”
   - 点击“直接登录”后进入已登录态
   - 不应再次要求创建资料

### 9.5 退出登录测试

1. 在“我的账户”页点击退出登录
2. 关闭并重新进入小程序
3. 再进入“足迹”页
3. 预期结果：
   - 不应自动恢复到已登录态
   - 足迹页提示“恢复登录”或“去完善资料”的对应文案
   - 个人页显示未登录态

### 9.6 保存记录拦截测试

1. 确保当前处于未登录状态
2. 完成一次漫步记录流程并点击保存
3. 预期结果：
   - 弹窗文案能区分“首次完善资料”或“恢复登录”
   - 可跳转到个人页
   - 不应生成新的历史记录

### 9.7 已登录保存历史测试

1. 先完成登录
2. 完成一次漫步并点击保存
3. 打开“足迹”页
4. 预期结果：
   - 保存成功
   - 历史页能看到新记录
   - 云数据库 `walkRecords` 中该记录的 `userId` 等于当前 `OPENID`

### 9.8 个人历史隔离测试

1. 使用测试账号 A 登录并生成若干记录
2. 切换到测试账号 B
3. 登录并进入“足迹”页
4. 预期结果：
   - 账号 B 看不到账号 A 的私有历史
   - 账号 B 只能看到自己的记录

### 9.9 权限校验测试

1. 记录下账号 A 的某条私有历史记录 ID
2. 切换到账号 B
3. 尝试通过详情页或调试方式读取该记录
4. 预期结果：
   - 私有记录读取失败
   - 删除、发布分享等操作应返回权限不足

## 10. 当前结论

当前这套云函数登录系统已经从“只有个人页展示资料”提升为：

- 有真实身份来源
- 有用户资料同步
- 有自动恢复
- 有手动退出后的暂停恢复
- 有关键页面接入
- 有历史记录隔离
- 有核心保存动作登录拦截

它已经可以支撑当前项目在云函数模式下的个人化能力。

但如果目标是正式生产环境、跨端统一账号体系、长期维护和商业化运营，仍建议继续推进：

- 登录能力统一封装
- 数据验证与日志体系补强
- 与 Web 后端账号体系对齐

当前阶段可视为：

- 云函数模式下的可用 MVP 登录系统
- 但不是最终态的生产级统一登录方案
