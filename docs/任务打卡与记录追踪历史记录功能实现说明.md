# 任务打卡、记录与追踪、历史记录功能实现说明

## 1. 文档目的

本文用于沉淀当前“小程序任务打卡、记录与追踪、历史记录”相关功能的真实实现情况，重点回答以下问题：

- 当前已经实现到了什么程度
- 采用了什么技术方案
- 最近这一轮改动解决了什么问题
- 还存在哪些不足
- 后续如何继续修改，才能更稳健地适配生产环境

本文内容基于当前仓库代码，不以设计稿或预期方案代替实际实现。

## 2. 功能范围与代码位置

本次涉及的主要模块如下：

- 记录页主逻辑：`miniprogram/pages/record/record.js`
- 记录页结构：`miniprogram/pages/record/record.wxml`
- 任务列表组件：`miniprogram/components/mission-list/mission-list.js`
- 任务列表视图：`miniprogram/components/mission-list/mission-list.wxml`
- 打卡卡片组件：`miniprogram/components/checkin-card/checkin-card.js`
- 历史列表页：`miniprogram/pages/history/history.js`
- 历史详情页：`miniprogram/pages/walk-detail/walk-detail.js`
- 历史详情结构：`miniprogram/pages/walk-detail/walk-detail.wxml`
- 接口归一化与 Web/云兼容：`miniprogram/services/api.js`
- 历史删除云函数：`cloudfunctions/deleteWalk/index.js`
- 草稿持久化：`miniprogram/app.js`、`miniprogram/utils/draft.js`

整体链路如下：

1. 探索页生成主题与任务。
2. 用户进入记录页，围绕一份 `walkDraft` 完成任务素材采集、路线记录与补充说明。
3. 记录页保存时上传媒体并调用 `createWalk` 落库。
4. 历史记录页通过 `listMyWalks` 展示个人记录列表。
5. 点击历史记录后进入详情页，在详情页按任务维度回看素材、卡片与记录内容。

## 3. 当前实现概览

### 3.1 总体完成度判断

可以把当前实现分成四档理解：

- `记录页主流程`：已达到可用状态
- `任务打卡`：已达到可用状态，但 AI 核验入口仍未完整前台接通
- `历史记录详情回看`：已达到可用状态，并完成了最近一轮任务维度增强
- `纪念卡册/成就体系`：仍处于占位或半接入状态

### 3.2 近期这一轮实际补齐的点

本轮已经完成的关键调整包括：

- 将“总结与补充”从普通全局备注升级为任务体系中的一个特殊任务位 `__summary__`
- 记录页中的图片、视频、录音、备注统一收敛到 `missionAssetMap`
- 记录页生成的任务打卡卡片会写回对应任务的 `cardImagePath`
- 历史详情页的任务列表不再只依赖 `themeSnapshot.missions`，而是采用“主题任务 + 已完成任务 + 核验记录 + missionAssetMap”的并集
- 历史详情页任务区改为纵向逐条展开式交互，不再全部挤在一行
- 历史详情页支持点击同一任务再次收起
- 历史详情页中的任务卡片改为“默认始终存在”，即使当时没有手动点击“生成打卡卡片”，详情页也会按当前任务素材自动生成一张展示卡片
- 历史详情页点击任务卡片后会弹出卡片弹窗，并支持保存到本地相册
- 历史详情页新增“删除这条历史记录”能力，并接入 `deleteWalk` 云函数
- `services/api.js` 中的 Web 请求归一化已补充 `missionAssetMap`、`missionReviews`、`routeStats`、`sticker`、`locationContext`、`locationAddress` 等字段透传

## 4. 任务打卡功能实现说明

### 4.1 已实现功能

记录页中的“任务打卡”目前已支持：

- 展示主题任务列表
- 点击任务切换展开/收起
- 手动标记完成/取消完成
- 针对某个任务上传图片
- 针对某个任务上传视频
- 针对某个任务录音
- 针对某个任务填写备注
- 基于任务素材生成任务打卡卡片
- 预览任务打卡卡片
- 保存任务打卡卡片到本地相册

### 4.2 当前技术方案

任务级数据统一收口到 `missionAssetMap`，结构如下：

```js
{
  [missionKey]: {
    photoList: [],
    videoList: [],
    audioList: [],
    noteText: '',
    cardImagePath: ''
  }
}
```

其中有一个特殊任务位：

- `__summary__`：表示“总结与补充”

当前逻辑里：

- 所有任务素材都按任务 key 落入 `missionAssetMap`
- 页面上的“总结与补充”也被视为一个任务位
- 全局 `noteText/photoList/videoList/audioList` 不再是独立编辑源，而是由 `__summary__` 同步聚合得到

这套结构的好处是：

- 任务级和总结级数据结构统一
- 保存和回放逻辑一致
- 历史详情页可以按任务完整还原展示

### 4.3 当前完成程度

从“用户能否真实用起来”的角度看，任务打卡已经达到 `可用`：

- 可以完成任务级素材沉淀
- 可以形成任务级卡片
- 可以在历史详情中按任务回看

但从“功能闭环完整度”看，仍然是 `可用但未封顶`：

- AI 任务核验逻辑底层已存在
- 但前台任务面板上还没有正式暴露核验入口
- 核验结果在历史详情中也还没有展开到足够完整的复盘层级

## 5. 记录与追踪功能实现说明

### 5.1 已实现功能

“记录与追踪”板块目前已支持：

- 图片记录
- 视频记录
- 录音记录
- 总结文字输入
- 实时轨迹记录
- 持续定位失败时自动降级为轮询采样
- 轨迹总时长统计
- 轨迹里程估算
- 轨迹起止时间展示
- 草稿自动持久化
- 保存整次漫步记录

### 5.2 当前技术方案

#### 1. 轨迹追踪

轨迹记录采用双层兜底策略：

- 优先 `wx.startLocationUpdateBackground`
- 其次 `wx.startLocationUpdate`
- 再失败则降级到 `setInterval + getCurrentLocation()` 轮询

采样点结构为：

```js
{
  latitude,
  longitude,
  timestamp,
  accuracy
}
```

距离计算采用 Haversine 公式。

#### 2. 草稿持久化

草稿通过以下机制持久化：

- 运行时状态：`App.globalData.walkDraft`
- 本地存储：`wx.setStorageSync`

即便用户中途离开页面，未保存内容也能恢复。

#### 3. 聚合同步

当前记录页使用 `syncDraftAggregates` 做一层统一同步：

- `missionAssetMap.__summary__.noteText -> draft.noteText`
- `missionAssetMap.__summary__.photoList -> draft.photoList`
- `missionAssetMap.__summary__.videoList -> draft.videoList`
- `missionAssetMap.__summary__.audioList -> draft.audioList`

这样既兼容旧的全局字段读取方式，又把真实数据源迁移到任务维度结构。

### 5.3 当前完成程度

“记录与追踪”是目前三块中实现最完整的一块，已达到 `主体流程可用 + 数据链路基本打通` 的程度。

目前仍欠缺的生产级能力包括：

- 定位权限失败后的更细粒度指引
- 更明确的录音/视频失败重试策略
- 长时间录制、弱网、大文件上传时的中断恢复策略
- 轨迹去抖、异常点过滤、精度修正

## 6. 历史记录功能实现说明

### 6.1 历史列表页当前状态

历史页目前保留两个入口：

- 纪念卡册
- 成就

其中当前真正可用的是：

- “纪念卡册”Tab 下的个人记录列表

当前列表功能已实现：

- 拉取最近 20 条记录
- 展示标题、地点、时长、距离、封面、摘要、时间
- 点击卡片进入详情页

“成就”当前仍是占位状态。

### 6.2 历史详情页当前状态

历史详情页目前已支持：

- 展示主题信息
- 展示地点信息
- 展示路线地图和时长/里程
- 按任务维度回看记录
- 任务展开/收起
- 任务级图片、视频、录音、备注展示
- 任务级打卡卡片展示
- 点击任务卡片弹窗预览
- 保存任务卡片到相册
- 删除当前这条历史记录

### 6.3 本轮详情页的重要改造

#### 1. 任务列表来源改为并集

历史详情页最初只按 `themeSnapshot.missions` 来构造任务列表，这会导致：

- 某些任务虽然有素材
- 或者有核验记录
- 或者有打卡卡片
- 但如果不在 `themeSnapshot.missions` 中，就不会显示出来

当前已改为以下来源的并集：

- `themeSnapshot.missions`
- `completedMissions`
- `missionReviews`
- `missionAssetMap`

这样任务回放更稳。

#### 2. 任务交互改为纵向展开式

详情页原来的任务切换更像一排横向标签，不适合任务数量较多或任务名较长的情况。当前已改为：

- 纵向逐条排列
- 点开一个任务展示其内容
- 再点一次同一任务收起

交互上更接近记录页任务列表。

#### 3. 打卡卡片改为“默认始终存在”

当前已不再把“点过生成卡片按钮”视为卡片存在的前提。

现在详情页里的任务卡片逻辑是：

- 如果任务已有已保存的卡片资源，也可以继续使用
- 即使没有显式保存过 `cardImagePath`，只要任务有素材和文字，详情页也会调用 `checkin-card` 自动渲染出一张卡片用于展示

这样“生成卡片按钮”更接近一个显式触发/保存入口，而不是“卡片存在与否”的开关。

#### 4. 已增加删除历史记录能力

当前删除入口放在历史详情页底部。

实现方式为：

- 前端详情页点击“删除这条历史记录”
- 弹出二次确认
- 调用 `services/walk.deleteWalk`
- 在云函数模式下转发到 `deleteWalk`
- 云函数按记录 `id` 查询 `walkRecords`
- 校验 `walk.userId === wxContext.OPENID`
- 校验通过后执行删除

当前删除能力的产品边界是：

- 仅“我的历史记录”详情页显示删除按钮
- 社区页、分享页不显示删除按钮
- 删除仅允许当前微信账号删除自己的记录
- 删除成功后返回上一页或回到历史列表页

### 6.4 当前完成程度

历史记录板块目前处于 `列表和详情已可用，卡册体系与成就体系未完成` 的状态：

- 历史列表：可用
- 历史详情：可用，且任务回放能力较之前明显增强
- 真正的纪念卡册聚合视图：未完成
- 成就体系：未完成

## 7. 技术实现说明

### 7.1 打卡卡片生成技术

任务卡片生成采用前端 canvas 本地渲染，而不是服务端出图。

核心特点：

- 使用纸张纹理、邮戳、条码等本地素材
- 标题、正文、地点、日期自动排版
- 支持任务图片自动拼接
- 通过 `wx.canvasToTempFilePath` 导出临时图

优点：

- 响应快
- 不依赖服务端图片生成
- 卡片样式统一

局限：

- 强依赖小程序 canvas 能力
- 弱设备上渲染时延和兼容性需要继续验证
- 没有统一的服务端卡片版本控制

### 7.2 数据归一化技术

`services/api.js` 中的 `normalizeWalkRecord()` 负责把不同来源的记录格式统一为前端可消费结构。

当前统一后的关键字段包括：

- `themeTitle`
- `locationName/locationContext/locationAddress`
- `routePoints`
- `routeStats`
- `completedMissions`
- `missionReviews`
- `missionAssetMap`
- `photoList/videoList/audioList`
- `sticker`
- `themeSnapshot`

同时 `coverImage` 已做增强选择策略：

- 优先 `item.coverImage`
- 再用 `__summary__` 的卡片或图片
- 再退化到全局图片
- 再退化到任意任务卡片

这样历史列表封面更稳定。

### 7.3 Web/云双通道兼容

当前接口层支持：

- 云函数模式
- Web 接口模式

本轮已把 Web 模式 `createWalk` 的透传字段补齐到更完整水平，包含：

- `themeSnapshot`
- `missionReviews`
- `missionAssetMap`
- `walkMode`
- `generationSource`
- `trackStartedAt`
- `trackStoppedAt`
- `routeStats`
- `sticker`
- `locationContext`
- `locationAddress`

这比之前的“只传最简字段”更接近真实业务需要。

### 7.4 历史删除的技术实现

历史删除当前采用“前端按钮 + 服务层封装 + 云函数权限校验”的实现方式。

关键点如下：

- 详情页通过 `walk.id / walk._id` 传入删除参数
- `services/api.js` 对 `deleteWalk` 做统一调用和返回格式归一化
- 云函数删除前会先读取记录，再校验是否属于当前 `OPENID`
- 前端对删除失败原因做了基础识别，包括：
  - 记录不存在
  - 权限不足
  - 缺少记录 id
  - 云函数未部署

这意味着当前删除功能已经具备基础安全性，但仍依赖云环境部署和环境一致性。

## 8. 当前仍存在的问题

### 8.1 任务核验入口未完整接通

虽然底层已有 `verifyMissionForTask`，但前台仍未给出稳定、明确的任务核验入口，导致：

- 任务核验能力存在但不显性
- 用户难以理解“完成任务”和“核验任务”的区别

### 8.2 详情页自动生成卡片仍是前端临时态

历史详情页中“默认存在的任务卡片”当前是现场临时生成的：

- 展示上已经满足需求
- 但未必会自动回写持久化
- 也没有版本管理

这意味着：

- 不同端可能生成结果略有差异
- 样式升级后历史卡片可能发生变化

### 8.3 任务回放信息仍未完全产品化

目前虽然能按任务展示图片、视频、录音、备注、卡片，但以下内容仍未充分利用：

- 核验得分
- 核验评论
- 核验证据
- 完成状态说明

### 8.4 弱网与异常链路未完全覆盖

当前保存链路虽然已基本可用，但仍缺：

- 上传中断恢复
- 分片上传
- 大文件失败后的补偿机制
- 并发上传失败后的更细粒度提示

### 8.5 缺少系统化测试

当前更偏开发中迭代态，缺少：

- 单元测试
- 任务数据结构回归测试
- 历史详情页渲染测试
- Web/云双通道一致性测试

### 8.6 历史删除依赖部署一致性

当前删除能力虽然代码已经接通，但它对运行环境一致性有明显依赖：

- `deleteWalk` 云函数必须实际部署到当前小程序所使用的云环境
- 删除函数和 `listMyWalks/getWalkDetail` 必须处于同一个云环境
- 前端传入的 `id` 必须和数据库文档 `_id` 对齐

如果这几项有一项不一致，就可能出现：

- 历史里还能看到记录
- 删除按钮点击后提示“记录不存在”
- 或者提示云函数未找到

## 9. 本轮过程中遇到的问题

这轮改动中实际遇到的主要问题如下：

### 9.1 任务数据源不统一

最初记录页同时维护：

- 全局 `noteText/photoList/videoList/audioList`
- 任务级 `missionAssetMap`

这样会导致：

- 保存时逻辑容易分裂
- 历史页不容易确定该看哪份数据

当前已经通过 `__summary__` 聚合位做了统一收口，但仍属于兼容式收敛，不是彻底移除旧字段。

### 9.2 历史详情任务列表容易漏任务

只按 `themeSnapshot.missions` 组装列表，会导致真实有记录的任务被漏掉。

当前已改为多来源并集。

### 9.3 打卡卡片“是否存在”的定义不一致

之前逻辑默认：

- 点过生成按钮，卡片才存在

而产品预期是：

- 卡片本质上一直存在，按钮只是显式展示或保存的入口

当前已在详情页按后者做兼容实现。

### 9.4 历史详情交互与记录页风格不一致

之前详情页任务区更像一个单薄的横向标签栏，和记录页的纵向任务交互风格差异较大。当前已调整为更接近记录页的展开式结构。

### 9.5 历史删除的报错语义与真实状态容易混淆

在删除功能刚接入时，前端只会统一提示“删除失败”，用户无法判断到底是：

- 没部署云函数
- 权限不对
- 记录确实不存在
- 还是传错了 id

当前虽然已经补了更具体的失败提示，但从联调过程看，删除链路最容易暴露“环境不一致”和“文档 id 不一致”这类问题，后续仍需通过日志和测试进一步固化。

## 10. 建议进一步改进的地方

### 10.1 产品层建议

建议优先补齐：

1. 明确“任务完成”“任务核验”“打卡卡片生成”三者关系。
2. 在记录页给任务核验提供正式按钮和状态反馈。
3. 在历史详情中展示任务核验结果，包括分数、建议、证据。
4. 为“纪念卡册”补一个真正的聚合视图，而不只是记录列表。
5. 正式定义“成就”的规则和数据结构。

### 10.2 前端工程层建议

建议继续做：

1. 把任务卡片生成逻辑从“只在页面内可用”提升到“可选持久化缓存”。
2. 把任务详情卡片、图片、视频、录音等回放区域拆成更小的展示组件。
3. 给 `missionAssetMap`、`missionReviews`、`routeStats` 建统一 schema 校验。
4. 增加页面级 loading、空态、异常态的更细分处理。
5. 给任务列表与详情切换补节流和异常渲染保护。

### 10.3 后端接口层建议

建议进一步确认：

1. Web 接口与云函数在记录字段上的一致性。
2. `missionAssetMap`、`cardImagePath`、`sticker` 是否都能稳定持久化。
3. 历史详情接口是否返回完整任务结构。
4. 删除接口是否在云函数和 Web 模式下都保持一致的鉴权和返回结构。
5. 是否需要服务端生成/缓存正式版卡片资源。

## 11. 如何修改得更稳健并适用于生产环境

### 11.1 数据结构层

生产环境建议把任务记录数据模型正式定下来，不再长期保留“双轨兼容”状态。建议目标模型：

- 一个主记录对象
- 一个稳定的 `missionAssetMap`
- 一个稳定的 `missionReviews`
- 一个明确的 `summary` 或 `__summary__` 约定

并在前后端同时做 schema 约束。

### 11.2 卡片生成层

当前卡片展示已经可用，但生产环境更建议：

1. 前端本地生成用于即时预览。
2. 服务端生成或持久化一份正式卡片资源。
3. 历史记录默认优先读正式资源，失败时再退回前端现场生成。

这样可以保证：

- 多端一致
- 历史卡片结果稳定
- 后续改版不会影响旧数据展示

### 11.3 上传与保存层

建议补齐：

- 文件大小限制
- 文件类型白名单
- 上传失败重试
- 批量上传的部分成功/部分失败处理
- 保存前校验任务数据完整性

### 11.4 历史删除与权限层

如果要稳健适用于生产环境，历史删除建议进一步补齐：

1. 删除成功后主动刷新历史列表，而不只依赖返回上一页。
2. 删除前后补日志，记录 `id/openid/result/reason`，便于排查。
3. 对“记录不存在”和“权限不足”做更明确的埋点监控。
4. 明确云环境配置，避免 `listMyWalks` 和 `deleteWalk` 指向不同环境。
5. 如果未来接 Web 后端，保持和云函数一致的鉴权语义。

### 11.5 定位与设备能力层

生产环境建议针对以下场景补专门策略：

- 权限被拒绝
- 后台定位不可用
- 录音能力不可用
- 老机型 canvas 渲染失败
- 大图导致内存抖动

### 11.6 测试与监控层

建议至少补这些：

- `missionAssetMap` 结构单元测试
- `normalizeWalkRecord` 归一化测试
- 记录页保存参数构造测试
- 历史详情任务列表组装测试
- 卡片自动生成回调测试
- 历史删除成功/失败分支测试
- Web/云双通道回归测试
- 上传失败与权限失败埋点

## 12. 当前结论

截至当前版本，可以给出如下判断：

- 任务打卡：`可用`
- 记录与追踪：`可用且相对完整`
- 历史详情回看：`可用，且已具备任务维度展示与任务卡片能力`
- 纪念卡册：`未产品化完成`
- 成就系统：`未实现`

如果下一阶段目标是“尽快达到稳定交付”，建议优先顺序如下：

1. 固化任务数据结构与后端存储协议。
2. 补齐任务核验入口与历史详情核验展示。
3. 把任务卡片从“前端临时生成”升级成“预览 + 持久化正式资源”双轨。
4. 补足上传、定位、录音、渲染失败的异常处理。
5. 为关键链路补测试与监控，再推进纪念卡册与成就系统。
