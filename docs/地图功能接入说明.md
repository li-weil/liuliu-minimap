## 2. 当前已经实现的功能

### 2.1 探索页

当前探索页已经实现：

- 当前位置获取
  - 使用微信 `wx.getLocation({ type: 'gcj02' })`
  - 已包含定位权限申请、设置引导、失败提示

- 页内地图展示
  - 使用微信官方 `map` 组件
  - 可展示当前探索点 Marker
  - 可展示探索范围圈层

- 地图拖动选点
  - 用户可直接拖动地图
  - 地图中心点可作为探索点来源
  - 点击“设为探索点”后正式生效

- 地点搜索
  - 可搜索地点或街区
  - 当前结果最多展示 20 条
  - 支持滚动查看更多结果

- 地点名称补全
  - 当前定位、搜索结果、拖图设点后都会做逆地理解析
  - 优先显示具体 POI 名称，不再优先显示门牌号

- 探索点周边 POI
  - 当前已恢复可用
  - 设定探索点后会自动拉取附近 POI
  - 用户也可以点击“展开”查看周边 POI 列表

- 地图与主题生成联动
  - 探索点确认后会更新：
    - `locationName`
    - `locationContext`
    - `locationAddress`
    - `latitude`
    - `longitude`
  - 后续可直接参与主题生成

### 2.2 记录页

当前记录页已经实现：

- 轨迹追踪开关
  - 点击可开启或停止追踪

- 定位采集
  - 使用前台持续定位
  - 失败时降级为 5 秒轮询一次

- 轨迹点记录
  - 每个点包含：
    - `latitude`
    - `longitude`
    - `timestamp`
    - `accuracy`

- 轨迹地图回显
  - 通过 `polyline` 绘制当前轨迹

- 轨迹统计
  - 展示时长
  - 展示里程估算
  - 展示开始与结束时间

### 2.3 云函数与数据链路

当前地图相关云函数已经包括：

- `getLocationContext`
  - 用于补充地点环境语义

- `fetchNearbyPois`
  - 用于获取探索点周边 POI
  - 当前由云函数代理请求高德 Web 服务

- `createWalk`
  - 用于保存漫步记录及轨迹信息

## 3. 当前技术方案说明

### 3.1 地图展示与交互

地图展示与交互主要使用：

- 微信 `map` 组件
- `MapContext.getCenterLocation()`
- `bindregionchange`

它负责：

- 地图渲染
- Marker / Circle / Polyline
- 拖动地图并读取中心点

### 3.2 当前定位与轨迹定位

定位能力主要使用微信原生接口：

- `wx.getLocation`
- `wx.startLocationUpdate`
- `wx.onLocationChange`

结论是：

- 当前定位不是高德定位
- 当前轨迹采集也不是高德定位
- 当前不使用后台持续定位
- 高德目前主要负责搜索与地理语义

### 3.3 地点搜索与逆地理

地点搜索与逆地理主要使用高德能力：

- 小程序 SDK：
  - `getInputtips`
  - `getRegeo`

- 高德 Web 服务：
  - `place/around`
  - `place/text`

当前搜索策略是：

- 优先高德周边 POI 搜索
- 补高德关键字 POI 搜索
- 最后再用 `getInputtips` 兜底

### 3.4 周边 POI 获取

当前周边 POI 链路是：

1. 小程序前端触发 `fetchNearbyPois`
2. 默认配置下走云函数
3. 云函数 `fetchNearbyPois` 请求高德 Web 服务 `place/around`
4. 云函数把结果归一化后返回前端

当前不是：

- 小程序前端直连高德
- 也不是走你们 Web 后端地图接口

而是：

- 前端 -> 云函数 -> 高德 Web 服务

### 3.5 配置方式

当前探索点周边 POI 所需的高德 Web 服务 key 配置在：

- `cloudfunctions/fetchNearbyPois/config.js`

当前联调阶段允许在该文件中直接写明文 key。

同时云函数也支持：

- `AMAP_WEB_KEY`

后续可切到环境变量。

## 5. 过程中遇到的主要问题

### 5.1 地点提示内容不够丰富

早期只依赖高德 `getInputtips` 时，搜索结果偏轻：

- 结果数量少
- 地点语义弱
- 更像输入提示，不像 POI 列表

后续已改为：

- POI 搜索优先
- `inputtips` 兜底

### 5.2 地点名称过度显示门牌号

早期地点命名更容易落到：

- `xx路xx号`

而不是：

- `中山大学`
- `广州塔`
- `太古汇`

后续已调整为：

- 优先采用逆地理最近 POI 名称
- 门牌号退为兜底信息

### 5.3 搜索结果数量不足

早期只展示少量结果，不利于用户选择大词场景下的目标地点。

后续已调整为：

- 最多 20 条
- 可滚动查看

### 5.4 搜索结果距离显示不稳定

尝试过在搜索结果右侧展示距离，但不同来源结果的距离信息不稳定，用户体验反而不统一。

当前处理为：

- 搜索结果去掉距离展示
- 保留更稳定的 POI 名称、区县、类型标签

### 5.5 周边 POI 原本依赖后端路由，默认配置下不可用

之前探索点周边 POI 依赖后端路由时，在当前默认配置下并不稳定。

后续已改为：

- 云函数直接代理高德 Web 服务

### 5.6 拖动地图后点击“设为探索点”，探索点会跳回旧位置

这是本轮接入过程中发现的明确 bug。

原因是：

- 确认探索点时，可能读取了旧缓存的地图中心点

后续修复为：

- 点击“设为探索点”时，再实时调用一次 `MapContext.getCenterLocation()`

### 5.7 周边 POI 失败时没有可见反馈

之前周边 POI 请求失败会被静默吞掉，用户感知为：

- 点了“展开”但没有反应

后续修复为：

- 改用完整 `wx.showModal`
- 展示完整错误信息

### 5.8 高德 key 类型问题

当前周边 POI 走的是高德 Web 服务，因此过程中遇到了典型的 key 问题：

- `USERKEY_PLAT_NOMATCH`
- `INVALID_USER_KEY`

结论已经明确：

- 小程序 SDK key 不能直接当 Web 服务 key 使用
- 周边 POI 云函数必须使用高德 Web 服务 key

## 7. 后续如何修改，才能稳健适用生产环境

### 7.1 配置收口

当前联调阶段，高德 Web 服务 key 可直接写在：

- `cloudfunctions/fetchNearbyPois/config.js`

生产环境建议改为：

- 去掉明文 key
- 改用云函数环境变量 `AMAP_WEB_KEY`

### 7.2 云函数部署规范

生产环境建议明确：

- 云函数部署流程
- 环境变量配置流程
- key 轮换流程
- 错误监控与日志查看方式

### 7.3 周边 POI 代理策略

当前“云函数代理高德”是合理方向，但生产环境建议继续补：

- 请求限流
- 失败重试
- 更明确的错误码映射
- 可选缓存能力

### 7.4 地图错误处理

生产环境建议：

- 所有地图关键请求都有完整可读错误提示
- 错误文案不要只保留技术堆栈
- 对常见错误类型建立统一说明

### 7.5 轨迹稳定性

生产环境建议优先提高：

- 轨迹精度过滤能力
- 弱网下的可恢复性
- 前台定位与轮询降级策略
- 轨迹记录中断后的可恢复策略

### 7.6 数据落库完整性

生产前建议确认：

- `locationAddress` 是否需要强制落库
- `locationName`、`locationContext`、`routePoints` 是否需要统一 DTO
- 历史记录与详情页读取字段是否完全一致

## 8. 当前联调建议

建议按这个顺序联调：

1. 先确认高德 Web 服务 key 是否正确
2. 部署云函数 `fetchNearbyPois`
3. 真机进入探索页
4. 点击“当前定位”
5. 拖动地图并设为探索点
6. 检查探索点名称是否为具体 POI
7. 检查周边 POI 是否可自动出现
8. 搜索大词验证结果列表是否足够丰富
9. 开始漫步并验证轨迹记录
