# 地图功能接入说明

## 1. 当前技术方案

当前小程序地图相关能力采用：

- 微信官方 `map` 组件：负责地图展示、Marker、Circle、Polyline、地图中心点读取
- 微信定位能力：负责当前位置获取与轨迹记录
- 高德小程序 SDK：负责地点搜索建议与逆地理解析

对应代码位置：

- 地图与定位权限封装：[miniprogram/utils/location.js](D:/liuliu-minimap/miniprogram/utils/location.js)
- 高德 SDK 封装：[miniprogram/utils/amap.js](D:/liuliu-minimap/miniprogram/utils/amap.js)
- 探索页地图选点与地点搜索：[miniprogram/pages/index/index.js](D:/liuliu-minimap/miniprogram/pages/index/index.js)
- 记录页轨迹记录：[miniprogram/pages/record/record.js](D:/liuliu-minimap/miniprogram/pages/record/record.js)
- 高德 SDK 文件：[miniprogram/libs/amap-wx.130.js](D:/liuliu-minimap/miniprogram/libs/amap-wx.130.js)
- 高德 Key 配置：[miniprogram/utils/config.js](D:/liuliu-minimap/miniprogram/utils/config.js)

## 2. 已经接入到哪一步

### 2.1 探索页

探索页目前已经完成以下能力：

- 获取当前位置
  - 使用 `wx.getLocation({ type: 'gcj02' })`
  - 已包含定位权限申请与失败提示

- 搜索地点
  - 使用高德 SDK `getInputtips`
  - 输入关键词后可返回地点候选列表

- 逆地理解析
  - 使用高德 SDK `getRegeo`
  - 选点后会补全地点名称、地址、区县等信息

- 页内地图选点
  - 已不再依赖 `wx.chooseLocation`
  - 当前实现为：用户在探索页直接拖动地图，页面通过 `map` 组件的 `bindregionchange` + `MapContext.getCenterLocation()` 获取地图中心点
  - 用户点击“设为探索点”后，将当前中心点写入页面状态，并更新探索地点信息

- 探索范围展示
  - 地图上已展示当前探索点 Marker
  - 已展示 3km 圈层，用于表达任务生成范围

### 2.2 记录页

记录页目前已经完成以下能力：

- 开启地图后切换到地图展示态
  - 顶部展示卡会切换成微信 `map`

- 轨迹记录
  - 点击“开启地图”后立即记录一次定位点
  - 后续通过定时器每 10 秒调用一次 `wx.getLocation`
  - 每个定位点写入 `draft.routePoints`

- 轨迹回显
  - 已通过 `polyline` 把 `routePoints` 画在地图上

- 停止追踪
  - 再次点击按钮后停止定时记录
  - 顶部区域恢复为原本的主题展示卡

### 2.3 数据流现状

地图相关数据目前已接入到主流程：

- 探索页确认地点后，会写入：
  - `locationName`
  - `locationContext`
  - `locationAddress`
  - `latitude`
  - `longitude`

- 开始漫步时，这些字段会写入 `walkDraft`

- 记录页追踪时会补充：
  - `routePoints`

- 保存漫步时，这些信息会一并传入创建记录接口：
  - `locationName`
  - `locationContext`
  - `locationAddress`
  - `routePoints`

## 3. 当前实现的边界

当前地图功能已经达到“可用的 MVP”状态，但还没有到完整产品化阶段。

### 3.1 已解决的问题

- 不再是“只有搜索没有真地图”
- 不再是“地图选点按钮只是系统占位”
- 探索页已经具备真实页内地图选点
- 记录页已经具备真实轨迹采集与回显

### 3.2 还未完善的部分

- 探索页地图选点目前是“拖地图中心点 + 点击确认”
  - 还没有做“点击地图某一点直接落点”
  - 还没有做“拖动结束后自动刷新地点信息”，目前仍是手动点击“设为探索点”后才正式生效

- 搜索结果和地图联动还比较基础
  - 当前是“搜索结果 -> 选中 -> 更新地图”
  - 还没有做“地图拖动后自动反查附近 POI 列表”

- 记录页轨迹记录还是轻量实现
  - 当前使用前台定时轮询定位
  - 没有做轨迹点去噪、抽稀、异常漂移过滤
  - 没有做暂停/恢复、后台持续记录、弱网补偿

- 地图相关数据还没有和 Web 后端接口文档完全对齐
  - 当前只是前端本地状态和保存接口侧准备好了字段
  - 后续还需要根据 Spring Boot 接口文档确认字段名、数据结构、是否需要额外 DTO 转换

## 4. 当前更适合怎么理解

可以把当前状态理解为：

- 探索页地图：已经从“假接入”升级为“真接入，可用”
- 记录页轨迹：已经从“视觉占位”升级为“真记录，可用”
- 整体阶段：适合真机联调、功能演示、继续对接后端，不适合直接定义为最终版

## 5. 建议下一步怎么做

建议按下面顺序继续推进。

### 第一步：做真机联调与验收

先确认当前 MVP 在真机上稳定可用：

- 探索页
  - 拖动地图后是否能正确设为探索点
  - 搜索地点后地图是否跟随更新
  - 当前位置是否能正常回填
  - 生成任务时是否正确携带地点信息

- 记录页
  - 开启地图后是否立刻记录首个点
  - 行走一段后是否能看到 polyline
  - 停止地图后是否能恢复原展示卡
  - 保存记录后轨迹是否被正常写入

### 第二步：补齐探索页地图体验

优先建议补这几个点：

- 支持点击地图直接设点
- 拖动地图结束后自动反查地点名称
- 根据中心点返回附近 POI，提升地点语义质量
- 在“地图选点”状态下增加更明确的交互提示

### 第三步：补齐记录页轨迹能力

建议补：

- 轨迹点过滤与抽稀
- 异常漂移点剔除
- 开始/停止时间记录
- 更完整的轨迹统计信息
  - 总时长
  - 总点数
  - 里程估算

### 第四步：对接 Web 后端接口文档

等 Spring Boot 接口文档到位后，优先确认：

- 地点字段命名是否一致
  - `locationName`
  - `locationContext`
  - `locationAddress`
  - `latitude`
  - `longitude`

- 轨迹字段格式
  - `routePoints` 是否允许直接传对象数组
  - 是否需要压缩、转字符串或转 DTO

- 多媒体与轨迹是否分接口上传
- 保存漫步记录时是否需要额外的地图元数据

## 6. 推荐的后续改造方向

如果后续要做成更完整的正式版，建议目标是：

- 探索页：
  - 搜索、选点、地图、POI 推荐一体化

- 记录页：
  - 更稳定的轨迹记录
  - 轨迹统计
  - 轨迹与照片/视频时间点联动

- 后端：
  - 明确轨迹 DTO
  - 支持漫步记录与地图数据统一存储

## 7. 一句话结论

当前地图功能已经不是占位，而是已经完成了：

- 探索页真实地图选点
- 地点搜索与逆地理解析
- 记录页轨迹记录与地图回显

但当前仍属于可联调、可演示、可继续对接的 MVP 版本，下一步重点应放在真机验收、交互补强和后端接口对齐。
