# 地图功能接入说明

## 1. 文档目的

本说明记录小程序地图能力的现状，主要回答五个问题：

- 已经实现哪些地图功能
- 技术方案由哪些能力组成
- 实现进度到哪里
- 在接入和联调过程中遇到了哪些问题
- 下一步怎样补齐生产环境需要的稳定性

## 2. 已实现功能

### 2.1 探索页

探索页已支持：

- 当前位置获取
  - 使用微信 `wx.getLocation({ type: 'gcj02' })`
  - 已包含定位权限申请、设置引导、失败提示

- 页内地图展示
  - 使用微信官方 `map` 组件
  - 可展示当前探索点 Marker
  - 可展示探索范围圈层

- 地图浏览
  - 用户可在地图上浏览、缩放、旋转当前探索点周边
  - 地图只负责展示和浏览，不再提供地图中心取点入口
  - 探索点来源为当前定位、地点搜索结果和附近地点推荐

- 地点搜索
  - 可搜索地点或街区
  - 当前结果最多展示 20 条
  - 支持滚动查看更多结果

- 地点名称补全
  - 当前定位、搜索结果、附近地点切换后都会做逆地理解析
  - 优先显示具体 POI 名称，不再优先显示门牌号

- 探索点周边 POI
  - 已恢复可用
  - 设定探索点后会通过高德逆地理 `getRegeo()` 补充附近地点推荐
  - 用户也可以点击“展开”查看逆地理返回的附近地点列表

- 地图与主题生成联动
  - 探索点确认后会更新：
    - `locationName`
    - `locationContext`
    - `locationAddress`
    - `latitude`
    - `longitude`
  - 可直接参与主题生成

### 2.2 记录页

记录页已支持：

- 轨迹追踪开关
  - 点击可开启或停止追踪

- 定位采集
  - 使用前台持续定位
  - 失败时降级为 5 秒轮询一次

- 轨迹点记录
  - 每个点包含：
    - `latitude`
    - `longitude`
    - `timestamp`
    - `accuracy`

- 轨迹地图回显
  - 通过 `polyline` 绘制当前轨迹

- 轨迹统计
  - 展示时长
  - 展示里程估算
  - 展示开始与结束时间

### 2.3 云函数与数据链路

地图相关云函数包括：

- `getLocationContext`
  - 用于补充地点环境语义

- `fetchNearbyPois`
  - 保留为云函数 / Web 后端周边 POI 代理能力
  - 当前探索页主链路没有直接调用它，附近地点推荐主要来自 `getRegeo()` 返回的 `pois`

- `createWalk`
  - 用于保存漫步记录及轨迹信息

## 3. 技术方案

### 3.1 地图展示与交互

地图展示与交互主要使用：

- 微信 `map` 组件

它负责：

- 地图渲染
- Marker / Circle / Polyline
- 展示当前探索点和探索范围

### 3.2 当前定位与轨迹采集

定位能力主要使用微信原生接口：

- `wx.getLocation`
- `wx.startLocationUpdate`
- `wx.onLocationChange`

结论：

- 当前位置和轨迹采集都使用微信定位，不使用高德定位
- 不使用后台持续定位
- 高德目前主要负责搜索与地理语义

### 3.3 地点搜索与逆地理

地点搜索与逆地理主要使用高德能力：

- 小程序 SDK：
  - `getInputtips`
  - `getRegeo`

- 高德 Web 服务：
  - `place/around`
  - `place/text`

搜索策略：

- 优先高德周边 POI 搜索
- 补高德关键字 POI 搜索
- 最后再用 `getInputtips` 兜底

### 3.4 附近地点推荐

当前探索页附近地点推荐流程：

1. 小程序前端在确认探索点后调用高德小程序 SDK 的 `getRegeo()`
2. 从逆地理结果中读取 `regeocode.pois`
3. 前端归一化为附近地点推荐列表
4. 生成主题前也会优先使用这份附近地点缓存

代码中仍保留 `fetchNearbyPois`：

- `miniprogram/services/map.js` 导出 `fetchNearbyPois()`
- `miniprogram/services/api.js` 已配置 `/map/pois/nearby` 和云函数兜底
- `cloudfunctions/fetchNearbyPois` 可以作为后续切回后端 POI 代理的能力

但按当前代码，探索页没有直接调用 `fetchNearbyPois`，因此不要把它写成当前附近地点推荐的主链路。

### 3.5 配置方式

保留的 `fetchNearbyPois` 云函数所需高德 Web 服务 key 配置在：

- `cloudfunctions/fetchNearbyPois/config.js`

联调阶段可以在该文件中直接写明文 key。

同时云函数也支持：

- `AMAP_WEB_KEY`

上线前建议切到环境变量。

## 4. 实现进度

### 4.1 当前完成度判断

地图功能已经可以：

- 可真机联调
- 可功能演示
- 可继续对接业务主链路

但还需要补：

- 完整产品化
- 稳健生产化

### 4.2 探索页完成度

探索页已经从占位功能变成真实可用：

- 不是单纯搜索框占位
- 不是只有假地图或静态图
- 已具备真实地图展示与探索点范围回显
- 已具备真实地点搜索
- 已具备探索点附近地点推荐展示

但体验还需要继续打磨：

- 交互可用
- 细节还可优化

### 4.3 记录页完成度

记录页已经可以真实记录轨迹：

- 可以真实记录路线
- 可以真实回显轨迹
- 可以计算基本轨迹统计

但还没有做：

- 去噪
- 抽稀
- 漂移过滤
- 暂停 / 恢复
- 更稳定的前台记录策略

### 4.4 数据链路完成度

地图数据已经接入主流程：

- 探索页确认地点后，可写入地点信息
- 开始漫步时，可写入 `walkDraft`
- 记录页可持续补充 `routePoints`
- 保存时可提交地点与轨迹信息

当前地图字段已覆盖：

- `locationName`
- `locationContext`
- `locationAddress`
- `latitude`
- `longitude`
- `routePoints`

## 5. 过程中遇到的主要问题

### 5.1 地点提示内容不够丰富

早期只依赖高德 `getInputtips` 时，搜索结果偏轻：

- 结果数量少
- 地点语义弱
- 更像输入提示，不像 POI 列表

现在改为：

- POI 搜索优先
- `inputtips` 兜底

### 5.2 地点名称过度显示门牌号

早期地点命名更容易落到：

- `xx路xx号`

而不是：

- `中山大学`
- `广州塔`
- `太古汇`

现在改为：

- 优先采用逆地理最近 POI 名称
- 门牌号退为兜底信息

### 5.3 搜索结果数量不足

早期只展示少量结果，不利于用户选择大词场景下的目标地点。

现在改为：

- 最多 20 条
- 可滚动查看

### 5.4 搜索结果距离显示不稳定

尝试过在搜索结果右侧展示距离，但不同来源结果的距离信息不稳定，用户体验反而不统一。

当前处理为：

- 搜索结果去掉距离展示
- 保留更稳定的 POI 名称、区县、类型标签

### 5.5 周边 POI 代理不是当前主链路

之前探索点周边 POI 依赖后端路由时，在当前默认配置下并不稳定。

现在改为：

- 探索页主链路优先使用高德逆地理 `getRegeo()` 里的 `pois`
- `fetchNearbyPois` 云函数继续保留，但当前不作为探索页附近地点推荐入口

### 5.6 地图中心设点入口已下线

早期探索页曾尝试支持拖动地图后把地图中心设为探索点。现在保留三种来源：

- 当前定位
- 地点搜索
- 附近地点推荐

代码不再通过 `MapContext.getCenterLocation()` 读取地图中心点，也不再保留“设为探索点 / 确认这里”入口。地图仍可滚动、缩放和旋转，但只用于浏览已选探索点周边。

### 5.7 附近地点失败反馈

手动展开附近地点时，如果请求失败，用户需要看到明确反馈。当前自动补充附近地点的路径仍会静默降级为空列表，避免阻断定位和主题生成。

用户感知风险主要是：

- 点了“展开”但没有反应
- 生成主题时附近地点为空，但主流程仍继续

当前处理为：

- 手动展开失败时弹出 `wx.showModal`
- 自动补充失败时降级为空列表

### 5.8 高德 key 类型问题

保留的 `fetchNearbyPois` 云函数走高德 Web 服务，因此过程中遇到了典型的 key 问题：

- `USERKEY_PLAT_NOMATCH`
- `INVALID_USER_KEY`

结论已经明确：

- 小程序 SDK key 不能直接当 Web 服务 key 使用
- 如果后续启用周边 POI 云函数，必须使用高德 Web 服务 key

## 6. 建议进一步改进的地方

### 6.1 探索页体验

建议补：

- 探索点与周边 POI 一体化展示
- 搜索结果继续加载或分页
- 更细粒度的 POI 类型筛选

### 6.2 地点语义质量

建议补：

- 更稳定的 POI 命名策略
- 探索点主标题与详细地址分开显示
- 周边 POI 与探索点的语义关联更清晰

### 6.3 轨迹质量

建议补：

- 轨迹去噪
- 精度过滤
- 轨迹抽稀
- 漂移点剔除
- 暂停 / 恢复

### 6.4 数据一致性

建议补：

- `createWalk` 落库 `locationAddress`
- 明确前后端地图字段命名
- 明确轨迹 DTO 结构

## 7. 上线前建议

### 7.1 配置收口

联调阶段，高德 Web 服务 key 可以写在：

- `cloudfunctions/fetchNearbyPois/config.js`

生产环境建议改为：

- 去掉明文 key
- 改用云函数环境变量 `AMAP_WEB_KEY`

### 7.2 云函数部署规范

生产环境建议明确：

- 云函数部署流程
- 环境变量配置流程
- key 轮换流程
- 错误监控与日志查看方式

### 7.3 周边 POI 代理策略

“云模式走云函数代理高德、Web 模式走后端地图接口”可以作为后续 `fetchNearbyPois` 代理策略保留。当前探索页主链路仍是 `getRegeo()`。上线前如果要重新启用代理，需要补：

- 请求限流
- 失败重试
- 更明确的错误码映射
- 可选缓存能力

### 7.4 地图错误处理

生产环境建议：

- 所有地图关键请求都有完整可读错误提示
- 错误文案不要只保留技术堆栈
- 对常见错误类型建立统一说明

### 7.5 轨迹稳定性

生产环境建议优先提高：

- 轨迹精度过滤能力
- 弱网下的可恢复性
- 前台定位与轮询降级策略
- 轨迹记录中断后的可恢复策略

### 7.6 数据落库完整性

生产前建议确认：

- `locationAddress` 是否需要强制落库
- `locationName`、`locationContext`、`routePoints` 是否需要统一 DTO
- 历史记录与详情页读取字段是否完全一致

## 8. 当前联调建议

建议按这个顺序联调：

1. 真机进入探索页
2. 确认高德小程序 SDK key 可用于 `getRegeo()`
3. 如果要联调 `fetchNearbyPois` 代理，再确认高德 Web 服务 key 并部署云函数
4. 点击“当前定位”
5. 用搜索结果或附近地点切换探索点
6. 检查探索点名称是否为具体 POI
7. 检查周边地点推荐是否可自动出现
8. 搜索大词验证结果列表是否足够丰富
9. 进入生成步骤，生成主题并开始漫步

## 9. 一句话结论

当前地图功能已经完成了：

- 真实地图展示
- 当前位置获取
- 地点搜索
- 探索点具体名称补全
- 附近地点推荐
- 记录页轨迹记录与回显

当前阶段可以定义为：

- 可联调
- 可演示
- 可继续迭代

但距离稳健生产环境还需要继续补齐：

- 配置安全性
- 错误处理
- 轨迹质量
- 数据完整落库
