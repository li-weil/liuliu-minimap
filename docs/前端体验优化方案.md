# 前端体验优化方案

## 1. 文档目的

本文记录小程序前端体验优化方案，重点解决三个问题：交互生硬、页面层级分散、状态反馈不够明确。

本文不讨论后端，也不推翻现有业务流程。优化会继续基于原生微信小程序结构，目标是：

- 让“探索 -> 生成主题 -> 开始漫步 -> 任务打卡 -> 足迹回看”的主流程更顺
- 让用户操作后的反馈更明确、更有触感
- 让页面从“功能卡片堆叠”升级为更有城市漫步氛围的体验
- 让视觉系统、交互状态和动效规则可复用
- 在不大改业务逻辑的前提下，优先通过 `wxml / wxss` 和少量状态类完成优化

本方案参考 `Redesign Existing Projects` 和 `Stitch Design Taste` 的方法，但不会套用网页式炫技设计。所有改动都以当前小程序的产品形态为准。

## 2. 前端现状

小程序已经具备比较完整的功能基础：

- 探索页支持纯粹模式、进阶模式、单人模式、同行模式
- 探索页支持搜索地点、当前定位、周边地点推荐，并用地图展示已选探索点与范围圈
- 支持 AI 生成主题、任务和生成过程调试
- 记录页支持位置显示、任务打卡、图片 / 视频 / 音频 / 文字素材
- 足迹页支持纪念卡册、成就、状态筛选和类型筛选
- 个人页支持登录、注册、资料编辑
- 全局已经统一使用 `#f5f2ed` 作为主背景基色
- 已经存在基础卡片、按钮、胶囊、输入框、空状态和生成 loading 状态

也就是说，前端不是“没有设计”，而是已经完成了基础可用版本。

下一步不是重做 UI，而是把现有界面从“功能能跑”推进到“操作舒服、状态自然、氛围统一”。

## 3. 主要问题

### 3.1 页面层级偏平均

目前探索页、记录页、足迹页大量使用类似的白色圆角卡片：

- `hero-card`
- `section-card`
- `theme-card`
- `walk-card`
- `mission-card`

这些卡片本身都可读，但同级容器太多后，用户很难判断当前最重要的操作在哪里。

典型表现：

- 探索页里搜索、定位、地图、主题选择、生成按钮、主题结果都像同级模块
- 记录页里主题、地图开关、定位信息、任务打卡、保存按钮的主次关系不够强
- 足迹页里顶部说明、分栏、筛选、列表都偏功能面板化

### 3.2 胶囊按钮使用过多

很多选择项和按钮都使用大圆角胶囊形态，包括：

- 模式切换
- 主题选择
- 状态筛选
- 类型筛选
- meta 信息
- 次级操作按钮

胶囊按钮在少量场景里很轻盈，但过多使用会带来两个问题：

- 所有可点击元素都长得很像，操作优先级不够明确
- 页面显得“软”和“散”，缺少稳定的行动锚点

### 3.3 操作反馈偏硬

许多操作主要靠文案变化、loading 状态或内容切换来反馈。

需要补足的反馈包括：

- 点击按钮的按压感
- chip 选择的过渡感
- 任务展开 / 收起的流动感
- 搜索结果出现时的入场感
- 地图拖动和确认选点的状态反馈
- 生成主题从等待到成功的节奏感
- 保存进度和完成状态的安全感

### 3.4 生成体验还可以更有仪式感

`theme-card` 里已经有生成进度条、步骤和 skeleton，是一个很好的基础。

但生成过程和结果之间的连接还可以更自然：

- 生成前用户需要更清楚地知道“我已经准备好了哪些条件”
- 生成中需要感觉系统正在“读取街区、整理线索、生成任务”
- 生成后需要让主题卡更像一张可以带出门的“城市任务票据”

### 3.5 记录页的任务打卡偏表单化

记录页的任务打卡能力完整，但展开后的内容比较重：

- 三个素材按钮
- 文字输入
- 素材预览
- 视频 / 音频列表
- 打卡按钮
- 生成卡片按钮

这些能力都必要，但视觉上需要更像“当前任务操作台”，而不是一个突然展开的大表单。

### 3.6 足迹页偏管理后台

足迹页目前的筛选能力清晰，但“状态 + 类型 + 数量”的筛选区略像管理列表。

对用户来说，足迹页更应该像：

- 一本纪念卡册
- 一面成就墙
- 一组走过城市的收藏

因此足迹页需要减少工具感，增强回看感和收藏感。

## 4. 设计方向

建议将整体前端体验定义为：

> 温暖纸感、城市线索、轻量仪表、柔和触感、克制动效。

这套方向有几个关键判断：

- 不做过度玻璃拟态，避免和 city walk 的温度感冲突
- 不做大面积霓虹渐变，避免 AI 生成感过重
- 不做网页式大 hero，保持小程序工具效率
- 不做过重社交信息流，优先服务“出门走一段”
- 保留当前暖色、橄榄色、纸张感的基础，但统一层级和状态

更具体地说，界面应该像一本反应灵敏的随身城市手册：

- 它知道你现在在哪
- 它帮你抽出一条线索
- 它陪你完成任务
- 它把结果收进足迹册

## 5. 设计系统建议

### 5.1 色彩

建议收敛为一套稳定的暖中性色系统。

基础色：

- 页面背景：`#F4EFE6`
- 次级背景：`#EFE7DA`
- 主表面：`#FFFDF8`
- 柔和表面：`#FAF4EA`
- 主文字：`#241F19`
- 正文文字：`#3A332B`
- 次级文字：`#746B5F`
- 弱文字：`#9A9082`
- 结构线：`rgba(126, 111, 89, 0.16)`

强调色：

- 主强调：`#626247`
- 主强调深色：`#3F402F`
- 主强调浅底：`rgba(98, 98, 71, 0.10)`

状态色：

- 成功：`#66704A`
- 警示：`#B88445`
- 错误：`#A55449`
- 信息：`#60717A`

使用规则：

- 全局最多只保留一个主要强调色，即橄榄色系
- 红色只用于危险、错误、删除，不用于普通装饰
- 蓝色只用于地图、定位或信息提示，不参与主按钮体系
- 不使用纯黑 `#000000`
- 不使用高饱和紫蓝渐变

### 5.2 字体与字号

小程序中文字体受系统限制较多，不建议强依赖外部字体。

建议采用系统字体，但建立更明确的字号层级：

- 页面主标题：`42rpx - 48rpx`，字重 `600 / 700`
- 模块标题：`32rpx - 38rpx`，字重 `600`
- 正文：`28rpx`，行高 `1.65 - 1.75`
- 辅助文案：`24rpx`，行高 `1.6 - 1.7`
- 标签 / 元信息：`20rpx - 22rpx`
- 数字统计：可使用系统等宽字体或开启 tabular number 思路

文案规则：

- 标题尽量使用短句，不使用“探索页”这类页面名当主标题
- 辅助文案要解释当前状态，不写泛泛的氛围句
- 按钮文案要直接表达动作，例如“下一步”“生成漫步主题”“开始这次漫步”
- 错误文案要直接说明问题和下一步，不使用“哎呀”“Oops”这类语气

### 5.3 圆角

当前大圆角使用较多，建议分层收敛：

- 页面主卡片：`32rpx`
- 次级面板：`28rpx`
- 输入框：`24rpx - 28rpx`
- 小标签：`18rpx - 999rpx`，按语义区分
- 图片 / 地图容器：`28rpx - 32rpx`
- 小按钮：`22rpx - 26rpx`

规则：

- 不再让所有容器都使用 `40rpx` 以上大圆角
- 大容器和内部按钮不要完全同圆角
- 元信息标签可以保持胶囊，但行动按钮不要全部胶囊化

### 5.4 阴影与边线

建议减少“边框 + 阴影 + 白底”三件套同时出现。

主卡片：

- 可使用浅阴影加弱边线
- 阴影颜色使用暖色：`rgba(77, 62, 38, 0.08)`

次级面板：

- 优先使用背景色和内描边
- 少用外阴影

列表项：

- 优先使用分隔线、留白和轻微背景变化
- 不强行每一项都做成卡片

### 5.5 按钮

按钮分为四类：

主按钮：

- 用于当前最重要动作
- 背景为主强调色
- 圆角为 `26rpx - 32rpx`
- 高度不低于 `88rpx`
- 点击有下压反馈

次按钮：

- 用于辅助操作
- 背景为浅暖色
- 文字为主文字或主强调
- 不与主按钮抢层级

文本按钮：

- 用于清除筛选、展开、收起、取消等低风险动作
- 不强行放入胶囊

危险按钮：

- 只用于删除、退出登录、取消重要状态
- 使用低饱和红色
- 默认不要和主按钮同视觉重量

### 5.6 Chip 与标签

chip 只保留在选择和筛选场景：

- 模式切换
- 主题方向选择
- 足迹筛选
- 状态摘要

规则：

- active 状态要清楚，但不要全部变成强色块
- chip 点击要有轻微缩放反馈
- chip 数量多时要有分组、摘要或收起能力
- 元信息标签和选择 chip 视觉上要区分，避免用户误以为都能点

## 6. 全局交互规则

### 6.1 点击反馈

所有可点击组件至少具备以下反馈：

- `transition-property: transform, opacity, background-color, box-shadow`
- `transition-duration: 160ms - 220ms`
- `transition-timing-function: cubic-bezier(0.22, 1, 0.36, 1)`
- active 状态：`transform: scale(0.98)` 或 `translateY(1rpx)`

覆盖范围：

- button
- mode chip
- filter chip
- walk card
- achievement badge
- nearby place item
- search result item
- mission row
- media remove button

### 6.2 页面入场

核心页面的首屏模块建议使用轻量入场：

- 初始 `opacity: 0`
- 初始 `transform: translateY(16rpx)`
- 进入后 `opacity: 1; transform: translateY(0)`
- 动画时长 `280ms - 420ms`

不建议：

- 大幅度飞入
- 无限弹跳
- 过多模块同时动画
- 动画阻塞用户操作

### 6.3 列表入场

列表类内容可以做轻微递进：

- 搜索结果
- 周边地点
- 足迹卡片
- 成就徽章
- 任务列表

递进间隔建议 `30ms - 60ms`。

如果实现成本高，可以先只做统一 fade-in，不强制 stagger。

### 6.4 展开与收起

任务面板、筛选面板、周边地点列表、调试面板都属于展开收起场景。

建议：

- 内容出现时使用 `opacity + translateY`
- 展开后的内部区域要有明确背景和边界
- 收起按钮与展开入口位置保持稳定

小程序里如果 `height auto` 动画不好控制，可以先用条件渲染和入场动画，不强求高度渐变。

### 6.5 Loading 状态

不建议只依赖系统 loading。

应该按场景区分：

- 搜索地点：按钮 loading + 结果区域 skeleton
- 周边推荐：列表 skeleton
- AI 生成：进度条 + 阶段步骤 + 任务 skeleton
- 保存记录：上传进度条 + 当前处理数量
- 成就加载：徽章 skeleton 或轻量占位

### 6.6 空状态

空状态必须给下一步，而不是只告诉用户没有数据。

例如：

- 足迹为空：提示去探索页开始第一次漫步
- 成就为空或未点亮：展示最近可尝试的成就方向
- 搜索无结果：提示换关键词或先使用当前定位
- 周边地点为空：允许直接以当前点生成主题

## 7. 探索页优化方案

涉及文件：

- `miniprogram/pages/index/index.wxml`
- `miniprogram/pages/index/index.wxss`
- `miniprogram/components/theme-card/theme-card.wxml`
- `miniprogram/components/theme-card/theme-card.wxss`

### 7.0 已实现内容

探索页已经落地页内两步式结构：

- 第一步设定起点，提供当前定位、地点搜索、附近地点推荐和地图浏览。
- 地图只展示已选探索点 Marker 与 3 公里范围圈，不提供地图中心确认入口。
- 用户设定探索点后仍停留在第一步，可继续调整地点；点击“下一步”或顶部“生成任务”进入第二步。
- 第二步展示探索点摘要、单人/同行、纯粹/进阶、主题方向和生成按钮。
- 纯粹模式生成 1 个任务；进阶模式生成 3 个任务。
- “随机”只是主题方向选项，前端抽取实际方向后继续走 `generateTheme`。
- 进阶模式选 2 个固定方向时走 `generateCombinedTheme`。
- 生成结果由 `theme-card` 以任务票据呈现，生成中展示三阶段进度。

下一步建议：

- 将 `location-flow` 和 `generate-panel` 拆成轻组件，降低 `index.wxml / index.js` 体积。
- 为搜索空状态补一个更明确的“使用当前定位”动作。
- 在微信开发者工具中补窄屏与主流宽度截图验收，重点看搜索行、当前定位按钮和任务票据长文本。

### 7.1 页面定位

探索页是主流程的起点，不应只是功能入口集合。

它的核心任务是：

1. 帮用户确定从哪里出发
2. 帮用户选择这次漫步的生成条件
3. 给出一张可以立即带走的主题任务卡

### 7.2 首屏结构调整

探索页不新增独立页面，采用页内两步式结构。

这样既能减少同屏拥挤感，又能保持 tabBar 里的“探索”仍然是主入口，避免用户在多个页面之间来回跳转。

新的页面结构是：

1. 第一步：设置探索点
2. 第二步：个性化选择与任务生成

页面结构建议：

```text
探索页

[顶部状态区]
今天从哪里开始
当前步骤：设置探索点 / 生成线索

[步骤切换条]
1 探索点
2 生成线索

Step 1：设置探索点
- 搜索地点
- 当前定位 / 地点搜索
- 地图主舞台
- 周边地点推荐
- 附近地点推荐

Step 2：生成线索
- 已选探索点摘要
- 出发方式：单人 / 同行
- 模式：纯粹 / 进阶
- 个性化条件
- 生成按钮
- 主题任务卡
```

### 7.3 两步式流程规则

两步式流程不是简单把内容上下分开，而是要让每一步只服务一个核心任务。

Step 1：设置探索点

- 用户只处理“从哪里开始”
- 页面重点是搜索、地图、附近地点和确认探索点
- 不出现天气、季节、状态、偏好等生成条件
- 不显示主题生成按钮，避免用户还没选点就开始思考任务

Step 2：生成线索

- 用户只处理“这次怎么走”
- 页面重点是单人 / 同行、纯粹 / 进阶、主题方向和个性化条件
- 不直接展示大地图，只展示已选探索点摘要
- 用户可以随时返回 Step 1 更换探索点

步骤进入规则：

- 初次进入探索页，若没有探索点，默认停留在 Step 1
- 已有探索点时，默认可以停留在 Step 2，但顶部仍显示地点摘要
- 用户在 Step 1 通过定位、搜索或附近推荐设定探索点后，仍停留在 Step 1；点击“下一步”或顶部“生成任务”进入 Step 2
- 用户在 Step 2 点击“更换地点”后，回到 Step 1
- 如果探索点被清空，生成按钮回到 disabled 状态

### 7.4 顶部状态区

顶部状态区需要承担“用户现在在哪一步”的提示作用。

建议内容：

- 标题从“探索页”改为“今天从哪里开始”
- 副标题根据步骤变化
- 显示当前探索点状态
- 显示两步流程进度

Step 1 副标题：

- `先选一个街区，66 会围绕这里整理今天的城市线索`

Step 2 副标题：

- `围绕这个探索点，选择今天想要的漫步方式`

步骤切换条建议：

- 使用两个相邻的步骤块，而不是普通 tab
- Step 1 文案：`1 探索点`
- Step 2 文案：`2 生成线索`
- Step 1 完成后显示已完成状态
- Step 2 在未设置探索点时禁用或弱化

### 7.5 Step 1：设置探索点

Step 1 承载探索页里的地图相关能力。

保留能力：

- 搜索输入框作为主要入口
- 当前定位
- 地图展示已选探索点
- 搜索结果
- 周边地点推荐
- 附近地点推荐

建议布局：

```text
[搜索区]
搜索地点或街区
当前定位

[地图主舞台]
大地图
已选探索点 marker
探索范围

[底部选点面板]
当前探索点 / 搜索选中点
附近地点推荐
下一步
```

地图主舞台：

- 地图高度建议提升到 `420rpx - 480rpx`
- 不恢复地图中心确认入口；地图下方展示探索点信息、当前定位、搜索和附近地点推荐
- 地图可保留滚动、缩放、旋转浏览能力，但不把地图中心作为探索点来源
- 地图区域尽量减少装饰层，保证拖动流畅

底部选点面板：

- 未选择地点时提示“先定位、搜索，或从附近地点里选一个”
- 已识别地点时显示地点名、区县、地址或周边摘要
- 点击周边地点后，地图展示点和当前探索点同步更新
- 主按钮为 `下一步`
- 设置成功后不自动切换到 Step 2，用户可以继续调整地点

### 7.6 Step 1 搜索结果

搜索结果建议从普通列表升级为“候选地点面板”。

出现方式：

- 用户输入并搜索后，搜索结果覆盖底部选点面板
- 搜索结果面板不遮住全部地图，保留地图上下文
- 用户点选结果后，结果面板收起，地图移动到该点

搜索结果内容：

- 面板顶部显示结果数量和当前关键词
- 每条结果显示名称、区县、类型
- 点击条目时有下压反馈
- 列表支持垂直滚动，但高度要受控

空状态：

- 搜索无结果时显示“没有找到这个地点，可以换个关键词，或先用当前定位”

### 7.7 Step 1 周边地点推荐

周边地点推荐不应抢主流程。

建议：

- 默认收起，只显示一行摘要，例如 `看看附近还有哪些地点`
- 探索点变化后更新推荐
- 展开后以轻列表展示
- 选择某个周边地点后，地图展示点和探索点同步变化
- 设置探索点后，周边推荐不再继续占用 Step 2 空间

视觉上不要再做厚重卡片，可以做成地图卡片内部的浅色面板。

### 7.8 Step 2：生成线索

Step 2 承载个性化选择和任务生成。

建议布局：

```text
[探索点摘要卡]
已选地点
周边 3 公里
更换地点

[出发方式]
单人 / 同行

[模式]
纯粹 / 进阶

[个性化条件]
纯粹：主题方向
进阶：天气 / 季节 / 状态 / 偏好 / 主题方向

[生成按钮]

[主题任务卡]
```

探索点摘要卡：

- 已设置时显示地点名、区域、来源方式
- 未设置时显示“还没有探索点”
- 右侧提供 `更换地点`
- 点击后回到 Step 1
- 不再展示大地图

这样 Step 2 会明显减负，用户只需要围绕已选地点做生成选择。

### 7.9 Step 2 纯粹模式

纯粹模式应该更轻，适合快速出发：

- 主题方向只允许单选
- 主题 chip 只展示一行或两行
- “随机生成”和“选择生成”视觉上要区分

建议：

- 主按钮为“生成主题”
- 次按钮为“换一个方向”
- 随机能力可以融入按钮旁的小动作，而不是两个同重量大按钮

如果仍保留两个按钮：

- `选择生成` 使用主按钮
- `随机生成` 使用次按钮

### 7.10 Step 2 进阶模式

进阶模式信息量较大，建议将天气、季节、状态、偏好做成“条件面板”。

默认状态：

- 显示当前摘要：`晴朗 / 春天 / 发呆 / 市井烟火`
- 显示“调整条件”入口

展开状态：

- 分组显示各类 chip
- 每组之间留出明确间距
- 主题方向支持 1 到 2 个选择，独立成组

这样可以降低进阶模式首屏压力。

### 7.11 生成按钮状态

生成按钮至少有四个状态：

- disabled：未设置探索点
- ready：可生成
- loading：正在生成
- error：生成失败，可重试

文案建议：

- disabled：`先设定探索点`
- ready：`生成今天的线索`
- loading：`正在整理街区线索`
- error：`重新生成`

在两步式流程下，disabled 状态需要更明确：

- 如果用户在 Step 2 但没有探索点，按钮文案为 `先设置探索点`
- 点击 disabled 区域不应静默，可以轻提示“先完成第 1 步”
- 如果探索点已设置但条件不完整，再提示具体缺少什么

### 7.12 Step 切换动效

Step 1 和 Step 2 的切换要轻，不要像换页面。

建议：

- 当前 step 内容 `opacity: 1; transform: translateX(0)`
- 离开 step 内容 `opacity: 0; transform: translateX(-16rpx / 16rpx)`
- 动画时长 `220ms - 320ms`
- 切换时保留顶部状态区，减少页面跳动

注意：

- 地图组件在小程序里比较特殊，隐藏 / 显示可能有渲染成本
- 如果 `wx:if` 导致地图频繁重建，可以考虑 Step 1 使用 `hidden` 或 class 控制显示
- 切到 Step 2 时，地图区域不需要保留在 DOM 上方占位
- 如果后续发现地图隐藏后回显异常，再考虑将地图抽成组件并做显式重建

### 7.13 Step 数据状态

探索页建议显式维护以下状态：

- `exploreStep`：`location` 或 `generate`
- `hasConfirmedExplorePoint`：是否已经确认探索点
- `locationName`：当前探索点名称
- `latitude / longitude`：探索点坐标
- `searchResults`：搜索结果
- `nearbyPlaces`：周边地点
- `walkMode`：纯粹 / 进阶
- `journeyMode`：单人 / 同行
- `generationBusy`：生成中状态

状态流转：

```text
进入探索页
  ↓
没有探索点 -> exploreStep = location
已有探索点 -> exploreStep = generate
  ↓
Step 1 搜索 / 定位 / 附近地点推荐
  ↓
点击定位、搜索结果或附近地点推荐
  ↓
写入 locationName / latitude / longitude / hasConfirmedExplorePoint
  ↓
用户点击下一步进入 exploreStep = generate
  ↓
选择生成条件
  ↓
生成主题
```

### 7.14 方案 A 的优缺点

优点：

- 不新增页面，路由和返回逻辑简单
- 用户仍然认为“探索”是一个完整流程
- 地图展示和个性化生成不会同时挤在一屏里
- 现有 `index` 页面里的逻辑可以渐进改造
- 主题生成调试面板仍可保留在同一页

缺点：

- `index` 页面代码仍然较重，需要注意拆分组件
- 地图组件仍在主 tab 页里，性能压力没有完全转移
- Step 切换需要处理地图隐藏 / 回显问题

后续建议把探索页拆成两个轻组件：

- `location-step`：负责地图、搜索、周边地点、确认探索点
- `generation-step`：负责模式、条件、生成按钮、主题卡

即使暂时不拆新页面，也不要让 `index.wxml` 继续无限膨胀。

## 13. 实施优先级

### 第一阶段：全局质感与触感

目标：用最小成本让整个小程序立刻更顺。

范围：

- `app.wxss`
- 全局按钮
- 全局卡片
- chip
- 输入框
- active 状态

验收：

- 所有主要点击元素都有按压反馈
- 页面整体圆角、阴影、色彩统一
- 不再出现明显突兀的纯黑、过重阴影或过亮渐变

### 第二阶段：探索页主流程

目标：把探索页改成页内两步式，让用户先专注设置探索点，再进入个性化生成。

范围：

- `exploreStep` 状态
- 顶部步骤切换条
- Step 1 探索点设置区
- Step 2 个性化生成区
- 搜索结果面板
- 地图主舞台与底部选点面板
- 纯粹 / 进阶条件面板
- 生成按钮状态
- 主题卡

验收：

- 未设置探索点时，默认停留在 Step 1
- 设置探索点后，可通过“下一步”或顶部步骤进入 Step 2
- Step 2 只展示地点摘要，不再嵌入大地图
- 用户能从 Step 2 快速回到 Step 1 更换地点
- 生成中状态有过程感
- 生成结果像一张可带走的任务卡

### 第三阶段：记录页任务打卡

目标：让打卡过程不像填表，而像完成当前任务。

范围：

- 记录页主题状态卡
- 位置状态条
- 任务手风琴
- 素材按钮
- 保存 sticky action

验收：

- 用户能快速找到当前任务
- 展开任务后层级清楚
- 录音、上传、保存状态明确
- 保存按钮始终容易触达

### 第四阶段：足迹页与个人页收尾

目标：增强回看和身份感。

范围：

- 足迹页顶部摘要
- 筛选区收起 / 展开
- walk card
- 成就墙
- 个人页账户状态

验收：

- 足迹页更像纪念卡册，而不是记录管理列表
- 成就进度更有收藏感
- 个人页状态切换不跳、不乱

## 14. 技术落地注意事项

### 14.1 不引入重依赖

当前是原生微信小程序，优先使用：

- `wxss transition`
- `wxss animation`
- 条件 class
- 现有组件拆分

不建议为了动效引入复杂第三方库。

### 14.2 动画只使用 transform 和 opacity

尽量避免动画这些属性：

- `top`
- `left`
- `width`
- `height`
- `margin`

优先使用：

- `transform`
- `opacity`

这样在小程序里更稳。

### 14.3 控制阴影和 backdrop-filter

微信小程序在不同机型上的渲染能力差异较大。

建议：

- 少用大面积 `backdrop-filter`
- 少用多层复杂阴影
- 地图页和列表页尤其要注意性能

### 14.4 保持触控目标足够大

所有主要可点击区域建议不小于：

- 高度 `80rpx - 88rpx`
- 小 icon 操作不小于 `56rpx`
- 重要按钮不低于 `88rpx`

### 14.5 避免文本溢出

需要重点检查：

- 主题标题过长
- 地点名过长
- 搜索结果名称过长
- 成就标题过长
- 同行房间成员昵称过长
- 按钮文案在小屏幕下换行

## 15. 验收标准

### 15.1 视觉验收

- 页面背景、卡片、按钮、chip 色彩统一
- 主行动按钮在每个页面只有一个最强视觉焦点
- 大圆角不再滥用
- 胶囊按钮只用于选择和标签，不充当所有按钮
- 阴影统一偏暖，不出现廉价黑色大阴影

### 15.2 交互验收

- 主要点击项都有明确 active 反馈
- 切换模式、切换 tab、选择 chip 不生硬
- 搜索结果、周边地点、任务展开出现自然
- 生成中、保存中、录音中状态明确
- 空状态都有下一步行动

### 15.3 流程验收

探索页：

- 用户能清楚知道是否已经设置探索点
- 用户能清楚知道当前是纯粹 / 进阶、单人 / 同行
- 用户能顺利生成主题并开始漫步

记录页：

- 用户能清楚看到任务完成进度
- 用户能顺利展开任务、添加素材、打卡
- 用户能随时找到保存入口

足迹页：

- 用户能区分纪念卡册和成就
- 用户能筛选记录，但不会被筛选面板压迫
- 进行中 / 待出发记录能被快速识别

个人页：

- 用户能明确当前登录状态
- 编辑资料过程稳定
- 退出登录不会误触

### 15.4 性能验收

- 页面滚动不明显卡顿
- 地图页拖动不受装饰层影响
- 列表卡片动画不影响点击
- 低端机不因阴影、模糊、渐变过多而明显掉帧

## 16. 建议的改造顺序

建议按以下顺序推进：

1. 先改全局设计 token 和基础交互反馈。
2. 再改探索页和主题卡，因为这是用户第一印象和主流程入口。
3. 再改记录页和任务打卡，因为这是用户停留时间最长的页面。
4. 最后改足迹页、成就墙和个人页，完成整体统一。

不建议一开始就大范围重排所有页面。

原因是：

- 当前业务功能较多
- 小程序页面之间状态联动复杂
- 先做全局和主流程，可以更快验证体验方向
- 后续页面可以复用前面整理出的组件规则

## 17. 结论

小程序前端已经具备完整功能，但体验还停留在“功能模块可用”的阶段。

本次优化建议将前端升级为“城市线索手册”的体验：

- 探索页负责帮用户确定出发点和生成线索
- 主题卡像一张可以带走的城市任务票据
- 记录页像正在漫步时的任务操作台
- 足迹页像一本纪念卡册和成就墙
- 个人页保持轻量、稳定、可信任

本方案的核心不是增加复杂视觉，而是统一层级、补足反馈、控制节奏，让用户每一步都知道自己在哪里、刚刚做了什么、下一步可以做什么。

后续建议先从 `app.wxss`、探索页和 `theme-card` 开始。这三个位置收益最大，也最容易验证整体方向。
