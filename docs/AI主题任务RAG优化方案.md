- 现有 AI 主题任务生成已经接入了统一检索层

4. 如何把“更贴题”和“更丰富”同时做出来，而不是二选一

## 2. 当前 RAG 现状

### 2.1 当前不是“只有单主题才有检索”

目前两个后端生成云函数 + 一个前端随机入口已经共用统一检索底座：

- `generateTheme`
- `generateCombinedTheme`

其中随机生成不再有独立云函数，而是由前端先随机挑选主题，再复用 `generateTheme`。

这意味着后续优化 RAG 时，只需要维护 `generateTheme` 和 `generateCombinedTheme` 两条后端生成链路；不要再新增或恢复 `generateRandomTheme`。

共享检索入口在：

- [generation-rag-runtime.js](/D:/liuliu-minimap/cloudfunctions/shared/generation-rag-runtime.js#L149)

当前统一检索会综合这些信息：

- 地点名
- `locationContext / sceneTag`
- 当前时间段
- 时间线索
- 附近 POI 名称
- 附近 POI 类型
- 附近活动线索
- 用户主题输入
- 心情 / 偏好 / 季节 / 天气

它会产出一份统一的 `ragContext`，主要包含：

- `selectedThemes`
- `timeContext`
- `nearbySummary`
- `scenes`
- `categories`
- `referenceMissions`

### 2.2 当前 RAG 的优势

已经有帮助的部分：

- 比起纯 prompt，自带更稳定的主题约束
- 可以把“地点、时间、附近”显式传给模型
- fallback 不再完全拍脑袋，而是有检索结果可兜
- 单主题、前端随机、组合生成入口终于开始站在同一检索底座上

### 2.3 当前 RAG 的不足

虽然已经统一，但目前仍然偏“轻量检索”，主要问题有 4 类：

1. 检索命中逻辑仍偏粗
2. 检索结果结构还不够细
3. 检索结果虽然更稳，但丰富度还不够高
4. 检索之后缺少一层“检索质量评估”

## 3. 当前痛点

### 3.1 更贴题，但有时不够“准”

当前检索会先推 `categories` 和 `referenceMissions`，再让模型生成。

这能防止明显跑偏，但还会有这些情况：

- 用户选了“数字”，任务里只是轻微带到数字，不够强命中
- 组合主题里，两个方向不是“融合”，只是并列出现
- 主题命中了，但任务没有命中
- description 看起来贴题，但 missions 不贴题

根因是：

- 现在主要还是用关键词和模板样例来拉齐方向
- 还缺“主题信号强度”这一层显式评分

### 3.2 更在地，但有时不够“活”

当前上下文已经有：

- `dominantScene`
- `activityHints`
- `timeHints`
- `poiNames`
- `poiTypes`

但模型最终仍可能写成：

- 很安全
- 很正确
- 但句式像
- 观察角度也像

根因是：

- 检索只解决“从哪类内容里写”
- 仍需要继续加强“从几个不同角度写”的控制力度

### 3.3 检索命中了，但结果依然重复

现在的重复感主要来自 3 个地方：

1. 检索返回的 `referenceMissions` 仍偏少
2. 多个场景会反复使用相似 cues
3. 后处理为了兜底，会把结果收敛回少数句式

所以会出现：

- 场景不同，但任务骨架差不多
- 主题不同，但句子结构差不多
- 内容不差，但看多了会觉得“都是这一套”

### 3.4 当前 RAG 缺少“检索质量”可视化

现在调试面板已经能看到：

- `source`
- `contextPacket`
- `validation`

但还看不到：

- 本次 RAG 选中了哪些 scene
- 本次 RAG 选中了哪些 category
- 检索命中分数是多少
- 为什么这个 scene 赢了
- 为什么这个 template 被选中了

这会导致后面调试时只能看结果，不能快速判断是：

- 检索错了
- prompt 不够强
- 模型没听话
- 后处理改坏了

---

## 4. RAG 优化总目标

下一版 RAG 不是单纯“检索更多内容”，而是同时实现 4 个目标：

1. 更贴题
2. 更在地
3. 更丰富
4. 更可解释

换句话说：

- 不只是让模型更容易写对
- 也要让模型更容易写出不同角度
- 同时让我们能知道它为什么这么写

---

## 5. 优化方向总览

建议把 RAG 优化拆成 6 个方向：

1. 检索输入优化
2. 检索排序优化
3. 检索结构优化
4. 检索后的生成约束优化
5. 检索质量评估与调试优化
6. 内容资产库扩充

---

## 6. 方向一：检索输入优化

### 6.1 让输入不只是“地点标签”，而是“附近状态”

当前输入已经有地点、时间、附近摘要，但还可以继续增强成更适合检索的字段：

- `themeIntent`
  - 用户显式选了什么主题
- `themeStrength`
  - 单主题 / 双主题 / 随机主题
- `sceneConfidence`
  - 当前 `dominantScene` 的置信度
- `topSceneCandidates`
  - 前 3 个候选场景
- `poiDiversity`
  - POI 是单一业态还是混合业态
- `movementStyle`
  - 当前地点更像穿行、停留、排队、慢行、等待

这些字段不是为了直接给用户看，而是为了让检索更理解“这次应该召回哪类任务”。

### 6.2 输入里加入“生成约束意图”

建议显式加一层 `generationIntent`：

```json
{
  "mustHitThemes": ["数字", "色彩"],
  "preferNearbyAnchor": true,
  "preferTaskVariety": true,
  "avoidGeneric": true,
  "mode": "advanced"
}
```

意义是：

- 把“这次生成最看重什么”从 prompt 文案变成结构化信号
- 检索排序时就能用，而不是等模型自由发挥

---

## 7. 方向二：检索排序优化

### 7.1 从关键词命中升级为多维打分

当前统一检索更像“关键词 + 偏好 bias”的轻量打分。

建议升级成多维分数：

`retrievalScore = themeScore + sceneScore + timeScore + nearbyScore + diversityScore + safetyScore`

各项建议如下：

- `themeScore`
  - 检索片段和目标主题的直接相关度
- `sceneScore`
  - 检索片段和当前场景的贴合程度
- `timeScore`
  - 检索片段是否适合当前时间段
- `nearbyScore`
  - 是否呼应当前附近 POI / 活动线索
- `diversityScore`
  - 是否有助于拉开与已选样例的差异
- `safetyScore`
  - 是否符合当前安全边界

### 7.2 不要只检索“最像的”，要检索“最像的一组”

建议把召回逻辑分成两层：

1. `core references`
   - 最贴题、最贴当前场景
2. `supporting references`
   - 稍弱一点，但能提供不同任务角度

例如对于“黄昏 + 数字 + 历史景区游览带”：

- `core`
  - 景区编号
  - 票口数字
  - 导览线索
- `supporting`
  - 建筑重复节奏
  - 路线与出口号
  - 排队长度与人数变化

这样模型更容易同时得到：

- 一个强主题锚点
- 两三个变化角度

### 7.3 组合主题要做“交叉排序”

当前双主题组合更容易出现并列，而不是真融合。

建议组合检索时不要先按主题 A 找一堆、再按主题 B 找一堆，而是增加交叉召回：

- `A 强 / B 弱`
- `A 弱 / B 强`
- `A 与 B 同时强`

例如“数字 + 色彩”：

- 票牌数字上的颜色对比
- 招牌上的数字亮面
- 倒计时屏幕的色温变化

这类内容比单纯“一个数字任务 + 一个色彩任务”更像真正融合。

## 8. 方向三：检索结构优化

### 8.1 检索结果不要只返回样例句子

当前 `referenceMissions` 主要包含：

- `category`
- `cues`
- `samples`

下一版建议扩成：

```json
{
  "category": "数字",
  "angle": "数量统计",
  "sceneFit": ["历史景区游览带", "交通换乘流动带"],
  "timeFit": ["上午", "黄昏"],
  "anchorType": ["门牌", "票口", "编号", "出口号"],
  "cues": ["门牌号", "票口编号", "导览数字"],
  "samples": ["找到一个出口号...", "..."],
  "antiPatterns": ["不要写成纯抽象观察", "不要偏成声音任务"]
}
```

这样检索结果就不只是“给模型看几句参考句”，而是：

- 明确这段内容适合什么主题
- 适合什么时间
- 适合什么场景
- 不适合写成什么

### 8.2 把任务骨架也纳入检索对象

建议把任务资产拆成两层：

1. `content reference`
   - 写什么
2. `task skeleton`
   - 怎么写

例如：

- `content reference`
  - 门牌号、展厅编号、票口数字
- `task skeleton`
  - 寻找
  - 比较
  - 统计
  - 跟随
  - 判断来源

这样生成时更容易做到：

- 内容在地
- 结构多样

### 8.3 引入“反例检索”

建议每个主题或场景都维护一批 `antiPatterns`：

- 数字主题
  - 不要只写“感受一下附近”
  - 不要变成纯声音观察
- 气味主题
  - 不要只有“像什么记忆”，要有来源或变化
- 历史景区
  - 不要只是“观察游客”
  - 要带可执行入口

这些反例不一定展示给模型全文，但可用于：

- 检索排序扣分
- 后处理验证
- AI 二次复核提示

## 9. 方向四：检索后的生成约束优化

### 9.1 让 prompt 显式消费检索结果的不同层

建议 prompt 不再只塞一坨 `ragContext JSON`，而是拆成：

- `核心检索结论`
- `候选场景`
- `推荐任务角度`
- `禁止偏航项`

例如：

```text
核心方向：数字
当前最相关场景：历史景区游览带
建议优先锚点：票口编号、导览牌、门洞序号
推荐角度：寻找 / 比较 / 行动指令
避免：抽象散文、纯游客观察、只写“感受一下”
```

这会比一整段 JSON 更容易被模型真正使用。

### 9.2 检索后要先做“生成计划”，再写内容

建议把生成拆两步：

1. 先让模型根据 RAG 输出 `generationPlan`
2. 再按 plan 写最终主题与任务

例如：

```json
{
  "targetThemes": ["数字"],
  "chosenScene": "历史景区游览带",
  "angles": ["编号识别", "数量统计", "路线线索"],
  "mustUseAnchors": ["票口", "门洞", "导览牌"]
}
```

这样可以明显减少：

- 检索明明很好，模型却没用
- 三个任务写成一个意思

### 9.3 对“丰富度”增加硬约束

建议进阶模式下，至少要求 3 个任务满足“角度不同”：

- 主体
- 关系
- 空间
- 时间
- 来源
- 变化
- 对比
- 统计

RAG 这边要给出可选角度，生成时强约束至少覆盖 2 到 3 个不同角度。

## 10. 方向五：检索质量评估与调试优化

### 10.1 调试面板增加 RAG 可视化

建议探索页调试面板继续新增：

- `ragSelectedCategories`
- `ragSelectedScenes`
- `ragSceneScores`
- `ragReferenceAngles`
- `ragAnchors`
- `ragAntiPatterns`

这样看到一条任务时，可以立刻判断：

- 是检索错了
- 还是模型没命中
- 还是后处理替换掉了

### 10.2 给每次生成输出一个 RAG 评分摘要

建议在云函数返回里增加：

```json
{
  "ragDebug": {
    "retrievalQuality": "high",
    "themeCoverage": ["数字"],
    "sceneCoverage": ["历史景区游览带"],
    "anchorCoverage": ["票口编号", "导览牌"],
    "diversityAngles": ["寻找", "统计", "跟随"]
  }
}
```

这对后续做抽样分析很有帮助。

### 10.3 建立人工抽检维度

建议上线前后用固定维度抽检：

1. 是否命中主题
2. 是否命中时间段
3. 是否命中附近场景
4. 是否任务角度有变化
5. 是否容易读懂
6. 是否像真实人会执行的任务

## 12. 最佳方案实现架构

### 12.1 核心判断

当前问题的本质，不是“RAG 太强”或“AI 自由发挥太少”二选一，而是：

- RAG 同时承担了“提供事实”和“暗示写法”两件事
- 生成层没有把“先定计划”和“再写文案”真正拆开
- 去重主要发生在结果收口阶段，而不是生成前的计划阶段

于是就会出现两种反复摇摆的状态：

1. RAG 给得太具体
   - 模型会围绕少数 cue、sample、骨架反复改写
   - 输出稳定，但容易单一、像样例、像模板

2. RAG 给得太松
   - 模型虽然更自由，但会开始抽象化、泛化、重复化
   - 可能贴题变弱，或者生成几次又回到同一个句式

所以最佳方案不是继续在 prompt 上来回拉扯，而是把系统拆成“分层流水线”。

### 12.2 推荐的四层流水线

建议把单次生成拆成 4 层，每层只负责一件事：

1. `Grounding Layer`
   - 只负责“此时此地”的事实锚定
   - 输出时间段、场景、POI、活动线索、季节、安全边界
   - 不输出完整任务样例句

2. `Planning Layer`
   - 只负责决定“这次写什么、从哪个角度写、避免什么”
   - 输出主题命中方向、主场景、任务动作类型、观察对象、禁用表达、去重信息

3. `Writing Layer`
   - 只负责把 plan 写成标题、描述和任务
   - 不直接接触大量原始 RAG 素材
   - 只消费已经裁剪过的 plan 和少量 grounding 信息

4. `Validation & Repair Layer`
   - 只负责检查是否贴题、是否具体、是否重复、是否可执行
   - 必要时只局部重写失败项，而不是整包推翻重来

这样做的意义是：

- RAG 不再直接塑形文案句式
- 模型有发挥空间，但发挥范围被 plan 限定
- 重复和抽象问题会在 plan 和 validation 两层提前被拦截

### 12.3 Grounding Layer 应该提供什么

这一层建议只保留“事实”和“约束”，不要再给整句 sample。

推荐输出结构：

```json
{
  "time": {
    "phase": "黄昏",
    "hints": ["人流回程中", "亮灯前后变化明显"]
  },
  "nearby": {
    "dominantScene": "历史景区游览带",
    "sceneCandidates": ["历史景区游览带", "城市地标与广场游览带"],
    "anchors": ["城楼", "广场边缘", "茶社门口"],
    "activityHints": ["停留拍照", "穿行经过", "短暂停顿"]
  },
  "themeGrounding": {
    "category": "声音",
    "allowedAngles": ["停一下听", "分辨来源", "顺着走几步", "等下一次出现"],
    "forbiddenDrift": ["不要写成空间关系散文", "不要把抽象边界当观察对象"]
  }
}
```

这一层的关键原则是：

- 给模型“现实世界里有什么”
- 不给模型“照着改写哪句话”

### 12.4 Planning Layer 才是解决重复和抽象的核心

这层建议新增一个明确的 `missionPlan`，而不是只靠 prompt 临时组织。

推荐输出结构：

```json
{
  "targetTheme": "声音",
  "chosenScene": "历史景区游览带",
  "anchor": "茶社门口",
  "actionType": "停一下听",
  "observationAngle": "分辨近处与远处声音来源",
  "objectType": "人群脚步、交谈、背景广播",
  "tone": "直接、口语、可执行",
  "avoidPhrases": ["边界", "关系", "秩序", "气质"],
  "avoidRecentPatterns": ["比较存包处与茶社的声浪差异"]
}
```

`missionPlan` 至少要显式决定 6 件事：

1. 这次主打哪个主题
2. 这次落在哪个场景
3. 这次抓哪个锚点
4. 这次用什么动作骨架
5. 这次看什么对象或变化
6. 这次不能再重复哪些最近用过的表达

只要这层做实，后面的文案才不会一直回到同一个模板。

### 12.5 Writing Layer 的职责要收窄

写作层不应该再自己决定主题方向，只负责把 `missionPlan` 写成自然语言。

这里建议保留 4 类硬约束：

1. 任务必须能落到具体对象
   - 能指认地点、物件、人群、声音源、颜色组合、气味来源或数字线索之一

2. 任务必须有明确动作
   - 如“找”“听”“看”“停一下”“绕过去”“换个位置”“等一会儿”“判断”

3. 任务必须是口语化的
   - 像真实人会对自己说的话
   - 避免“空间关系”“声音边界”“现场气质”这种抽象名词做主语

4. 任务必须短
   - 纯粹模式优先一条短任务
   - 进阶模式三条任务也要短而分明

这意味着写作层应当少看 JSON、多看 plan。

### 12.6 Validation & Repair Layer 要从“打分器”升级为“纠偏器”

验证层不应该只给一个“过/不过”，而要能回答：

1. 为什么不过
2. 具体哪条任务不过
3. 应该往哪个方向改
4. 改完后的候选内容是什么

建议 AI 复核稳定输出这些字段：

```json
{
  "ok": false,
  "score": 5,
  "failedChecks": ["themeHit", "concreteness", "novelty"],
  "reason": "任务虽然是声音主题，但对象过抽象，且与最近一次生成表达过近。",
  "comment": "方向是对的，但还不够落地。",
  "rewriteAdvice": "保留茶社和黄昏线索，改成具体可听对象，不要再写边界或关系。",
  "suggestedRewrite": {
    "title": "...",
    "description": "...",
    "missions": ["..."]
  }
}
```

这样前端看到的“AI 评语”和“改写建议”才是真正可用的，而不是只有一个笼统结论。

### 12.7 单主题和双主题不应该共用同一套生成心智

最佳方案里，单主题和组合主题要继续共用底层 grounding，但不要共用同一套 writing 习惯。

单主题建议：

- 重点是“一个主题下的多样化”
- 允许更自由的动作和对象组合
- 不再依赖该主题的任务样例库直接塑形句式

双主题建议：

- 重点是“两个主题的交集”
- 必须显式输出 `intersectionPlan`
- 每条任务都要回答“两个主题如何在同一个对象或动作上同时成立”

也就是说：

- 单主题的难点是避免重复和抽象
- 双主题的难点是避免并列和串味

两者要共用底座，但不要强行共用同一层 plan 模板。

### 12.8 解决“总是那几个动作”的最佳办法

动作骨架不能只放在 prompt 里作为一串提示词，而应该升级成“主题动作池 + 冷却机制”。

以单主题为例，可以先做这样的主题动作池：

- 形状
  - `找一处`
  - `绕着看`
  - `换个角度`
  - `从远到近`
  - `贴近一个细节`

- 色彩
  - `找一组颜色`
  - `比较两处颜色`
  - `看颜色怎么变`
  - `找最突出的那一块`
  - `顺着颜色再找下一处`

- 声音
  - `停一下听`
  - `分辨来源`
  - `顺着声音走几步`
  - `等下一次出现`
  - `回头再听`

- 数字
  - `先猜再确认`
  - `找一个规则`
  - `核对两个线索`
  - `判断哪个更像提示`
  - `找一个会影响行动的数字`

- 气味
  - `闻到之后找来源`
  - `换个位置再闻`
  - `等风过来`
  - `比较前后变化`
  - `找最容易停下来的那一处`

再增加两个机制：

1. `actionCooldown`
   - 最近 3 到 5 次已用动作在同地点降权

2. `anchorCooldown`
   - 同一 POI 或同一表述连续出现时直接降权或禁用

这样才能真正减少“声音老是比较”“形状老是找弧度”这类问题。

### 12.9 解决“抽象但不接地气”的最佳办法

建议把“具体性”从软约束升级成硬约束。

每条任务在写出前，至少满足以下结构之一：

1. `动作 + 具体对象`
   - 例如“停一下，听茶社门口最靠近你的那层声音”

2. `动作 + 具体对象 + 变化`
   - 例如“绕到广场边缘，看看颜色从亮面到背光面怎么变”

3. `动作 + 具体对象 + 判断`
   - 例如“先猜哪一个数字更像给游客看的，再走近确认”

同时明确禁止以下表达直接成为观察核心：

- 边界
- 关系
- 节奏感
- 气质
- 秩序
- 张力
- 氛围

这些词可以作为补充说明存在，但不能替代真正的观察对象。

### 12.10 解决“同地点反复生成还是很像”的最佳办法

最佳方案里必须加入“跨次生成去重”，而不是只做单次结果内去重。

建议前后端协同维护 `recentMissionHistory`，至少记录：

- 最近 5 到 10 次任务文本
- 最近使用过的 `actionType`
- 最近使用过的 `anchor`
- 最近使用过的 `observationAngle`
- 最近被判失败的 bad case

生成下一次 plan 时直接参与排序：

- 与最近任务相似度过高的 plan 不进入写作层
- 与最近 bad case 太像的 plan 直接淘汰
- 同地点连续生成时，优先换锚点，再换动作，再换角度

这一层不到位，后面无论怎样优化 prompt，最终还是会重新撞回熟悉句式。

### 12.11 推荐的最小可落地版本

如果只做一轮最有价值的升级，建议优先做这 5 件事：

1. RAG 只保留 grounding 信息，不再向单主题写作层直接暴露任务样例
2. 新增显式 `missionPlan`，把动作、锚点、对象、禁用词拆出来
3. 增加 `recentMissionHistory`，做跨次去重
4. AI 验证稳定返回 `score / comment / rewriteAdvice / suggestedRewrite`
5. 单主题使用主题动作池，不再主要依赖通用骨架顺序

这 5 件事做完，通常就能同时改善：

- 重复
- 抽象
- 主题被 RAG 带偏
- 调试时看不出到底哪里错了

---

## 13. 推荐落地顺序

### 第一阶段：低成本提效

目标：

- 先让 RAG 更可解释
- 先解决“为什么总像”和“为什么有时不贴题”

建议落地：

1. 调试面板展示 `ragDebug`
2. 检索结果增加 `angle / anchorType / antiPatterns`
3. 进阶模式增加“任务角度不重复”约束
4. 双主题增加交叉召回

### 第二阶段：中成本提效

目标：

- 让内容同时更贴题和更丰富

建议落地：

1. 检索排序升级为多维打分
2. RAG 先产出 `generationPlan`
3. 时段专属资产入库
4. 组合主题改成“交叉召回 + 计划生成”

### 第三阶段：长期优化

目标：

- 建立真正可评估的生成系统

建议落地：

1. 建生成抽样池
2. 做主题命中率和重复率统计
3. 维护 bad case 库
4. 用 bad case 反推 RAG 资产和排序规则

---

## 14. 验收标准

如果 RAG 优化有效，应该至少出现这些变化：

1. 主题命中率更高
   - 用户选了什么，任务就更明显围绕什么

2. 时间区分度更高
   - 清晨、黄昏、夜间、凌晨不会再写得像一个时段

3. 在地感更强
   - 更容易看出附近 POI、活动线索、场景标签被真正用了

4. 丰富度更高
   - 相同主题下，不同场景能写出更不同的任务
   - 相同场景下，不同主题也能拉开

5. 重复率更低
   - 不再经常落回同一条锚点句

6. 调试更可解释
   - 能看出是 RAG 召回问题、模型问题，还是后处理问题

---

## 15. 最终建议

这套系统下一步最该做的，不是继续堆 prompt，而是把 RAG 从“轻量样例参考”升级成“生成计划的上游”。

一句话概括就是：

- 现在的 RAG 更像“给模型一些参考”
- 下一版的 RAG 应该变成“告诉模型这次该围绕什么、从哪些角度写、不能怎么跑偏”

只有这样，主题任务内容才会同时做到：

- 更接近主题
- 更接近附近
- 更接近此刻
- 也更丰富、更少 AI 味
