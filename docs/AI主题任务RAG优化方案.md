# AI 主题任务 RAG 优化方案

## 1. 文档目的

本文档用于回答一个更具体的问题：

- 现有 AI 主题任务生成已经接入了统一检索层
- 下一步如何继续优化 RAG，才能让内容更贴近主题、更像此时此地、也更丰富不重复

这份文档重点关注 5 件事：

1. 当前 RAG 已经做到什么
2. 为什么现在仍然会“贴题不够稳、丰富度不够高、重复感偏强”
3. 下一版 RAG 应该往哪些方向补
4. 如何把“更贴题”和“更丰富”同时做出来，而不是二选一
5. 如何分阶段落地，并评估效果

## 2. 当前 RAG 现状

### 2.1 当前不是“只有单主题才有检索”

目前两个后端生成云函数 + 一个前端随机入口已经共用统一检索底座：

- `generateTheme`
- `generateCombinedTheme`

其中随机生成不再有独立云函数，而是由前端先随机挑选主题，再复用 `generateTheme`。

这意味着后续优化 RAG 时，只需要维护 `generateTheme` 和 `generateCombinedTheme` 两条后端生成链路；不要再新增或恢复 `generateRandomTheme`。

共享检索入口在：

- [generation-rag-runtime.js](/D:/liuliu-minimap/cloudfunctions/shared/generation-rag-runtime.js#L149)

当前统一检索会综合这些信息：

- 地点名
- `locationContext / sceneTag`
- 当前时间段
- 时间线索
- 附近 POI 名称
- 附近 POI 类型
- 附近活动线索
- 用户主题输入
- 心情 / 偏好 / 季节 / 天气

它会产出一份统一的 `ragContext`，主要包含：

- `selectedThemes`
- `timeContext`
- `nearbySummary`
- `scenes`
- `categories`
- `referenceMissions`

### 2.2 当前 RAG 的优势

已经有帮助的部分：

- 比起纯 prompt，自带更稳定的主题约束
- 可以把“地点、时间、附近”显式传给模型
- fallback 不再完全拍脑袋，而是有检索结果可兜
- 单主题、前端随机、组合生成入口终于开始站在同一检索底座上

### 2.3 当前 RAG 的不足

虽然已经统一，但目前仍然偏“轻量检索”，主要问题有 4 类：

1. 检索命中逻辑仍偏粗
2. 检索结果结构还不够细
3. 检索结果虽然更稳，但丰富度还不够高
4. 检索之后缺少一层“检索质量评估”

## 3. 当前痛点

### 3.1 更贴题，但有时不够“准”

当前检索会先推 `categories` 和 `referenceMissions`，再让模型生成。

这能防止明显跑偏，但还会有这些情况：

- 用户选了“数字”，任务里只是轻微带到数字，不够强命中
- 组合主题里，两个方向不是“融合”，只是并列出现
- 主题命中了，但任务没有命中
- description 看起来贴题，但 missions 不贴题

根因是：

- 现在主要还是用关键词和模板样例来拉齐方向
- 还缺“主题信号强度”这一层显式评分

### 3.2 更在地，但有时不够“活”

当前上下文已经有：

- `dominantScene`
- `activityHints`
- `timeHints`
- `poiNames`
- `poiTypes`

但模型最终仍可能写成：

- 很安全
- 很正确
- 但句式像
- 观察角度也像

根因是：

- 检索只解决“从哪类内容里写”
- 仍需要继续加强“从几个不同角度写”的控制力度

### 3.3 检索命中了，但结果依然重复

现在的重复感主要来自 3 个地方：

1. 检索返回的 `referenceMissions` 仍偏少
2. 多个场景会反复使用相似 cues
3. 后处理为了兜底，会把结果收敛回少数句式

所以会出现：

- 场景不同，但任务骨架差不多
- 主题不同，但句子结构差不多
- 内容不差，但看多了会觉得“都是这一套”

### 3.4 当前 RAG 缺少“检索质量”可视化

现在调试面板已经能看到：

- `source`
- `contextPacket`
- `validation`

但还看不到：

- 本次 RAG 选中了哪些 scene
- 本次 RAG 选中了哪些 category
- 检索命中分数是多少
- 为什么这个 scene 赢了
- 为什么这个 template 被选中了

这会导致后面调试时只能看结果，不能快速判断是：

- 检索错了
- prompt 不够强
- 模型没听话
- 后处理改坏了

---

## 4. RAG 优化总目标

下一版 RAG 不是单纯“检索更多内容”，而是同时实现 4 个目标：

1. 更贴题
2. 更在地
3. 更丰富
4. 更可解释

换句话说：

- 不只是让模型更容易写对
- 也要让模型更容易写出不同角度
- 同时让我们能知道它为什么这么写

---

## 5. 优化方向总览

建议把 RAG 优化拆成 6 个方向：

1. 检索输入优化
2. 检索排序优化
3. 检索结构优化
4. 检索后的生成约束优化
5. 检索质量评估与调试优化
6. 内容资产库扩充

---

## 6. 方向一：检索输入优化

### 6.1 让输入不只是“地点标签”，而是“附近状态”

当前输入已经有地点、时间、附近摘要，但还可以继续增强成更适合检索的字段：

- `themeIntent`
  - 用户显式选了什么主题
- `themeStrength`
  - 单主题 / 双主题 / 随机主题
- `sceneConfidence`
  - 当前 `dominantScene` 的置信度
- `topSceneCandidates`
  - 前 3 个候选场景
- `poiDiversity`
  - POI 是单一业态还是混合业态
- `movementStyle`
  - 当前地点更像穿行、停留、排队、慢行、等待

这些字段不是为了直接给用户看，而是为了让检索更理解“这次应该召回哪类任务”。

### 6.2 输入里加入“生成约束意图”

建议显式加一层 `generationIntent`：

```json
{
  "mustHitThemes": ["数字", "色彩"],
  "preferNearbyAnchor": true,
  "preferTaskVariety": true,
  "avoidGeneric": true,
  "mode": "advanced"
}
```

意义是：

- 把“这次生成最看重什么”从 prompt 文案变成结构化信号
- 检索排序时就能用，而不是等模型自由发挥

---

## 7. 方向二：检索排序优化

### 7.1 从关键词命中升级为多维打分

当前统一检索更像“关键词 + 偏好 bias”的轻量打分。

建议升级成多维分数：

`retrievalScore = themeScore + sceneScore + timeScore + nearbyScore + diversityScore + safetyScore`

各项建议如下：

- `themeScore`
  - 检索片段和目标主题的直接相关度
- `sceneScore`
  - 检索片段和当前场景的贴合程度
- `timeScore`
  - 检索片段是否适合当前时间段
- `nearbyScore`
  - 是否呼应当前附近 POI / 活动线索
- `diversityScore`
  - 是否有助于拉开与已选样例的差异
- `safetyScore`
  - 是否符合当前安全边界

### 7.2 不要只检索“最像的”，要检索“最像的一组”

建议把召回逻辑分成两层：

1. `core references`
   - 最贴题、最贴当前场景
2. `supporting references`
   - 稍弱一点，但能提供不同任务角度

例如对于“黄昏 + 数字 + 历史景区游览带”：

- `core`
  - 景区编号
  - 票口数字
  - 导览线索
- `supporting`
  - 建筑重复节奏
  - 路线与出口号
  - 排队长度与人数变化

这样模型更容易同时得到：

- 一个强主题锚点
- 两三个变化角度

### 7.3 组合主题要做“交叉排序”

当前双主题组合更容易出现并列，而不是真融合。

建议组合检索时不要先按主题 A 找一堆、再按主题 B 找一堆，而是增加交叉召回：

- `A 强 / B 弱`
- `A 弱 / B 强`
- `A 与 B 同时强`

例如“数字 + 色彩”：

- 票牌数字上的颜色对比
- 招牌上的数字亮面
- 倒计时屏幕的色温变化

这类内容比单纯“一个数字任务 + 一个色彩任务”更像真正融合。

## 8. 方向三：检索结构优化

### 8.1 检索结果不要只返回样例句子

当前 `referenceMissions` 主要包含：

- `category`
- `cues`
- `samples`

下一版建议扩成：

```json
{
  "category": "数字",
  "angle": "数量统计",
  "sceneFit": ["历史景区游览带", "交通换乘流动带"],
  "timeFit": ["上午", "黄昏"],
  "anchorType": ["门牌", "票口", "编号", "出口号"],
  "cues": ["门牌号", "票口编号", "导览数字"],
  "samples": ["找到一个出口号...", "..."],
  "antiPatterns": ["不要写成纯抽象观察", "不要偏成声音任务"]
}
```

这样检索结果就不只是“给模型看几句参考句”，而是：

- 明确这段内容适合什么主题
- 适合什么时间
- 适合什么场景
- 不适合写成什么

### 8.2 把任务骨架也纳入检索对象

建议把任务资产拆成两层：

1. `content reference`
   - 写什么
2. `task skeleton`
   - 怎么写

例如：

- `content reference`
  - 门牌号、展厅编号、票口数字
- `task skeleton`
  - 寻找
  - 比较
  - 统计
  - 跟随
  - 判断来源

这样生成时更容易做到：

- 内容在地
- 结构多样

### 8.3 引入“反例检索”

建议每个主题或场景都维护一批 `antiPatterns`：

- 数字主题
  - 不要只写“感受一下附近”
  - 不要变成纯声音观察
- 气味主题
  - 不要只有“像什么记忆”，要有来源或变化
- 历史景区
  - 不要只是“观察游客”
  - 要带可执行入口

这些反例不一定展示给模型全文，但可用于：

- 检索排序扣分
- 后处理验证
- AI 二次复核提示

## 9. 方向四：检索后的生成约束优化

### 9.1 让 prompt 显式消费检索结果的不同层

建议 prompt 不再只塞一坨 `ragContext JSON`，而是拆成：

- `核心检索结论`
- `候选场景`
- `推荐任务角度`
- `禁止偏航项`

例如：

```text
核心方向：数字
当前最相关场景：历史景区游览带
建议优先锚点：票口编号、导览牌、门洞序号
推荐角度：寻找 / 比较 / 行动指令
避免：抽象散文、纯游客观察、只写“感受一下”
```

这会比一整段 JSON 更容易被模型真正使用。

### 9.2 检索后要先做“生成计划”，再写内容

建议把生成拆两步：

1. 先让模型根据 RAG 输出 `generationPlan`
2. 再按 plan 写最终主题与任务

例如：

```json
{
  "targetThemes": ["数字"],
  "chosenScene": "历史景区游览带",
  "angles": ["编号识别", "数量统计", "路线线索"],
  "mustUseAnchors": ["票口", "门洞", "导览牌"]
}
```

这样可以明显减少：

- 检索明明很好，模型却没用
- 三个任务写成一个意思

### 9.3 对“丰富度”增加硬约束

建议进阶模式下，至少要求 3 个任务满足“角度不同”：

- 主体
- 关系
- 空间
- 时间
- 来源
- 变化
- 对比
- 统计

RAG 这边要给出可选角度，生成时强约束至少覆盖 2 到 3 个不同角度。

## 10. 方向五：检索质量评估与调试优化

### 10.1 调试面板增加 RAG 可视化

建议探索页调试面板继续新增：

- `ragSelectedCategories`
- `ragSelectedScenes`
- `ragSceneScores`
- `ragReferenceAngles`
- `ragAnchors`
- `ragAntiPatterns`

这样看到一条任务时，可以立刻判断：

- 是检索错了
- 还是模型没命中
- 还是后处理替换掉了

### 10.2 给每次生成输出一个 RAG 评分摘要

建议在云函数返回里增加：

```json
{
  "ragDebug": {
    "retrievalQuality": "high",
    "themeCoverage": ["数字"],
    "sceneCoverage": ["历史景区游览带"],
    "anchorCoverage": ["票口编号", "导览牌"],
    "diversityAngles": ["寻找", "统计", "跟随"]
  }
}
```

这对后续做抽样分析很有帮助。

### 10.3 建立人工抽检维度

建议上线前后用固定维度抽检：

1. 是否命中主题
2. 是否命中时间段
3. 是否命中附近场景
4. 是否任务角度有变化
5. 是否容易读懂
6. 是否像真实人会执行的任务

## 11. 方向六：内容资产库扩充

### 11.1 扩充 sceneProfiles

目前场景还不够细。

建议继续扩到至少这些：

- 历史景区游览带
- 文博展览停留带
- 城市地标与广场游览带
- 公园或滨水慢行带
- 校园与教育生活带
- 居民街区生活带
- 商业办公停留带
- 餐饮与市井烟火带
- 交通换乘流动带
- 医院与民生服务带
- 夜间餐饮停留带
- 雨天街区反光带
- 旅游服务与排队等候带
- 老旧社区修补带
- 城市边缘混合业态带

### 11.2 扩充每个主题下的 angle 库

每个主题建议至少拆成 4 到 6 个子角度：

- 形状
  - 轮廓
  - 弧度
  - 重复
  - 边界
  - 对称
- 色彩
  - 色块
  - 渐变
  - 冷暖
  - 反光

- 声音
  - 连续背景声

### 11.3 扩充“时段专属资产”

建议给每个时间段维护专属检索资产：

## 12. 推荐落地顺序

### 第一阶段：低成本提效

目标：

- 先让 RAG 更可解释
- 先解决“为什么总像”和“为什么有时不贴题”

建议落地：

1. 调试面板展示 `ragDebug`
2. 检索结果增加 `angle / anchorType / antiPatterns`
3. 进阶模式增加“任务角度不重复”约束
4. 双主题增加交叉召回

### 第二阶段：中成本提效

目标：

- 让内容同时更贴题和更丰富

建议落地：

1. 检索排序升级为多维打分
2. RAG 先产出 `generationPlan`
3. 时段专属资产入库
4. 组合主题改成“交叉召回 + 计划生成”

### 第三阶段：长期优化

目标：

- 建立真正可评估的生成系统

建议落地：

1. 建生成抽样池
2. 做主题命中率和重复率统计
3. 维护 bad case 库
4. 用 bad case 反推 RAG 资产和排序规则

---

## 13. 验收标准

如果 RAG 优化有效，应该至少出现这些变化：

1. 主题命中率更高
   - 用户选了什么，任务就更明显围绕什么

2. 时间区分度更高
   - 清晨、黄昏、夜间、凌晨不会再写得像一个时段

3. 在地感更强
   - 更容易看出附近 POI、活动线索、场景标签被真正用了

4. 丰富度更高
   - 相同主题下，不同场景能写出更不同的任务
   - 相同场景下，不同主题也能拉开

5. 重复率更低
   - 不再经常落回同一条锚点句

6. 调试更可解释
   - 能看出是 RAG 召回问题、模型问题，还是后处理问题

---

## 14. 最终建议

这套系统下一步最该做的，不是继续堆 prompt，而是把 RAG 从“轻量样例参考”升级成“生成计划的上游”。

一句话概括就是：

- 现在的 RAG 更像“给模型一些参考”
- 下一版的 RAG 应该变成“告诉模型这次该围绕什么、从哪些角度写、不能怎么跑偏”

只有这样，主题任务内容才会同时做到：

- 更接近主题
- 更接近附近
- 更接近此刻
- 也更丰富、更少 AI 味
