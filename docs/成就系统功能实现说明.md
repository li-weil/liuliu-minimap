# 成就系统功能实现说明

## 1. 文档目的
本文档用于说明当前小程序成就系统的实现内容、技术方案、部署要求、维护方式，以及后续如何进一步演进到更稳健的生产版本。

适用范围：
- 微信小程序端成就页
- 成就数据计算与存储
- 成就图片展示链路
- 成就相关记录字段保存链路
- 微信云开发 `userAchievements` 集合

## 3. 技术说明

### 3.1 当前技术方案
当前版本采用“云端统一计算 + `userAchievements` 持久化 + 前端只读展示 + 云存储 fileID 引用”的方案。

具体方式：
- 成就规则集中维护在 `cloudfunctions/shared/achievement-runtime.js`
- 通过 `node scripts/sync_cloud_achievement_runtime.js` 同步到相关云函数目录
- 单人记录保存完成后由 `createWalk` 重算
- 同行记录结束后由 `finishTeamWalk` 为所有成员重算
- 删除单人记录后由 `deleteWalk` 为当前用户回滚重算
- 删除同行记录后由 `deleteTeamWalk` 只对当前用户做“个人视角删除 + 成就回滚重算”
- 历史页和成就详情页通过 `listMyAchievements` 直接读取结果
- 成就图片不打包进小程序
- 图片 fileID 通过配置文件单独填写

当前配置入口：
- `miniprogram/utils/achievement-assets.js`

### 3.2 成就数据来源
当前成就依赖以下数据：
- 单人记录：`listMyWalks`
- 同行记录：`listMyTeamWalks` 返回的“当前用户仍可见”的同行记录
- 成就结果：`listMyAchievements`

当前云端持久化集合：

- `userAchievements`

集合主要字段：

- `_id` = 当前用户 `openid`
- `userId`
- `achievements`
- `summary`
- `updatedAt`

判定依赖的字段包括：
- `themeCategory`
- `themeSnapshot.category`
- `routeStats.distanceMeters`
- `photoList`
- `missionAssetMap`
- `locationName`
- `season`
- `generationContext.season`
- `createdAt`

### 3.3 图片展示链路
当前图片展示链路已统一为：
- 在 `achievement-assets.js` 中填写微信云后台返回的真实 `cloud://fileID`
- 页面首次展示时统一批量换取临时 URL，并做本地缓存
- 历史页会把成就结果快照缓存到 `app.globalData`
- 详情页优先复用历史页快照，减少重复解析与重复拉图

当前生产建议：
- `achievement-assets.js` 中统一填写真实 `cloud://fileID`
- 上传时使用英文目录与英文文件名
- 对应云文件设置为“所有用户可读”

当前不建议：
- 手工猜测或拼接 fileID 路径
- 使用临时签名 HTTPS 地址作为正式长期配置

### 3.4 页面兜底
为了保证成就页不会因为资源问题直接空白，当前已实现：
- 图片未配置时，显示文字占位卡
- 图片加载失败时，自动回退到占位卡
- 历史页和详情页成就图已开启 `lazy-load`

这保证了：
- 逻辑可联调
- 页面不会因为单张图失败而崩掉

### 3.5 云函数共享规则维护方式
当前为了适配微信云函数“按目录部署”的方式，成就规则采用：

- 单一源码：`cloudfunctions/shared/achievement-runtime.js`
- 目标文件：
  - `cloudfunctions/createWalk/achievement.js`
  - `cloudfunctions/finishTeamWalk/achievement.js`
  - `cloudfunctions/listMyAchievements/achievement.js`
  - `cloudfunctions/deleteWalk/achievement.js`
  - `cloudfunctions/deleteTeamWalk/achievement.js`
- 同步脚本：`scripts/sync_cloud_achievement_runtime.js`

维护规则：

1. 只修改 `cloudfunctions/shared/achievement-runtime.js`
2. 执行 `node scripts/sync_cloud_achievement_runtime.js`
3. 重新部署上面 5 个云函数
4. 再做联调验证

## 5. 过程中遇到的问题

### 5.1 小程序包体问题
最初尝试把成就图片直接放到项目里，但原素材体积较大，不适合作为小程序静态资源直接打包。

结论：
- 已调整为图片走外部地址/云后台引用，不进项目包

### 5.2 透明背景 PNG 接入需求
成就图存在透明背景需求，因此不能简单退化成普通 JPG 方案。

结论：
- 当前文档与实现已经统一为透明背景 PNG 方案

### 5.3 `cloud://` 图片展示链路不稳定
在实际接入过程中，先后出现过两类误判：
- 误以为 `cloud://` 本身不能直出
- 误以为中文路径或中文文件名是核心问题

实际排查后的结论是：
- `cloud://fileID` 在当前项目中可以直接渲染
- 真正影响渲染的是文件权限与 fileID 是否真实有效
- 将成就图设置为“所有用户可读”后，fileID 直出可用

当前结论：
- 生产环境推荐直接使用真实 `cloud://fileID`
- 页面运行时再统一解析成临时 URL，并做缓存

### 5.4 图片空白与已解锁显示不一致
出现过“已解锁但没有图片”的情况，原因主要是：
- 图片路径为空
- 图片路径无效
- fileID 对应的云文件并不存在
- 云文件权限不是“所有用户可读”
- 图片加载失败但页面无明确兜底

当前处理：
- 已增加占位卡和失败回退机制

### 5.5 云文件权限问题
这次排查中已确认：
- 同一云环境下，某些 `cloud://fileID` 能渲染，某些不能
- 进一步验证后发现根因是云文件权限差异，而不是前端渲染方式本身

当前结论：
- 成就图上传后必须统一设置为“所有用户可读”
- 否则即使 fileID 正确，前端 `<image>` 也可能无法渲染

### 5.6 素材目录与命名问题
为避免上传到云存储后再出现路径混乱，本地素材目录已经统一整理为：
- `D:\achievements`

当前命名规则：
- 目录名使用英文
- 文件名使用英文 snake_case
- 与 `achievement-assets.js` 中的 key 一一对应

### 5.7 四季成就历史数据不完整
旧记录中没有 `season` 字段，因此历史老数据无法严格按新规则回算季节成就。

当前结论：
- 新规则对新记录已经生效
- 老记录若缺少字段，不建议强行回推

### 5.8 前端分页导致成就漏算
之前历史页和成就详情页只拉 `20/50` 条记录，并在本地计算成就，老用户早期数据会被漏掉。

当前处理：

- 已改为云端全量回算
- 成就结果持久化到 `userAchievements`
- 前端只读，不再本地参与计算

## 6. 建议进一步改进的地方

### 6.1 图片链路统一成真实 fileID
当前最优先建议：
- 所有成就图统一使用真实 `cloud://fileID`
- 不再混用 `cloud://` 和临时 HTTPS
- 上传后立刻核对权限是否为“所有用户可读”
- 素材继续保持英文目录与英文文件名

### 6.2 新解锁状态服务端化
当前“新解锁提示”依赖本地缓存，不同设备之间不会同步。

建议后续：
- 服务端记录每个用户的成就状态
- 增量解锁时由后端返回

### 6.4 图片配置从代码迁移到后台配置
当前图片地址写在前端配置文件里，维护成本偏高。

建议后续：
- 成就配置也由后端接口返回
- 前端不再手工维护每个成就图片地址

### 6.5 “不同地点”规则继续细化
当前“掌量世界”采用 `locationName` 优先、坐标兜底的弱去重方案。

建议后续：
- 引入统一地理网格规则
- 或由后端根据 POI / 经纬度聚类做去重

## 7. 后续如何修改，才能稳健适用生产环境

建议按下面顺序推进：

### 第一步：先稳定资源链路
- 统一成就图资源来源
- 全部改成真实 `cloud://fileID`
- 上传后统一设置“所有用户可读”
- 避免手工猜测路径或继续混用临时 HTTPS

### 第二步：把成就配置外移
- 让成就标题、描述、图片地址、排序由后端统一返回
- 前端只保留渲染逻辑

### 第三步：把成就配置进一步后台化
- 当前已经实现云端计算和持久化
- 下一步可以把成就标题、描述、图片地址也改成后台可配置
- 前端进一步收缩成纯展示层

### 第四步：补齐历史数据迁移策略
- 对旧记录中缺少 `season` 等字段的情况做迁移方案
- 明确哪些老数据参与回算，哪些不参与

### 第五步：增加可观测性
- 为成就计算和资源加载增加日志
- 能快速定位“成就没解锁”“图片不显示”“规则不一致”等问题

## 8. 当前结论
当前成就系统已经完成了 V1 主要功能，适合继续联调和产品验证。

当前最重要的现实结论是：
- 业务规则层已经迁移到云端统一维护
- 页面展示层已经可用
- 季节成就链路已经补齐
- 成就结果已经持久化到 `userAchievements`
- 图片资源链路已经回归为真实 `cloud://fileID` 配置 + 运行时缓存解析
- 成就图是否可显示，关键取决于 fileID 是否真实有效、云文件是否对所有用户可读

如果要进一步走向生产环境，最优先改进的方向是：
- 固化云文件上传与权限检查流程
- 把“新解锁提示”也收口到后端
- 逐步把成就配置迁移到后台配置
