适用范围：
- 微信小程序端成就页
- 成就数据计算
- 成就图片展示链路
- 成就相关记录字段保存链路

## 3. 技术说明

### 3.1 当前技术方案
V1 采用“前端本地计算 + 记录字段持久化 + 云存储 fileID 直连引用”的方案。

具体方式：
- 成就规则配置在本地代码中维护
- 历史页拉取 `listMyWalks + listMyTeamWalks` 后直接计算
- 成就图片不打包进小程序
- 图片 fileID 通过配置文件单独填写

当前配置入口：
- `miniprogram/utils/achievement-assets.js`

### 3.2 成就数据来源
当前成就依赖以下历史数据：
- 单人记录：`listMyWalks`
- 同行记录：`listMyTeamWalks`

判定依赖的字段包括：
- `themeCategory`
- `themeSnapshot.category`
- `routeStats.distanceMeters`
- `photoList`
- `missionAssetMap`
- `locationName`
- `season`
- `generationContext.season`
- `createdAt`

### 3.3 图片展示链路
当前图片展示链路已统一为：
- 在 `achievement-assets.js` 中填写微信云后台返回的真实 `cloud://fileID`
- 页面直接把 `cloud://fileID` 传给 `<image>` 组件
- 不再在页面层做 `getTempFileURL` 转换
- 不再依赖临时签名 HTTPS 地址

当前生产建议：
- `achievement-assets.js` 中统一填写真实 `cloud://fileID`
- 上传时使用英文目录与英文文件名
- 对应云文件设置为“所有用户可读”

当前不建议：
- 手工猜测或拼接 fileID 路径
- 使用临时签名 HTTPS 地址作为正式长期配置

### 3.4 页面兜底
为了保证成就页不会因为资源问题直接空白，当前已实现：
- 图片未配置时，显示文字占位卡
- 图片加载失败时，自动回退到占位卡

这保证了：
- 逻辑可联调
- 页面不会因为单张图失败而崩掉

## 4. 当前实现到了什么程度

### 4.1 已达到的程度
从产品可用性上，当前已经达到：
- 成就页可展示
- 成就规则可计算
- 四季成就口径已对齐业务要求
- 历史老记录可部分参与回算
- 新记录可正确参与成就判定
- 成就图可通过云存储 fileID 方式接入
- 佩戴成就可在足迹页顶部直接展示
- 成就详情页可直接分享

从开发阶段判断，当前属于：
- “V1 可联调可验证版本”

### 4.2 还没有达到的程度
当前还没有达到严格生产级方案，原因包括：
- 成就计算仍在前端本地完成
- 解锁状态没有服务端统一存储
- 图片 fileID 依赖手工配置
- 云文件权限若未设置为“所有用户可读”会直接导致渲染失败
- 成就素材上传流程仍依赖人工规范执行

## 5. 过程中遇到的问题

### 5.1 小程序包体问题
最初尝试把成就图片直接放到项目里，但原素材体积较大，不适合作为小程序静态资源直接打包。

结论：
- 已调整为图片走外部地址/云后台引用，不进项目包

### 5.2 透明背景 PNG 接入需求
成就图存在透明背景需求，因此不能简单退化成普通 JPG 方案。

结论：
- 当前文档与实现已经统一为透明背景 PNG 方案

### 5.3 `cloud://` 图片展示链路不稳定
在实际接入过程中，先后出现过两类误判：
- 误以为 `cloud://` 本身不能直出
- 误以为中文路径或中文文件名是核心问题

实际排查后的结论是：
- `cloud://fileID` 在当前项目中可以直接渲染
- 真正影响渲染的是文件权限与 fileID 是否真实有效
- 将成就图设置为“所有用户可读”后，fileID 直出可用

当前结论：
- 生产环境推荐直接使用真实 `cloud://fileID`
- 不再依赖页面层 `cloud:// -> https` 转换

### 5.4 图片空白与已解锁显示不一致
出现过“已解锁但没有图片”的情况，原因主要是：
- 图片路径为空
- 图片路径无效
- fileID 对应的云文件并不存在
- 云文件权限不是“所有用户可读”
- 图片加载失败但页面无明确兜底

当前处理：
- 已增加占位卡和失败回退机制

### 5.5 云文件权限问题
这次排查中已确认：
- 同一云环境下，某些 `cloud://fileID` 能渲染，某些不能
- 进一步验证后发现根因是云文件权限差异，而不是前端渲染方式本身

当前结论：
- 成就图上传后必须统一设置为“所有用户可读”
- 否则即使 fileID 正确，前端 `<image>` 也可能无法渲染

### 5.6 素材目录与命名问题
为避免上传到云存储后再出现路径混乱，本地素材目录已经统一整理为：
- `D:\achievements`

当前命名规则：
- 目录名使用英文
- 文件名使用英文 snake_case
- 与 `achievement-assets.js` 中的 key 一一对应

### 5.7 四季成就历史数据不完整
旧记录中没有 `season` 字段，因此历史老数据无法严格按新规则回算季节成就。

当前结论：
- 新规则对新记录已经生效
- 老记录若缺少字段，不建议强行回推

## 6. 建议进一步改进的地方

### 6.1 图片链路统一成真实 fileID
当前最优先建议：
- 所有成就图统一使用真实 `cloud://fileID`
- 不再混用 `cloud://` 和临时 HTTPS
- 上传后立刻核对权限是否为“所有用户可读”
- 素材继续保持英文目录与英文文件名

### 6.2 成就结果服务端化
当前前端本地计算可以满足 V1，但不适合长期生产。

建议后续改成：
- 后端统一计算成就
- 后端返回用户已解锁成就和进度
- 前端只负责展示

这样可以解决：
- 跨设备一致性
- 规则升级时的兼容问题
- 计算性能与逻辑分散问题

### 6.3 成就状态持久化到云端
当前“新解锁提示”依赖本地缓存，不同设备之间不会同步。

建议后续：
- 服务端记录每个用户的成就状态
- 增量解锁时由后端返回

### 6.4 图片配置从代码迁移到后台配置
当前图片地址写在前端配置文件里，维护成本偏高。

建议后续：
- 成就配置也由后端接口返回
- 前端不再手工维护每个成就图片地址

### 6.5 “不同地点”规则继续细化
当前“掌量世界”采用 `locationName` 优先、坐标兜底的弱去重方案。

建议后续：
- 引入统一地理网格规则
- 或由后端根据 POI / 经纬度聚类做去重

## 7. 后续如何修改，才能稳健适用生产环境

建议按下面顺序推进：

### 第一步：先稳定资源链路
- 统一成就图资源来源
- 全部改成真实 `cloud://fileID`
- 上传后统一设置“所有用户可读”
- 避免手工猜测路径或继续混用临时 HTTPS

### 第二步：把成就配置外移
- 让成就标题、描述、图片地址、排序由后端统一返回
- 前端只保留渲染逻辑

### 第三步：把成就计算迁到后端
- 保存漫步后由后端异步/同步更新成就
- 历史页直接读取成就结果
- 前端不再重复本地计算

### 第四步：补齐历史数据迁移策略
- 对旧记录中缺少 `season` 等字段的情况做迁移方案
- 明确哪些老数据参与回算，哪些不参与

### 第五步：增加可观测性
- 为成就计算和资源加载增加日志
- 能快速定位“成就没解锁”“图片不显示”“规则不一致”等问题

## 8. 当前结论
当前成就系统已经完成了 V1 主要功能，适合继续联调和产品验证。

当前最重要的现实结论是：
- 业务规则层已经基本可用
- 页面展示层已经可用
- 季节成就链路已经补齐
- 图片资源链路已经回归为直接使用 `cloud://fileID`
- 成就图是否可显示，关键取决于 fileID 是否真实有效、云文件是否对所有用户可读

如果要进一步走向生产环境，最优先改进的方向是：
- 统一稳定的 `cloud://fileID` 资源配置
- 固化云文件上传与权限检查流程
- 逐步把成就配置与成就计算迁移到后端
