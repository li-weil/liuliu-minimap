# 云开发环境重建说明

## 1. 文档目的

本文用于把当前项目迁移到一个新的微信云开发环境，并尽量一次性说明清楚：

- 需要新建哪些云数据库集合
- 建议补哪些索引
- 需要上传部署哪些云函数
- 需要配置哪些云函数环境变量
- 云存储里哪些文件会自动生成，哪些需要你手动迁移
- 小程序端还需要改哪些配置

适用前提：

- 当前项目继续运行在“微信云开发模式”
- `miniprogram/utils/config.js` 中 `apiBaseUrl` 为空
- `useCloudWalkStorage: true`
- `useCloudMediaStorage: true`

如果你准备完全切到 Web 后端模式，这份文档只需要参考其中的小程序配置和静态资源部分。

---

## 2. 先说结论

新云环境最少需要准备：

1. 创建 7 个云数据库集合
2. 上传部署 29 个云函数
3. 配置 AI / 地图相关环境变量
4. 把小程序里的 `cloudEnvId` 改成新环境 ID

云存储方面：

- 业务上传目录主要只有 `walks/` 和 `stickers/`
- 这两个目录不需要手工预创建
- 用户上传图片、视频、音频，以及 AI 生成贴纸时会自动写入
- 如果你要保留旧环境历史记录，则需要额外迁移旧环境中的媒体文件

也就是说：

- 想“从空环境重新跑起来”，不需要手工先上传业务文件到云存储
- 想“把旧数据完整搬过去”，才需要迁移旧的数据库记录和 `walks/`、`stickers/` 文件

## 3. 小程序侧必须修改的配置

修改文件：

- [miniprogram/utils/config.js](D:/liuliu-minimap/miniprogram/utils/config.js)

至少确认这几项：

```js
module.exports = {
  cloudEnvId: '你的新云环境 ID',
  apiBaseUrl: '',
  brandFontBaseUrl: 'https://你的静态站点域名/fonts',
  useCloudWalkStorage: true,
  useCloudMediaStorage: true,
};
```

建议同步检查：

- `cloudEnvId` 改成新环境 ID
- `apiBaseUrl` 保持空字符串
- `brandFontBaseUrl` 指向你自己的静态站点字体目录
- `useCloudWalkStorage` 保持 `true`
- `useCloudMediaStorage` 保持 `true`

项目配置中云函数目录和小程序目录已经配好，一般不用改：

- [project.config.json](D:/liuliu-minimap/project.config.json)

### 3.1 品牌字体静态资源

当前品牌字体已经不再从 `fonts.gstatic.com` 直拉。

请使用你自己的微信云服务静态网站托管字体文件，例如在静态站点中放置：

- `fonts/zcool-kuaile.ttf`
- `fonts/zcool-xiaowei.ttf`

然后把 [config.js](/D:/liuliu-minimap/miniprogram/utils/config.js) 中的：

- `brandFontBaseUrl`

配置成对应字体目录的基地址，例如：

- `https://your-static-site.domain/fonts`

这样小程序启动时会加载：

- `https://your-static-site.domain/fonts/zcool-kuaile.ttf`
- `https://your-static-site.domain/fonts/zcool-xiaowei.ttf`

如果未配置 `brandFontBaseUrl`，当前代码会直接跳过品牌字体加载，避免在国内网络下再去请求国外字体域名。

## 5. 建议创建的数据库索引

代码能在“无索引”时先跑起来，但为了避免查询报错或性能过差，建议在新环境创建以下索引。

### 5.1 `walkRecords`

建议索引：

- `userId` + `createdAt desc`
- `isPublic` + `createdAt desc`

对应查询：

- `listMyWalks` 按 `userId` 过滤，再按 `createdAt` 倒序
- `listPublicWalks` 按 `isPublic` 过滤，再按 `createdAt` 倒序

### 5.2 `teamWalkMembers`

建议索引：

- `roomId` + `status` + `joinedAt asc`
- `roomId` + `userId`
- `roomId` + `userId` + `status`
- `userId` + `status` + `joinedAt desc`

对应查询：

- 房间成员列表
- 当前用户是否已加入房间
- 列出我参与的同行记录

### 5.3 `teamWalkContributions`

建议索引：

- `roomId` + `updatedAt desc`
- `roomId` + `userId` + `missionKey`

对应查询：

- 房间贡献列表按更新时间倒序
- 当前成员提交某个任务时查重 / 更新

### 5.4 `teamWalkActivities`

建议索引：

- `roomId` + `createdAt desc`

对应查询：

- 房间动态流按时间倒序

### 5.5 `teamWalkRooms`

当前代码主要通过：

- 文档 ID 直查
- `_id in roomIds`

因此 `teamWalkRooms` 暂时不强制额外索引，但如果后续要做后台筛选，可以再补 `status`、`createdAt` 相关索引。

### 5.6 `userAchievements`

建议：

- 以 `openid` 作为文档 ID，通常不需要额外索引

如果后续要做后台统计或运营查询，再补：

- `updatedAt desc`

## 8. 云函数响应时间上限配置

除了环境变量，新环境里还有一个很容易漏掉的配置：

- 云函数响应时间上限

当前项目里，凡是会调用外部 API 或 AI 模型的云函数，都不建议继续使用很保守的默认超时。

原因：

- AI 文本生成本身就有波动
- 多模态核验会先换临时文件 URL，再调视觉 / 音频 / 文本模型
- 贴纸生图还包含“发起生成任务 -> 轮询结果 -> 下载图片 -> 上传云存储”这整条链路
- 地图周边 POI 虽然比 AI 轻，但也属于外部服务请求，默认超时太短时容易误判失败

建议你在微信云开发控制台里，把下面这些云函数的“超时时间 / 响应时间上限”单独调高。

### 8.1 建议提高响应时间上限的云函数

#### AI 相关

- `generateTheme`
- `generateRandomTheme`
- `generateCombinedTheme`
- `getLocationContext`
- `verifyMission`
- `generateSticker`

#### 外部地图 API 相关

- `fetchNearbyPois`

### 8.2 建议值

如果你希望先求稳，建议直接这样配：

- `generateSticker`: `60s`
- `verifyMission`: `60s`
- `generateTheme`: `30s`
- `generateRandomTheme`: `30s`
- `generateCombinedTheme`: `30s`
- `getLocationContext`: `15s`
- `fetchNearbyPois`: `15s`

如果你的云开发控制台支持更高超时，也可以按下面的思路理解：

- 文本类 AI：至少 `30s`
- 多模态核验：至少 `60s`
- 生图类：优先 `60s`
- 普通外部地图查询：`10s-15s`

### 8.3 为什么 `generateSticker` 和 `verifyMission` 要特别高

`generateSticker` 不是单次简单请求，它实际可能包含：

1. 文本模型生成贴纸方案
2. 图像模型生图
3. 轮询生图任务结果
4. 下载生成图片
5. 上传到云存储

`verifyMission` 也不是单次请求，它实际可能包含：

1. 把 `cloud://` 文件换成临时 URL
2. 图片 / 视频走视觉模型
3. 音频走音频模型
4. 再用文本模型汇总评分结果

所以这两个函数如果仍然沿用过短的响应时间上限，很容易在新环境里表现成：

- 偶发超时
- 有时成功有时失败
- 控制台看起来像“网络问题”，其实是函数超时

### 8.4 与代码中的请求超时不是一回事

这里要区分两层超时：

1. 云函数运行时的“响应时间上限”
2. 代码里调用外部服务时的请求超时

例如：

- `AI_REQUEST_TIMEOUT_MS`
- `MAP_REQUEST_TIMEOUT_MS`
- `generateSticker/config.js` 里的 `timeoutMs`

这些只是“函数内部某次 HTTP 请求等多久”，并不能替代云函数自身的响应时间上限。

所以迁移新环境时，两个都要配：

- 云函数控制台里调高响应时间上限
- 环境变量里设置内部请求超时

### 8.5 迁移时可以直接照抄的配置结论

新环境里，以下云函数建议在部署后立刻检查响应时间上限：

- `generateTheme`: `30s`
- `generateRandomTheme`: `30s`
- `generateCombinedTheme`: `30s`
- `getLocationContext`: `15s`
- `fetchNearbyPois`: `15s`
- `verifyMission`: `60s`
- `generateSticker`: `60s`

如果你只想记一句话：

- 凡是“会调外部 API 或 AI”的云函数，都不要用默认短超时

## 9. 云存储需要上传什么

### 8.1 从空环境启动时

不需要手工预上传任何业务文件。

原因：

- 单人记录里的图片、视频、音频由前端上传时自动写入 `walks/`
- 同行模式里的任务素材也复用同一套上传能力，自动写入 `walks/`
- AI 贴纸由 `generateSticker` 云函数自动写入 `stickers/`

当前代码里的主要写入路径：

- `walks/xxx.xxx`
- `stickers/xxx.png`

也就是说，新环境里不需要你先建空目录或先传占位文件。

### 8.2 如果要迁移旧环境历史数据

你需要一起迁移：

1. `walkRecords` 里的历史记录数据
2. `teamWalkRooms`
3. `teamWalkMembers`
4. `teamWalkContributions`
5. `teamWalkActivities`
6. 云存储中的 `walks/` 文件
7. 云存储中的 `stickers/` 文件

否则会出现：

- 数据库里有记录，但媒体文件失效
- 详情页能打开，但图、视频、音频或贴纸显示不出来

### 8.3 是否需要上传仓库里的本地素材到云存储

通常不需要。

例如这些资源：

- `miniprogram/assets/**`
- `docs/image*.png`

它们是仓库静态文件，不走云存储业务桶。

只要正常上传小程序代码即可，不需要额外手工传到云存储。

## 12. 容易漏掉的点

### 12.1 代码里还保留了旧环境 ID

当前仓库里的 [miniprogram/utils/config.js](D:/liuliu-minimap/miniprogram/utils/config.js) 仍写着旧的 `cloudEnvId`。迁移到新环境后，这里必须改。

### 12.2 代码里存在硬编码 key，但不建议继续沿用

仓库里部分云函数配置文件还保留了旧 key 兜底。新环境可以先跑，但不建议继续把生产 key 写在代码里。

更稳妥的做法是：

- 把 key 全部改为环境变量
- 把配置文件只保留非敏感默认值

### 12.3 同行模式不是只建一个集合就够

同行模式实际依赖 4 个集合联动：

- `teamWalkRooms`
- `teamWalkMembers`
- `teamWalkContributions`
- `teamWalkActivities`

少建任何一个，相关云函数都会失败。

### 12.4 云存储目录不需要预建，但旧数据迁移时一定要迁文件

这是最容易混淆的一点：

- 新环境冷启动：不用预传文件
- 旧环境平移：必须迁媒体文件

### 12.5 只配环境变量，不调响应时间上限也会出问题

这也是迁移新环境时的高频遗漏项。

特别是下面这些函数：

- `generateTheme`
- `generateRandomTheme`
- `generateCombinedTheme`
- `getLocationContext`
- `fetchNearbyPois`
- `verifyMission`
- `generateSticker`

即使 key 都配对了，如果云函数响应时间上限太短，仍然会表现为：

- 生成超时
- 偶发失败
- 外部接口请求看起来正常，但函数提前被平台中断

所以建议把第 8 节当成必做项，而不是优化项。

## 14. 结论

如果你的目标只是“在另一个新云开发环境把项目重新跑起来”，你现在真正需要做的只有五件事：

1. 建 7 个集合
2. 部署 29 个云函数
3. 配好 AI / 地图环境变量
4. 把调用外部 API / AI 的云函数响应时间上限调高
5. 把小程序 `cloudEnvId` 改到新环境

如果你的目标是“把旧环境完整迁过去继续用”，再额外补：

6. 迁数据库历史记录
7. 迁云存储里的 `walks/` 和 `stickers/`

这份仓库当前并不依赖“预先手工上传一批固定业务文件到云存储”才能启动，云存储主要是运行时自动生成和上传。
