# 同行模式 MVP PRD

## 1. 文档目的

本文定义“小程序多人模式”第一期 MVP 的产品方案。

多人模式第一期不做完整社交产品，只做最容易验证、也最适合当前代码架构的 `同行模式`。

本文重点回答以下问题：

- 为什么要做同行模式
- MVP 具体要做什么、不做什么
- 用户如何从发起到结束完成一场同行漫步
- 已实现哪些页面、状态、能力和数据对象
- 这一期的验收标准是什么

本文基于当前仓库已实现的单人流程：

- 探索页生成主题与任务
- 记录页任务打卡与轨迹记录
- 足迹页历史回看
- 打卡卡片 / 分享能力

## 2. 背景与目标

### 2.1 背景

小程序最初的主流程围绕“一个人完成一场漫步”设计：

1. 选择地点
2. 生成主题和任务
3. 进入记录页
4. 上传图片 / 视频 / 录音 / 文字
5. 完成后保存到足迹

这条流程已经能支撑单人体验。但在真实使用场景里，城市漫步常常是多人一起发生，例如：

- 朋友一起逛街、一起做主题任务
- 情侣共同记录一场散步
- 同学或搭子一起完成一次城市探索
- 小型社群线下活动一起走一条主题路线

因此，多人模式第一期不做“聊天群”或“实时定位社交”。它只解决一件事：几个人围绕同一个漫步主题，一起出发、一起参与、一起留下纪念。

### 2.2 产品目标

同行模式 MVP 有三个目标：

1. `一起出发`
让一个用户可以顺畅地发起一场多人漫步，并邀请好友加入同一房间。

2. `一起参与`
让房间内所有成员都围绕同一套主题和任务贡献内容，而不是只有发起人操作。

3. `一起留下纪念`
让一场多人漫步结束后，能生成一份团队纪念结果，同时每个参与者都能在自己的历史里看到这次同行经历。

### 2.3 成功标准

同行模式 MVP 上线后，重点验证以下问题：

- 用户是否愿意发起多人漫步
- 被邀请者是否愿意加入并实际参与
- 多人任务是否提升内容贡献量
- 团队纪念页是否能带来更强分享意愿

建议重点关注的指标包括：

- 同行房间创建数
- 邀请打开率
- 入房成功率
- 开始漫步率
- 房间人均贡献次数
- 团队任务完成率
- 团队结束率
- 团队结果页分享率

## 3. 版本定位

### 3.1 本期模式定义

本期只做 `同行模式`。

同行模式定义为：

- 多名用户线下同时参与
- 共享同一地点、同一主题、同一任务集合
- 每个人都可以提交自己的任务素材
- 任务完成情况以“团队完成”为主，同时保留个人参与痕迹
- 结束后生成一份团队结果页

### 3.2 为什么先做同行模式

相比“接力模式”或“PK 模式”，同行模式最适合当前单人架构：

- 可以直接复用现有探索页的选点与主题生成能力
- 可以复用当前记录页的任务、素材上传与轨迹逻辑
- 可以复用历史详情页的结果展示思路
- 用户理解成本最低，最接近真实线下使用场景

### 3.3 本期不做的内容

为了让第一期可落地、可验证，本期不做：

- 房间聊天
- 实时语音
- 精确实时位置共享
- 多人自由编辑主题
- 多角色权限体系
- 复杂分工任务
- 团队 PK / 排行榜
- 社交关系维护
- 常用搭子、群组、社团

## 4. 用户角色

### 4.1 房主

房主是本次同行漫步的发起人，拥有以下权限：

- 创建同行房间
- 分享邀请
- 查看成员加入情况
- 开始漫步
- 结束漫步
- 在未开始前解散房间

### 4.2 成员

成员是通过邀请加入房间的参与者，拥有以下权限：

- 查看房间主题、地点和任务
- 加入房间
- 在开始后参与任务记录
- 上传图片、视频、录音和备注
- 查看团队进度与团队结果

### 4.3 未登录用户

未登录用户可以打开邀请入口，但正式加入前，需要先完成登录和资料同步。

## 5. 核心用户场景

### 5.1 场景一：朋友同行

用户 A 在探索页选好地点并生成主题后，发起“同行漫步”，把邀请分享给朋友 B、C。

B、C 加入后，三个人看到同一套任务。漫步过程中，每个人都可以对任务提交自己的图片和备注。结束后，系统生成一页团队结果，三个人都能保存和查看。

### 5.2 场景二：情侣散步

一名用户发起同行漫步并邀请另一人加入，两人共同完成一套任务，各自上传看到的街景或声音，最后得到一份共同纪念页。

### 5.3 场景三：小型线下活动

活动组织者作为房主发起房间，成员扫码或分享进入房间，开始后围绕统一主题进行探索。活动结束后生成统一结果页，用于回看和二次分享。

## 6. MVP 范围

### 6.1 本期包含

本期必须包含以下能力：

- 从探索页发起同行房间
- 通过微信分享邀请加入房间
- 房间等待页
- 房主开始同行漫步
- 团队共享主题、地点、任务
- 成员围绕任务提交素材
- 团队任务进度展示
- 团队动态流
- 房主结束同行漫步
- 团队结果页
- 个人历史页接入同行记录

### 6.2 本期排除

本期不进入开发范围的内容：

- 房间评论区
- 素材下回复
- 成员间点赞表情
- 成员实时地图位置
- 自动匹配陌生人
- 团队成就体系
- 团队长期关系链

## 7. 功能概览

同行模式 MVP 拆成 6 个功能模块：

1. 组队发起与邀请
2. 房间等待与成员管理
3. 团队共享任务
4. 成员贡献与轻动态
5. 团队结束与结果页
6. 个人历史接入

## 8. 详细功能设计

### 8.1 组队发起与邀请

#### 8.1.1 入口

入口位于探索页。

探索页已有：

- 地点选择
- 模式选择
- 主题生成

本期新增：

- `单人 / 同行` 模式入口
- 在同行模式下，把原本“开始漫步”调整为“发起同行漫步”

#### 8.1.2 发起条件

满足以下条件后才能创建同行房间：

- 已选定地点
- 已生成当前主题
- 当前用户已具备登录态

若用户未登录，则先引导到个人页完成登录 / 资料同步。

#### 8.1.3 创建房间后的结果

创建成功后进入“房间等待页”，房间中固定以下信息：

- 房间编号 / roomId
- 房主信息
- 主题快照
- 地点信息
- 模式信息
- 任务清单
- 房间状态

#### 8.1.4 邀请方式

本期采用微信小程序原生分享作为唯一邀请方式。

分享卡信息建议包含：

- 主题标题
- 地点名
- 当前人数 / 可加入人数提示
- 一句邀请文案

分享落地后打开“加入房间页”。

### 8.2 房间等待与成员管理

#### 8.2.1 房间等待页信息结构

房间等待页用于承接“创建成功”与“开始前等待队友加入”的阶段。

页面建议包含以下模块：

- 主题卡
- 地点信息
- 成员列表
- 房间状态文案
- 邀请按钮
- 房主操作按钮

#### 8.2.2 成员列表

成员列表展示：

- 头像
- 昵称
- 房主标识
- 当前加入状态

本期成员列表不做复杂管理，不支持转移房主，不支持踢人。

#### 8.2.3 房主操作

房主在等待页具备以下操作：

- 再次邀请
- 开始漫步
- 解散房间

#### 8.2.4 成员操作

普通成员在等待页具备以下操作：

- 查看主题和任务
- 查看当前加入成员
- 退出房间（仅开始前）

### 8.3 团队共享任务

#### 8.3.1 任务形态

MVP 只做 `共享任务`，不做分工任务和协作任务。

共享任务的定义是：

- 全队看到同一份任务列表
- 任意成员都可以对任意任务提交素材
- 团队完成度按任务聚合展示

#### 8.3.2 任务来源

任务直接复用当前探索页生成的单人主题结果：

- 纯粹模式：通常 1 个任务
- 进阶模式：通常 3 个任务

多人模式不额外生成一套全新任务，而是复用现有主题生成结果，以降低实现复杂度。

#### 8.3.3 任务完成规则

建议本期采用最轻规则：

- 某任务只要有任意成员完成，即记为团队完成
- 某成员是否完成，单独记录为个人完成状态

这样页面上会有两层状态：

- `团队完成`
- `我的完成`

#### 8.3.4 任务补充位

保留当前单人模式中的“总结与补充”概念，但在同行模式中升级为：

- 团队总结位
- 每个成员都可以往其中提交补充素材

### 8.4 成员贡献与轻动态

#### 8.4.1 可贡献素材类型

本期每个成员可提交的素材类型与当前单人模式保持一致：

- 图片
- 视频
- 录音
- 文字备注

#### 8.4.2 贡献方式

每位成员在团队记录页中：

- 选择一个任务
- 提交自己的素材
- 触发自己的任务完成状态

团队页应能看到该任务下来自不同成员的贡献。

#### 8.4.3 动态流

本期做 `系统动态流`，不做自由聊天。

动态流事件包括：

- 某成员加入房间
- 房主开始漫步
- 某成员完成某任务
- 某成员上传了图片 / 视频 / 录音 / 备注
- 房主结束漫步

动态流的产品目标是：

- 让成员感知“队友也在参与”
- 形成轻量实时陪伴感
- 提升参与欲望

#### 8.4.4 为什么不做聊天

聊天会显著增加产品复杂度和审核压力，也容易让页面从“围绕漫步任务”偏离到“普通群聊”。

因此本期坚持：

- 互动围绕任务展开
- 用动态流替代聊天

### 8.5 团队地图与位置

#### 8.5.1 本期设计原则

本期不做精确实时位置共享。

主要原因包括：

- 隐私压力大
- 实时同步复杂度高
- 容易把 MVP 做重

#### 8.5.2 本期保留的地点感

本期只保留以下与位置相关的能力：

- 房间统一出发地点
- 主题与地点绑定
- 成员可选择记录自己的轨迹
- 团队结果页展示路线摘要

#### 8.5.3 轨迹呈现建议

如开发成本允许，可在团队结果页展示：

- 团队总时长
- 团队总素材数
- 房主轨迹或合并轨迹摘要图

如果时间不足，本期也可以先只展示地点名和团队统计信息，不强依赖团队地图。

### 8.6 团队结束与结果页

#### 8.6.1 结束动作

本期由房主控制结束漫步。

结束条件：

- 房间状态必须为 `进行中`
- 只有房主可触发“结束同行漫步”

#### 8.6.2 结束后生成内容

结束后系统需要生成一份团队结果页，至少包含：

- 团队主题
- 地点
- 参与成员
- 团队任务完成情况
- 图片 / 视频 / 录音总数
- 任务贡献摘要
- 团队精选内容

#### 8.6.3 团队纪念页的价值

团队纪念页是同行模式的核心结果物，承担三项职责：

- 让这次多人漫步有“共同完成”的结束感
- 让用户有可分享内容
- 让个人历史中能看到“我参与了这场同行漫步”

### 8.7 个人历史接入

#### 8.7.1 接入原则

个人历史页应同时容纳：

- 单人漫步记录
- 同行漫步参与记录

#### 8.7.2 展示方式

建议在历史卡片中增加类型标签：

- `单人`
- `同行`

对于同行记录，点击后进入团队详情页，而不是单人详情页。

#### 8.7.3 个人视角信息

个人历史里应至少能知道：

- 这是一次同行漫步
- 我是参与者还是房主
- 这次同行的主题和地点
- 团队成员数
- 我是否有贡献内容

## 9. 页面方案

本节记录当前已经落地的同行模式页面，而不是待开发页面清单。

### 9.1 探索页

页面位置：

- `miniprogram/pages/index/index`

新增内容：

- 单人 / 同行切换
- 发起同行漫步按钮

复用内容：

- 地点选择
- 主题生成
- 模式切换

### 9.2 加入房间页

当前已实现页面：

- `pages/team-join/team-join`

页面内容：

- 房间摘要
- 主题与地点
- 任务预览
- 当前人数摘要
- 加入按钮

权限边界：

- 未加入用户只能看到房间基础预览，不能读取完整成员列表、贡献内容和动态流。
- 房间开始或结束后不允许再加入。

### 9.3 房间等待页

当前已实现页面：

- `pages/team-room/team-room`

页面内容：

- 主题卡
- 地点信息
- 成员列表
- 最近动态
- 邀请按钮
- 开始按钮
- 解散 / 退出按钮

当前规则：

- 房主在 `waiting` 状态可开始同行或解散房间。
- 普通成员在 `waiting` 状态可退出房间。
- 房间进入 `active` 后，成员从房间页进入团队记录页继续参与。

### 9.4 团队记录页

当前已实现页面：

- `pages/team-record/team-record`

页面内容：

- 团队主题头图
- 团队进度
- 任务列表
- 我的提交区
- 动态流
- 结束按钮

当前规则：

- `同步给团队` 即为当前任务的打卡动作，不再单独提供“我已完成该任务”勾选。
- 文字、图片、视频、录音会先形成本地任务草稿；用户同步成功后再写入团队贡献。
- 房主结束同行前，会拦截自己或成员仍有带内容但未同步的任务草稿。

### 9.5 团队详情页

当前已实现页面：

- `pages/team-detail/team-detail`

页面内容：

- 团队主题
- 地点
- 成员
- 团队统计
- 任务回顾
- 成员贡献素材
- 任务级团队打卡卡片
- 分享按钮
- 已结束记录删除入口

当前规则：

- `active` 状态展示“重新进入这场同行”。
- `waiting` 状态展示“进入房间继续组队”。
- 已结束同行记录的删除只影响当前用户自己的纪念卡册和成就统计，不影响其他成员。

## 10. 核心流程

### 10.1 发起流程

1. 用户在探索页选择 `同行`
2. 选定地点并生成主题
3. 点击“发起同行漫步”
4. 创建房间成功
5. 进入房间等待页
6. 分享邀请给好友

### 10.2 加入流程

1. 被邀请者打开分享卡
2. 进入加入房间页
3. 若未登录，则先引导登录
4. 登录成功后加入房间
5. 加入成功后进入房间等待页

### 10.3 开始流程

1. 房主在等待页点击“开始同行漫步”
2. 房间状态变为进行中
3. 所有成员可进入团队记录页
4. 开始围绕任务提交素材

### 10.4 过程流程

1. 成员选择某个任务
2. 上传图片 / 视频 / 录音 / 备注
3. 系统记录该成员贡献
4. 更新团队任务进度
5. 生成一条动态流记录

### 10.5 结束流程

1. 房主点击“结束同行漫步”
2. 系统汇总任务、成员和素材
3. 生成团队结果页
4. 在每位参与者个人历史中落一条同行记录

## 11. 状态与规则

### 11.1 房间状态

当前房间状态定义如下：

- `waiting`：等待加入
- `active`：进行中
- `finished`：已结束
- `dissolved`：历史兼容状态。当前房主在开始前解散房间时，会归档为 `finished`，旧 `dissolved` 数据通过 `repairDissolvedTeamRooms` 一次性云函数迁移。

### 11.2 成员状态

成员状态定义如下：

- `joined`：已加入
- `left`：已退出

### 11.3 关键规则

当前已经落地的规则：

1. 仅房主可开始和结束漫步。
2. 房间开始后不允许新成员加入。
3. 房间未开始前，成员可退出房间。
4. 房主未开始前可解散房间，并归档为已结束记录。
5. 房间结束后只允许查看，不允许继续提交内容。
6. 已结束同行记录支持当前成员从自己的纪念卡册删除，不联动其他成员视角。
7. 团队完成不等于所有成员都完成，当前团队完成以“某任务有任意成员完成”为准。

## 14. 风险与约束

### 14.1 产品风险

- 如果房间等待页太弱，邀请成功后用户容易流失
- 如果任务只是简单共享，没有团队感，体验会退化成“同房间单人模式”
- 如果团队结果页没有纪念价值，分享动力会不足

### 14.2 技术风险

- 多成员贡献聚合会引入新的数据模型复杂度
- 房间状态同步与成员状态同步需要控制并发
- 历史页需要兼容单人记录与团队记录两种结构

### 14.3 范围风险

如果第一期同时加入聊天、实时定位和复杂任务规则，交付风险会明显增大，因此必须控制范围。

## 15. 验收标准

本期上线前，至少应满足以下验收标准：

1. 用户可以从探索页成功创建同行房间。
2. 用户可以通过分享进入加入房间页并成功加入。
3. 房间等待页可正确展示主题、地点和成员信息。
4. 房主可以开始同行漫步。
5. 成员可以对团队任务提交至少一种素材。
6. 团队页可以展示任务进度和动态流。
7. 房主可以结束同行漫步。
8. 系统可以生成团队结果页。
9. 参与者可以在个人历史中看到本次同行记录。

## 17. 当前结论

同行模式 MVP 的最佳定义可以概括为：

“几个人围绕同一个漫步主题一起出发、一起贡献、一起完成，并在结束后获得一份共同纪念结果。”

对于当前项目来说，这是一条既有产品价值、又能在现有单人架构基础上相对平滑扩展的路径。

本期最重要的不是把多人模式做重，而是先验证以下三件事：

- 是否有人愿意发起
- 是否有人愿意加入并共同贡献
- 团队纪念结果是否足够有吸引力

如果这三件事成立，再往后扩展聊天、实时位置、分工任务和关系链，才更稳妥。
