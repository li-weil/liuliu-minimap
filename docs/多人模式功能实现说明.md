# 多人模式功能实现说明

## 1. 文档目的

本文用于沉淀当前“小程序多人模式”功能的真实实现情况。

本文不以 PRD 预期范围代替代码实际状态，而是基于当前仓库已经落地的代码，回答以下问题：

- 当前已经实现了哪些多人模式功能
- 这些功能采用了什么技术方案
- 当前实现到了什么程度
- 当前仍存在哪些问题与风险
- 后续如何继续修改，才能更稳健地适配生产环境
- 当前版本应该如何测试

本文中的“多人模式”，当前特指第一期 `同行模式`。

## 2. 功能范围与代码位置

当前多人模式相关的主要代码位置如下：

- 探索页同行入口：`miniprogram/pages/index/index.js`
- 探索页同行入口结构：`miniprogram/pages/index/index.wxml`
- 同行加入页：`miniprogram/pages/team-join/team-join.js`
- 同行房间页：`miniprogram/pages/team-room/team-room.js`
- 同行记录页：`miniprogram/pages/team-record/team-record.js`
- 团队结果页：`miniprogram/pages/team-detail/team-detail.js`
- 多人模式服务层：`miniprogram/services/team.js`
- 接口归一化与团队记录归一化：`miniprogram/services/api.js`
- 历史列表与团队记录接入：`miniprogram/pages/history/history.js`
- 历史卡片类型兼容：`miniprogram/components/walk-card/walk-card.js`

当前新增的云函数如下：

- `cloudfunctions/createTeamRoom`
- `cloudfunctions/getTeamRoomDetail`
- `cloudfunctions/joinTeamRoom`
- `cloudfunctions/leaveTeamRoom`
- `cloudfunctions/startTeamWalk`
- `cloudfunctions/submitTeamContribution`
- `cloudfunctions/listTeamActivities`
- `cloudfunctions/finishTeamWalk`
- `cloudfunctions/getTeamWalkDetail`
- `cloudfunctions/listMyTeamWalks`

当前多人模式涉及的云数据库集合如下：

- `teamWalkRooms`
- `teamWalkMembers`
- `teamWalkContributions`
- `teamWalkActivities`

整体链路如下：

1. 探索页切换到“同行”模式。
2. 基于当前地点与主题创建同行房间。
3. 房主通过分享邀请队友。
4. 队友打开邀请页加入房间。
5. 房主开始同行漫步。
6. 队员在团队记录页围绕同一任务集合提交素材。
7. 房主结束同行漫步。
8. 进入团队结果页。
9. 个人历史页展示“单人记录 + 同行记录”混合列表。

## 3. 当前实现概览

### 3.1 当前完成度判断

如果按产品完成度理解，当前多人模式可以分为以下几档：

- `同行模式主链路`：已初步打通
- `房间与邀请`：已达到基础可用
- `团队任务与提交`：已达到基础可用
- `团队结果回看`：已达到基础可用
- `社交体验与生产化能力`：仍较初级

### 3.2 一句话总结当前状态

当前已经不是“只有设计稿”的阶段，而是已经具备一条可以跑通的最小闭环：

- 能发起房间
- 能邀请加入
- 能开始同行
- 能提交团队任务素材
- 能结束并回看团队结果

但当前仍属于 `MVP P0 的初步实现版本`，离可稳定上线的生产级实现还有明显距离。

## 4. 当前已经实现的功能

### 4.1 探索页同行入口

当前在探索页已经增加：

- `单人 / 同行` 模式切换
- 同行模式下复用现有地点选择和主题生成能力
- 主题卡底部按钮可切换为“发起同行漫步”

当前逻辑为：

- 单人模式仍进入原有记录页
- 同行模式则调用 `createTeamRoom` 创建房间并跳转到房间页

### 4.2 同行房间创建

当前已实现房主发起同行房间能力。

房间创建时会保存：

- 房主信息
- 当前主题快照
- 地点信息
- 模式信息
- 任务集合
- 房间状态

创建成功后进入“同行房间页”。

### 4.3 房间加入与成员展示

当前已实现：

- 房主分享邀请卡
- 队友通过 `roomId` 打开加入页
- 已登录用户加入房间
- 房间页展示当前成员列表
- 房间动态展示“谁发起了房间”“谁加入了房间”

当前成员展示信息包括：

- 昵称
- 头像
- 房主标识

### 4.4 房间等待与出发

当前房间页支持：

- 查看主题
- 查看地点
- 查看任务列表
- 查看最近动态
- 房主开始同行漫步
- 成员或房主在开始前退出 / 解散房间

开始后，房间状态从 `waiting` 切换为 `active`。

### 4.5 团队任务记录

当前团队记录页支持：

- 查看共享任务列表
- 切换当前任务
- 填写文字备注
- 上传图片
- 上传视频
- 录音
- 勾选“我已完成该任务”
- 将当前任务内容同步到团队

同步后会更新：

- 当前任务参与人数
- 当前任务团队完成状态
- 团队动态流
- 团队统计信息

### 4.6 团队动态流

当前已实现轻量动态流，不做自由聊天。

已实现的动态类型包括：

- 发起房间
- 成员加入
- 成员退出
- 房主开始同行
- 某成员更新任务记录
- 某成员完成任务
- 房主结束同行

### 4.7 团队结束与结果页

当前已实现：

- 房主结束同行漫步
- 汇总团队统计信息
- 生成团队摘要文案
- 进入团队结果页

当前团队结果页支持展示：

- 主题信息
- 地点信息
- 团队统计
- 成员列表
- 任务回顾
- 每个任务下的成员贡献
- 团队动态

### 4.8 历史页接入

当前历史页已不再只展示单人记录，而是合并了：

- `listMyWalks`
- `listMyTeamWalks`

历史卡片中已经区分：

- `单人`
- `同行`

点击同行卡片时，会跳到团队结果页，而不是单人详情页。

## 5. 技术实现说明

### 5.1 前端页面结构

当前多人模式前端采用“独立页面 + 复用现有入口”的实现方式。

设计上没有强行把多人逻辑塞进原单人 `record` 页，而是新增了以下页面：

- `team-join`
- `team-room`
- `team-record`
- `team-detail`

这样做的好处是：

- 不会过度污染原有单人记录页状态
- 多人模式可以独立演进
- 页面职责相对清晰

### 5.2 服务层封装

多人模式单独抽出 `miniprogram/services/team.js` 作为服务层入口。

当前已封装的服务包括：

- 创建房间
- 拉取房间详情
- 加入房间
- 退出房间
- 开始同行
- 提交团队贡献
- 拉取动态流
- 结束同行
- 拉取团队结果
- 拉取我参与的团队记录

这样前端页面不直接依赖云函数名，后续切 Web 后端时也更容易迁移。

### 5.3 接口归一化

`miniprogram/services/api.js` 当前新增了团队数据归一化逻辑，主要包括：

- `normalizeTeamMember`
- `normalizeTeamContribution`
- `normalizeTeamActivity`
- `normalizeTeamWalkRecord`

归一化后的团队记录结构中，已经统一包含：

- `themeTitle`
- `locationName/locationContext/locationAddress`
- `status`
- `walkMode`
- `themeSnapshot`
- `members`
- `contributions`
- `activities`
- `teamStats`
- `routeStats`

这样历史页和团队结果页可以用统一的前端消费结构。

### 5.4 云函数实现结构

当前团队模式采用“一个动作一个云函数”的轻量拆分方式。

#### 1. 房间主表

`teamWalkRooms` 当前保存：

- 房主
- 房间状态
- 主题快照
- 地点信息
- 团队统计
- 团队摘要
- 封面图

#### 2. 成员表

`teamWalkMembers` 当前保存：

- roomId
- userId
- 昵称
- 头像
- 身份角色
- 加入状态

#### 3. 贡献表

`teamWalkContributions` 当前按“成员 + 任务”维度保存任务提交内容，包括：

- noteText
- photoList
- videoList
- audioList
- completed

#### 4. 动态表

`teamWalkActivities` 当前保存动态流事件，包括：

- type
- content
- payload
- userId
- createdAt

### 5.5 媒体上传策略

团队记录页目前复用了单人模式现有的上传能力：

- 图片、视频、录音先走前端上传
- 由 `requestUpload` 负责上传到云存储
- 上传后的 `cloud://` 或 URL 再写回团队贡献表

当前团队模式并没有单独实现新的上传协议，而是沿用现有单人模式上传链路。

### 5.6 登录与权限依赖

当前多人模式直接复用已有用户系统。

即：

- 发起房间前需要已登录
- 加入房间前需要已登录
- 房间与提交权限通过当前用户 `OPENID` / userId 识别

当前房主权限的判断主要依赖：

- `room.ownerUserId`
- `member.role === 'owner'`

## 6. 当前实现到了什么程度

### 6.1 从功能闭环角度判断

当前已经达到：

- `产品链路可跑通`

也就是：

- 用户可以完成一次最基本的“多人发起 -> 多人参与 -> 团队结束 -> 团队回看”

### 6.2 从产品体验角度判断

当前体验属于：

- `基础版多人模式`

已经具备多人协作雏形，但社交感、氛围感和纪念产物丰富度都还偏弱。

### 6.3 从工程稳定性角度判断

当前工程状态属于：

- `开发中可联调`

但还不能认为已经达到稳定生产交付状态。

主要原因包括：

- 缺少系统化测试
- 缺少数据库初始化规范
- 缺少异常链路兜底
- 仍有较多模板和数据结构刚性依赖

## 7. 当前仍存在的问题

### 7.1 数据库集合依赖强

当前团队模式依赖以下数据库集合必须预先存在：

- `teamWalkRooms`
- `teamWalkMembers`
- `teamWalkContributions`
- `teamWalkActivities`

如果未先创建集合，创建房间会直接报：

- `database collection not exists`

这说明当前实现仍缺少：

- 环境初始化说明
- 数据库建表脚本或最少化初始化指引

### 7.2 WXML 对复杂表达式较敏感

在实现过程中，团队页面曾出现 WXML 编译错误，主要原因是模板里使用了过于复杂的表达式，例如：

- `slice()`
- `||`
- 三元表达式嵌套

当前虽然已经改为“在 JS 中先算好，再交给模板渲染”，但这说明当前页面实现仍然容易受小程序模板能力限制影响。

### 7.3 团队记录没有独立草稿层

当前团队记录页直接围绕页面态 `editorDraft` 进行编辑，没有像单人模式那样建立统一的本地持久化草稿层。

这意味着：

- 页面中途退出后不会像单人模式那样稳定恢复
- 弱网或误触返回时可能丢失未提交内容

### 7.4 团队任务模型仍较简化

当前只做了“共享任务”。

虽然这符合 P0 目标，但当前团队模式仍然缺：

- 分工任务
- 协作任务
- 任务级角色分配
- 任务完成阈值配置

从产品视角看，当前更像“多人共看同一套任务”，还没有完全进化到“强协作玩法”。

### 7.5 团队结果页纪念感仍偏弱

当前团队结果页已经有：

- 团队概览
- 任务回顾
- 动态记录

但仍缺少更强的纪念产物，例如：

- 团队纪念卡
- 团队贴纸
- 精选图集
- 团队高光总结页

因此当前结果页更偏“回顾详情”，还不像真正适合转发的团队纪念页。

### 7.6 权限与状态机仍偏轻量

当前已经限制：

- 只有房主可以开始
- 只有房主可以结束

但整体状态管理仍偏轻量，缺少更严谨的状态保护，例如：

- 已结束房间的重复结束保护日志
- 已开始房间的重复加入审计
- 成员离开后再进入的完整恢复策略

### 7.7 历史页是并列拼接，不是真正统一建模

当前历史页只是将：

- 单人记录
- 团队记录

做了前端列表合并。

虽然功能上已经可用，但从长期看还不是最优方案，因为：

- 排序逻辑目前在前端拼接
- 单人记录与团队记录模型差异仍较大
- 历史筛选与统计能力还不统一

## 8. 本轮过程中遇到的问题

### 8.1 `app.json` 权限配置不合法

在接入多人模式过程中，曾出现：

- `permission["scope.writePhotosAlbum"]`
- `permission["scope.userLocationBackground"]`

不被当前配置校验接受的问题。

后续已改为只保留合法且必要的：

- `scope.userLocation`

### 8.2 团队页面模板编译失败

在团队页面开发初期，`team-detail.wxml` 等页面曾因模板中使用复杂表达式而报错，导致：

- WXML 编译失败
- 渲染层出现连带错误

后续通过“把逻辑搬回 JS 计算层”解决。

### 8.3 房间创建初期即遇到集合不存在

在第一次联调创建房间时，直接遇到：

- `teamWalkRooms` 集合不存在

这暴露了当前多人模式对云开发数据库初始化的强依赖，也说明当前文档和环境准备环节需要补齐。

### 8.4 用户身份字段存在兼容压力

当前团队记录页在识别“当前用户是否已有任务提交”时，需要在这些字段之间兼容判断：

- `openid`
- `userId`
- `_id`

这意味着当前用户态模型还不够完全统一，未来如果接 Web 后端，会有更明显的一致性压力。

### 8.5 房间与详情页字段较多，模板脆弱性高

因为团队页面需要展示：

- 房间状态
- 成员数量
- 地点上下文
- 任务进度
- 按钮文案

前期如果直接在模板里拼逻辑，会使页面结构变得很脆弱。当前虽然已通过“预计算标签字段”缓解，但长期仍建议继续组件化。

## 9. 建议进一步改进的地方

### 9.1 产品层建议

建议下一阶段优先补齐：

1. 团队结果页纪念产物升级，例如团队卡片或团队贴纸。
2. 更明确地区分“我提交了什么”和“团队整体完成了什么”。
3. 补一个更清晰的房间空态、异常态和结束态说明。
4. 将动态流做得更有氛围感，而不只是简单事件列表。
5. 明确多人模式是否保留轨迹，还是只保留地点与任务。

### 9.2 前端工程层建议

建议继续做：

1. 为团队记录页增加本地草稿恢复能力。
2. 把团队任务卡、成员列表、动态流拆为独立组件。
3. 对房间状态、团队统计字段做统一的 view model 预处理。
4. 在多人页面中进一步避免模板层逻辑分支膨胀。
5. 增加更明确的 loading、空态、失败重试 UI。

### 9.3 后端与数据层建议

建议优先加强：

1. 给 `teamWalkRooms/teamWalkMembers/teamWalkContributions/teamWalkActivities` 补字段规范文档。
2. 统一用户标识字段，避免前端多字段兜底判断。
3. 给团队记录补操作日志和异常日志。
4. 为团队结束汇总增加更稳定的统计过程。
5. 评估是否需要单独的 `teamWalkSummaries` 汇总表。

## 10. 如何修改得更稳健并适用于生产环境

### 10.1 数据结构层

生产环境建议正式固化多人模式的数据模型。

建议至少明确：

- 房间对象 schema
- 成员对象 schema
- 贡献对象 schema
- 动态对象 schema
- 团队结果对象 schema

并对关键字段做前后端一致约束。

### 10.2 环境初始化层

建议补齐：

1. 云数据库集合初始化文档。
2. 新云函数部署清单。
3. 当前环境名与 `cloudEnvId` 对照说明。
4. 团队模式必需集合的权限建议。

否则很容易出现：

- 代码已更新，但环境未初始化
- 创建房间直接失败
- 开发与测试环境不一致

### 10.3 状态机与权限层

建议把房间状态机正式收敛为：

- `waiting`
- `active`
- `finished`
- `dissolved`

并对每个状态允许的操作做统一限制。

同时建议：

- 统一房主鉴权
- 统一成员鉴权
- 给重复操作提供更清晰的后端返回语义

### 10.4 媒体与弱网层

当前团队任务提交虽然能用，但生产环境建议补齐：

- 上传失败重试
- 大文件失败兜底
- 批量素材部分成功 / 部分失败提示
- 团队提交过程中断后的恢复机制

### 10.5 历史接入层

当前历史页只是前端列表拼接，生产环境建议考虑：

1. 统一“记录卡片”模型。
2. 统一详情跳转协议。
3. 明确团队记录排序与筛选策略。
4. 如果后续接 Web 后端，保证单人和团队记录接口在分页和排序语义上统一。

### 10.6 体验层

如果要接近正式上线体验，建议继续补：

- 房间邀请态优化
- 团队结束态纪念产物
- 更清晰的错误提示
- 更清晰的权限说明
- 更明确的多人定位边界

## 11. 如何测试

### 11.1 测试前准备

在测试多人模式前，需要先完成以下准备：

1. 部署新增的 10 个云函数。
2. 创建以下数据库集合：
   - `teamWalkRooms`
   - `teamWalkMembers`
   - `teamWalkContributions`
   - `teamWalkActivities`
3. 确认当前云环境与小程序 `cloudEnvId` 一致。
4. 准备两个以上可登录的小程序账号。

### 11.2 主链路测试

建议至少走通以下链路：

1. 首页切换到“同行”模式。
2. 生成主题后发起同行房间。
3. 分享邀请给另一个账号。
4. 第二个账号加入房间。
5. 房主开始同行漫步。
6. 两个账号分别对任务提交素材。
7. 房主结束同行漫步。
8. 进入团队结果页回看。
9. 在历史页确认看到“同行”记录。

### 11.3 异常分支测试

建议覆盖：

1. 未登录时发起同行。
2. 未登录时加入房间。
3. 房间开始后再尝试加入。
4. 房主解散房间。
5. 普通成员退出房间。
6. 非房主尝试结束同行。
7. 数据库集合未创建时的报错表现。

### 11.4 数据正确性测试

建议检查：

1. `teamWalkRooms` 中房间状态是否正确流转。
2. `teamWalkMembers` 中成员数量是否正确。
3. `teamWalkContributions` 中提交内容是否正确落库。
4. `teamWalkActivities` 中动态记录是否齐全。
5. 团队结束后 `teamStats` 与 `teamSummary` 是否正确生成。

### 11.5 回归测试

因为当前多人模式接入了探索页和历史页，所以必须补以下回归：

1. 单人模式仍能正常开始和保存。
2. 原有单人历史记录仍能正常展示。
3. 单人详情页仍能正常打开。
4. 多人记录不会误跳到单人详情。

### 11.6 真机测试建议

多人模式建议至少有一轮真机测试，重点验证：

- 分享卡打开房间
- 图片 / 视频 / 录音上传
- 录音按钮行为
- 多账号切换
- 团队页面渲染
- 弱网下的提交反馈

## 12. 当前结论

截至当前版本，可以给出如下判断：

- 同行模式入口：`已实现`
- 房间创建与加入：`已实现`
- 团队记录提交：`已实现`
- 团队结果页：`已实现`
- 历史页接入：`已实现`
- 社交体验深化：`未完成`
- 生产级稳健性：`未完成`

如果下一阶段目标是“尽快达到可上线试运行”，建议优先顺序如下：

1. 补齐数据库初始化与环境部署说明。
2. 固化团队模式数据结构与状态机。
3. 为团队记录页增加草稿恢复与异常兜底。
4. 补测试与埋点。
5. 再继续加强纪念产物、动态流氛围和多人体验。

一句话总结当前多人模式：

它已经从“设计书”进入了“可跑通的初版实现”，但当前更适合继续内部联调和小范围验证，还不适合直接当作成熟生产能力上线。
