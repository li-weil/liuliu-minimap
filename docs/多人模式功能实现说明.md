# 多人模式功能实现说明

## 1. 文档目的

本文用于沉淀当前“小程序多人模式”功能的真实实现情况。

本文不以 PRD 预期范围代替代码实际状态，而是基于当前仓库已经落地的代码，回答以下问题：

- 当前已经实现了哪些多人模式功能
- 这些功能采用了什么技术方案
- 当前实现到了什么程度
- 当前仍存在哪些问题与风险
- 后续如何继续修改，才能更稳健地适配生产环境
- 当前版本应该如何测试

本文中的“多人模式”，当前特指第一期 `同行模式`。

建议与以下文档配套阅读：

- `docs/同行模式MVP-PRD.md`：用于查看目标范围、页面设计和产品验收口径
- `docs/云函数登录系统说明.md`：用于查看登录态、头像上传和登录回跳的基础能力说明

## 1.1 本次对齐更新点

本次文档已额外对齐以下后续修正内容：

- 被邀请用户首次登录后的房间回跳机制
- 房间页成员与动态的静默自动同步
- 未加入用户仅可查看房间基础预览，不能读取成员、贡献与动态明细
- 成员头像先上传再入库的跨端显示策略
- 小程序重开后的“进行中同行房间”启动提醒

- 团队记录结束后，成员与房主统一回到 `足迹 - 纪念卡册`
- 同行记录删除入口与“仅已结束可删”的限制
- 历史卡片改为纯文字卡片，不再在列表中展示封面图
- 团队结果页的任务展示已对齐单人历史详情页的结构

- 房主结束同行前，会检查房间内成员是否还有带内容但未同步的任务草稿
- 团队卡片会把所有房间成员都纳入汇总；未同步或未打卡成员使用默认兜底内容

- 历史卡片类型兼容与恢复入口：`miniprogram/components/walk-card/walk-card.js`

- `cloudfunctions/createTeamRoom`

- `cloudfunctions/leaveTeamRoom`
- `cloudfunctions/startTeamWalk`
- `cloudfunctions/submitTeamContribution`

- `teamWalkRooms`

- `teamWalkContributions`
- `teamWalkActivities`

当前整体链路如下：

1. 探索页切换到“同行”模式。
2. 基于当前地点与主题创建同行房间。
3. 房主通过分享邀请队友。
4. 队友打开加入页加入房间。
5. 房主开始同行漫步。
6. 队员在团队记录页围绕同一任务集合提交素材。
7. 房主结束同行漫步。
8. 进入团队结果页。
9. 历史页展示“单人记录 + 同行记录”混合列表。
10. 若团队仍在进行中或待出发，可通过历史详情页继续进入。

## 3. 当前实现概览

### 3.1 当前完成度判断

如果按产品完成度理解，当前多人模式可以分为以下几档：

- `同行模式主链路`：已打通
- `房间与邀请`：已达到基础可用
- `团队任务与提交`：已达到基础可用
- `团队结果回看`：已达到基础可用
- `中断恢复与成员同步`：已达到基础可用
- `社交体验与生产化能力`：仍较初级

### 3.2 一句话总结当前状态

当前已经不是“只有设计稿”的阶段，而是已经具备一条可以跑通的最小闭环：

- 能发起房间
- 能邀请加入
- 能开始同行
- 能提交团队任务素材
- 能结束并回看团队结果
- 能在登录后自动回到原加入房间链路
- 能在房间页自动同步新成员和新动态
- 能在小程序重开后通过历史详情页重新进入进行中的同行记录
- 能在历史详情页继续进入待出发房间重新组队
- 能在已结束的同行详情页删除记录

但当前仍属于 `MVP P0 的增强版实现`，离可稳定上线的生产级实现还有明显距离。

## 4. 当前已经实现的功能

### 4.1 探索页同行入口

当前在探索页已经增加：

- `单人 / 同行` 模式切换
- 同行模式下复用现有地点选择和主题生成能力
- 主题卡底部按钮切换为“发起同行漫步”

当前逻辑为：

- 单人模式仍进入原有记录页
- 同行模式则调用 `createTeamRoom` 创建房间并跳转到房间页

### 4.2 同行房间创建

当前已实现房主发起同行房间能力。

房间创建时会保存：

- 房主信息
- 当前主题快照
- 地点信息
- 模式信息
- 任务集合
- 房间状态

创建成功后进入“同行房间页”。

### 4.3 房间加入与成员展示

当前已实现：

- 房主分享邀请卡
- 队友通过 `roomId` 打开加入页
- 已登录用户加入房间
- 房间页展示当前成员列表
- 房间动态展示“谁发起了房间”“谁加入了房间”

当前成员展示信息包括：

- 昵称
- 头像
- 房主标识

当前还补了一条权限边界：

- 未加入房间的用户打开加入页时，只能拿到房间基础摘要用于决定是否加入
- 未加入前不会返回完整成员列表、任务贡献内容和动态流明细

这样可以避免仅凭 `roomId` 就提前读取团队过程数据。

### 4.4 首次登录回房间链路

当前已经补齐了被邀请用户首次使用小程序时的登录回跳问题。

当前逻辑为：

- 被邀请用户打开加入页
- 若未登录，则跳转到个人页
- 在跳转前由 `app.js` 记录待恢复目标地址
- 登录成功后，个人页自动消费该目标并回到原加入页

这意味着：

- 用户首次登录后不会丢失原来的房间入口
- 不需要再次打开邀请链接才能继续加入

### 4.5 房间等待与出发

当前房间页支持：

- 查看主题
- 查看地点
- 查看任务列表
- 查看最近动态
- 房主开始同行漫步
- 成员或房主在开始前退出 / 解散房间

开始后，房间状态从 `waiting` 切换为 `active`。

### 4.6 房间自动同步

当前房间页已经补充自动刷新机制。

具体表现为：

- 房间页会定时轮询最新房间详情
- 成员加入后，其他成员停留在房间页也能自动看到变化
- 新成员会出现在成员列表中
- “某某加入了房间”会自动出现在动态流中

当前刷新策略已改为静默刷新：

- 首次进入时显示 loading
- 后续轮询不再整页切回 loading
- 仅当成员、动态、状态等关键数据变化时才更新页面

这样可以避免早期实现中的明显闪烁。

### 4.7 团队任务记录

当前团队记录页支持：

- 查看共享任务列表
- 切换当前任务
- 填写文字备注
- 上传图片
- 上传视频
- 录音
- 将当前任务内容同步到团队

当前已进一步收口为：

- 不再单独提供“我已完成该任务”的勾选框
- `同步给团队` 本身就是这条任务在同行模式下的打卡入口
- 只要用户改动了文字、图片、视频、录音，这条任务就会重新变成“待同步”状态
- 点击 `同步给团队` 时，如果当前任务已有内容，会先尝试生成该任务的 `companionNote`
- 如果当前任务没有内容，也允许保持未同步，后续团队卡片会走默认兜底

同步后会更新：

- 当前任务参与人数
- 当前任务团队完成状态
- 团队动态流
- 团队统计信息
- 当前成员在服务端的未同步任务状态

最近这一块还补了一项关键修复：

- 团队记录页虽然仍会轮询房间状态
- 但轮询不再直接覆盖当前用户尚未提交的编辑内容
- 用户在某个任务下输入到一半的文字、图片、视频、录音，会先写入任务级本地草稿缓存
- 切换任务后再切回来，本地草稿仍会保留
- 只有真正“同步给团队”成功后，才把这份草稿视为已提交状态

### 4.8 团队动态流

当前已实现轻量动态流，不做自由聊天。

已实现的动态类型包括：

- 发起房间
- 成员加入
- 成员退出
- 房主开始同行
- 某成员更新任务记录
- 某成员完成任务
- 房主结束同行

### 4.9 团队结束与结果页

当前已实现：

- 房主结束同行漫步
- 汇总团队统计信息
- 生成团队摘要文案
- 进入团队结果页

当前团队结果页支持展示：

- 主题信息
- 地点信息
- 团队统计
- 成员列表
- 任务回顾
- 每个任务下的成员贡献
- 团队动态

最近这一页已经进一步对齐单人历史详情页：

- 任务区改成与单人详情页相同的纵向展开结构
- 展开后按成员分段展示文字、图片、视频、录音
- 每个任务都有独立的团队打卡卡片区域
- 卡片支持弹窗预览和保存到相册

团队结果页当前与单人历史详情页保持相同的多媒体渲染原则：

- 图片、视频、录音在数据库中优先保存稳定地址
- 图片、视频直接使用数据库里保存的 fileID / 原始地址渲染
- 录音在团队结果页会先下载成本地临时文件后再播放
- 团队结果页不再单独对图片、视频做临时链接转换

这样单人和多人详情页的多媒体链路已经基本统一，只保留了录音这一个为兼容性做的本地缓存特例。

团队卡片链路当前也做了相同方向的提速：

- 每位成员点击 `同步给团队` 时，如果这条任务已有内容，会在同步前尝试生成 `companionNote`
- 如果房间里已经有 `missionCardMap[mission].cardImagePath`，详情页直接展示
- 只有第一次没有卡片时，用户主动点击“生成卡片”才会本地出图
- 新生成的团队卡片会上传并持久化到房间的 `missionCardMap`
- `checkin-card` 组件只在手动触发生成时临时挂载，生成后立即卸载
- 团队结果页平时只展示卡片图片，不再常驻隐藏 canvas 组件
- 页面不再提供“重新生成卡片”入口
- 团队卡片不再因为个别成员缺少 `companionNote` 而卡住整张卡片
- 未同步或未打卡的成员会直接使用默认兜底内容进入团队卡片汇总

这样团队结果页从“每次打开都现做卡”改成了“首次生成一次、后续长期复用”的方式，体感会明显更稳。

如果当前任务已有成员贡献，但团队卡片还没出图，页面会显示：

- `66 正在整理卡片...`

用于提示当前仍在生成中，而不是页面异常。

### 4.9.1 团队打卡卡片生成方式

当前团队卡片不再只取某一个成员的内容，而是基于当前任务下的全部成员内容进行汇总。

当前生成顺序为：

- 先按房间全部成员构建固定顺序的条目列表
- 已同步且已打卡的成员使用真实文字、图片、录音等内容
- 尚未同步、未打卡或当前任务没有内容的成员，使用默认兜底条目
- 把每个成员自己的记录分别送去生成对应的小猫记录
- 最终卡片按“成员自己的记录 -> 对应小猫的记录”的顺序，从上到下纵向排列
- 若该成员上传了图片，则会跟在这位成员对应的记录段落下方

也就是说，当前团队卡片已经是“多人纵向堆叠卡片”，而不是单人卡片的简单复制。

### 4.9.2 房主结束同行前的同步约束

当前房主结束同行前，系统会额外检查 `teamWalkMembers.pendingMissionKeys`：

- 成员在本地编辑某个任务且该任务已有内容时，会把这个任务上报为“待同步”
- 成员点击 `同步给团队` 成功后，对应任务会从待同步列表里清掉
- 只要房间里任何一个已加入成员仍有待同步任务，房主就不能结束同行
- 前端会直接提示“还有成员未同步”，并尽量带上成员昵称

这样可以避免成员明明还有本地内容没同步，房主却提前把整场同行结束掉。

### 4.10 历史页接入与卡片样式

当前历史页已不再只展示单人记录，而是合并了：

- `listMyWalks`
- `listMyTeamWalks`

历史卡片中已经区分：

- `单人`
- `同行`

点击同行卡片时，会跳到团队结果页，而不是单人详情页。

当前纪念卡册列表已调整为纯文字卡片样式：

- 每条记录都有独立卡片边界
- 卡片间通过留白、阴影和圆角区分
- 不再在列表中展示封面图或打卡卡片图
- 仅保留标题、状态、地点、时长、人数 / 里程、摘要等文字信息

### 4.11 进行中与待出发房间恢复入口

当前已补充“进行中的同行记录恢复入口”。

具体规则为：

- 历史列表先进入详情页
- 在同行详情页中，如果状态为 `active`，显示 `重新进入这场同行`
- 在同行详情页中，如果状态为 `waiting`，显示 `进入房间继续组队`
- `active` 点击后进入团队记录页
- `waiting` 点击后进入房间页

当前恢复路径为：

- `足迹 -> 纪念卡册 -> 同行详情页 -> 重新进入这场同行 / 进入房间继续组队`

### 4.12 小程序启动提醒

当前在小程序启动后增加了进行中任务的统一检测提醒。

当前逻辑为：

- 小程序启动后等待用户态初始化
- 若用户仍有进行中的同行记录或进行中的单人记录
- 且当前不在团队页面或单人记录页内
- 就弹出一个明显提示

提示内容会明确告知用户：

- 去“足迹 - 纪念卡册”
- 打开最新一条显示“进行中”的记录详情页
- 再点击“继续记录”或“重新进入这场同行”

### 4.13 成员头像跨端显示

当前已经修复了“多人房间里看不到其他同行者头像”的问题。

当前头像链路为：

- 用户在个人页选择头像
- 在“创建资料并登录”或“保存资料”时，前端先上传头像
- 上传成功后，把稳定地址写入 `users.avatarUrl`
- 成员加入房间时，再把这个稳定地址写入 `teamWalkMembers.avatarUrl`

这样多人房间中的成员头像已经不再依赖本地临时路径。

### 4.14 团队结束后的统一退场

当前在团队记录页中，已经补上“结束后统一回到纪念卡册”的行为。

具体表现为：

- 房主点击结束同行漫步后，会直接回到 `足迹 - 纪念卡册`
- 成员留在团队记录页时，会通过更高频的状态检测感知房间已结束
- 成员感知到房间结束后，也会自动回到 `足迹 - 纪念卡册`

### 4.15 同行记录删除

当前已为同行历史详情页增加删除入口，删除规则已改为“成员个人视角删除”，且仍受状态限制。

当前规则为：

- `finished` 状态的同行记录允许删除
- `waiting` 或 `active` 状态的同行记录不允许删除
- 若用户在未结束状态下尝试删除，会得到明确提示
- 删除只会把当前成员自己的这条同行记录标记为“个人已删除”
- 其他成员如果没有主动删除，仍然可以继续在自己的纪念卡册和团队详情里看见这条记录
- 删除后仅当前用户的成就会回滚重算，其他成员不受影响
- 后端云函数也会再次校验，避免前端绕过后误删或误影响他人视角

## 5. 技术实现说明

### 5.1 前端页面结构

当前多人模式前端采用“独立页面 + 复用现有入口”的实现方式。

设计上没有强行把多人逻辑塞进原单人 `record` 页，而是新增了以下页面：

- `team-join`
- `team-room`
- `team-record`
- `team-detail`

这样做的好处是：

- 不会过度污染原有单人记录页状态
- 多人模式可以独立演进
- 页面职责相对清晰

### 5.2 服务层封装

多人模式单独抽出 `miniprogram/services/team.js` 作为服务层入口。

当前已封装的服务包括：

- 创建房间
- 拉取房间详情
- 加入房间
- 退出房间
- 开始同行
- 提交团队贡献
- 拉取动态流
- 结束同行
- 拉取团队结果
- 拉取我参与的团队记录
- 删除已结束的同行记录

同时，`app.js` 已开始同时调用 `listMyTeamWalks` 和 `listMyWalks` 做进行中任务提醒检测。

### 5.3 接口归一化

`miniprogram/services/api.js` 当前新增了团队数据归一化逻辑，主要包括：

- `normalizeTeamMember`
- `normalizeTeamContribution`
- `normalizeTeamActivity`
- `normalizeTeamWalkRecord`

归一化后的团队记录结构中，已经统一包含：

- `themeTitle`
- `locationName/locationContext/locationAddress`
- `status`
- `walkMode`
- `themeSnapshot`
- `members`
- `contributions`
- `activities`
- `teamStats`
- `routeStats`

这样历史页和团队结果页可以用统一的前端消费结构。

### 5.4 云函数实现结构

当前团队模式采用“一个动作一个云函数”的轻量拆分方式。

#### 1. 房间主表

`teamWalkRooms` 当前保存：

- 房主
- 房间状态
- 主题快照
- 地点信息
- 团队统计
- 团队摘要
- 封面图

#### 2. 成员表

`teamWalkMembers` 当前保存：

- roomId
- userId
- 昵称
- 头像
- 身份角色
- 加入状态

#### 3. 贡献表

`teamWalkContributions` 当前按“成员 + 任务”维度保存任务提交内容，包括：

- noteText
- photoList
- videoList
- audioList
- completed

#### 4. 动态表

`teamWalkActivities` 当前保存动态流事件，包括：

- type
- content
- payload
- userId
- createdAt

### 5.5 媒体上传策略

团队记录页目前复用了单人模式现有的上传能力：

- 图片、视频、录音先走前端上传
- 由 `requestUpload` 负责上传到云存储
- 上传后的 `cloud://` 或 URL 再写回团队贡献表

当前团队模式并没有单独实现新的上传协议，而是沿用现有单人模式上传链路。

### 5.6 头像上传策略

当前多人模式成员头像显示依赖登录系统中的头像上传策略。

当前实现方式为：

- 个人页中的头像不再直接保存本地临时路径
- 在注册或保存资料时，先通过 `requestUpload` 上传头像
- 上传成功后，把稳定的 `cloud://` 或媒体 URL 写入 `users.avatarUrl`
- 成员加入房间时，再把这个稳定地址写入 `teamWalkMembers.avatarUrl`

这样可以避免过去那种：

- 自己设备上头像能看见
- 其他同行者设备上头像看不见

的情况。

### 5.7 登录与权限依赖

当前多人模式直接复用已有用户系统。

即：

- 发起房间前需要已登录
- 加入房间前需要已登录
- 房间与提交权限通过当前用户 `OPENID` / userId 识别

当前房主权限的判断主要依赖：

- `room.ownerUserId`
- `member.role === 'owner'`

并且当前已经补了一个重要链路：

- 被邀请用户如果第一次进入小程序、尚未登录
- 在加入房间时被引导去个人页
- 登录成功后会自动恢复到原房间加入路径

### 5.8 页面同步策略

当前房间页已经增加前端轮询同步能力。

当前策略为：

- `onShow` 时开启轮询
- `onHide/onUnload` 时清理轮询
- 每隔固定时间拉取一次房间详情
- 仅在关键数据变化时才触发页面更新

这比早期“每轮刷新整页 loading”的方式更稳定，但仍然是基于轮询的同步，不是真正的实时推送。

团队记录页当前也已经有一条明确的结束同步规则：

- 如果某位成员或房主所在的另一端先把同行漫步结束
- 仍停留在团队记录页的其他端会在更高频状态检测中感知 `room.status === finished`
- 然后自动退出团队记录页并统一回到 `足迹 - 纪念卡册`

因此当前团队模式在“结束后统一退场”这件事上，已经与单人记录页保持一致心智。

### 5.9 中断恢复策略

当前尚未给团队记录页增加完整的本地草稿恢复，但已经补了两层入口恢复：

1. 登录中断恢复
   - 用于首次登录后回到原加入页

2. 房间进行中恢复
   - 用于退出小程序后从历史页重新进入团队记录页

这说明当前已经解决了“回不去房间”的问题，但还没有完全解决“未提交编辑内容不丢失”的问题。

## 6. 当前实现到了什么程度

### 6.1 从功能闭环角度判断

当前已经达到：

- `产品链路可跑通`

也就是：

- 用户可以完成一次最基本的“多人发起 -> 多人参与 -> 团队结束 -> 团队回看”
- 用户中途去登录后可以自动回到原加入链路
- 用户退出小程序后也可以通过历史页重新进入进行中的团队记录

### 6.2 从产品体验角度判断

当前体验属于：

- `基础版多人模式`

已经具备多人协作雏形，并补了房间自动同步、头像显示和恢复入口，但社交感、氛围感和纪念产物丰富度都还偏弱。

### 6.3 从工程稳定性角度判断

当前工程状态属于：

- `开发中可联调`

但还不能认为已经达到稳定生产交付状态。

主要原因包括：

- 缺少系统化测试
- 缺少数据库初始化规范
- 缺少异常链路兜底
- 仍有较多模板和数据结构刚性依赖
- 团队记录页虽然已经有页面级草稿保护，但仍未建立真正的本地持久化草稿恢复能力

## 7. 当前仍存在的问题

### 7.1 数据库集合依赖强

当前团队模式依赖以下数据库集合必须预先存在：

- `teamWalkRooms`
- `teamWalkMembers`
- `teamWalkContributions`
- `teamWalkActivities`

如果未先创建集合，创建房间会直接报：

- `database collection not exists`

这说明当前实现仍缺少：

- 环境初始化说明
- 数据库建表脚本或最少化初始化指引

### 7.2 WXML 对复杂表达式较敏感

在实现过程中，团队页面曾出现 WXML 编译错误，主要原因是模板里使用了过于复杂的表达式，例如：

- `slice()`
- `||`
- 三元表达式嵌套

当前虽然已经改为“在 JS 中先算好，再交给模板渲染”，但这说明当前页面实现仍然容易受小程序模板能力限制影响。

### 7.3 团队记录仍缺少真正的持久化草稿层

当前团队记录页已经补了“任务级页面草稿缓存”，可以解决轮询覆盖编辑区的问题。

也就是说，当前已经支持：

- 轮询时保留未提交文字和素材
- 切换任务后再切回来仍能保留当前页面内的草稿

但它仍然不是像单人模式那样的本地持久化草稿层，因为目前这份草稿还主要存在页面内存里。

这意味着：

- 页面停留期间体验已明显改善
- 但如果用户中途退出小程序、被系统回收页面，未提交内容仍可能丢失

所以这部分问题已经从“编辑中会被实时刷掉”降级为“退出页面后还不能稳定恢复”。

### 7.4 团队任务模型仍较简化

当前只做了“共享任务”。

虽然这符合 P0 目标，但当前团队模式仍然缺：

- 分工任务
- 协作任务
- 任务级角色分配
- 任务完成阈值配置

从产品视角看，当前更像“多人共看同一套任务”，还没有完全进化到“强协作玩法”。

### 7.5 团队结果页纪念感已有提升，但仍未完全产品化

当前团队结果页已经有：

- 团队概览
- 任务回顾
- 动态记录
- 任务级团队打卡卡片

但仍缺少更强的纪念产物，例如：

- 团队纪念卡
- 团队贴纸
- 精选图集
- 团队高光总结页

因此当前结果页虽然已经比最初版本更有纪念感，但整体仍偏“回顾详情”，还不像真正适合转发的团队纪念页。

### 7.6 权限与状态机仍偏轻量

当前已经限制：

- 只有房主可以开始
- 只有房主可以结束

但整体状态管理仍偏轻量，缺少更严谨的状态保护，例如：

- 已结束房间的重复结束保护日志
- 已开始房间的重复加入审计
- 成员离开后再进入的完整恢复策略

### 7.7 历史页是并列拼接，不是真正统一建模

当前历史页只是将：

- 单人记录
- 团队记录

做了前端列表合并。

虽然功能上已经可用，但从长期看还不是最优方案，因为：

- 排序逻辑目前在前端拼接
- 单人记录与团队记录模型差异仍较大
- 历史筛选与统计能力还不统一

### 7.8 房间同步当前仍依赖轮询

当前房间页新增了自动刷新，但实现方式仍是：

- 前端定时轮询 `getTeamRoomDetail`

这虽然已经足够支撑当前 MVP，但仍有以下局限：

- 实时性有限
- 页面仍会有一定频率的数据刷新
- 房间页和团队记录页还没有做到真正一致的同步策略

### 7.9 进行中房间恢复入口仍然偏绕

当前已补入口，但实际恢复路径仍然是：

- 启动弹窗提醒
- 用户手动去足迹页
- 打开记录详情页
- 再点击继续入口

这已经比完全无入口好很多，但从产品体验看还不够短链路。

## 8. 本轮过程中遇到的问题

### 8.1 `app.json` 权限配置不合法

在接入多人模式过程中，曾出现：

- `permission["scope.writePhotosAlbum"]`
- `permission["scope.userLocationBackground"]`

不被当前配置校验接受的问题。

后续已改为只保留合法且必要的：

- `scope.userLocation`

### 8.2 团队页面模板编译失败

在团队页面开发初期，`team-detail.wxml` 等页面曾因模板中使用复杂表达式而报错，导致：

- WXML 编译失败
- 渲染层出现连带错误

后续通过“把逻辑搬回 JS 计算层”解决。

### 8.3 房间创建初期即遇到集合不存在

在第一次联调创建房间时，直接遇到：

- `teamWalkRooms` 集合不存在

这暴露了当前多人模式对云开发数据库初始化的强依赖，也说明当前文档和环境准备环节需要补齐。

### 8.4 用户身份字段存在兼容压力

当前团队记录页在识别“当前用户是否已有任务提交”时，需要在这些字段之间兼容判断：

- `openid`
- `userId`
- `_id`

这意味着当前用户态模型还不够完全统一，未来如果接 Web 后端，会有更明显的一致性压力。

### 8.5 房间与详情页字段较多，模板脆弱性高

因为团队页面需要展示：

- 房间状态
- 成员数量
- 地点上下文
- 任务进度
- 按钮文案

前期如果直接在模板里拼逻辑，会使页面结构变得很脆弱。当前虽然已通过“预计算标签字段”缓解，但长期仍建议继续组件化。

### 8.6 被邀请用户首次登录后丢失原房间入口

在一次联调中实际出现过：

- 被邀请用户首次打开小程序
- 由于未登录被引导到个人页
- 登录完成后无法自动回到原来的房间加入页

后续已通过：

- `app.js` 增加待恢复跳转目标
- `team-join` 写入待跳转目标
- `profile` 登录成功后消费待跳转目标

完成修复。

### 8.7 房间页虽然能同步，但会明显闪烁

在早期房间自动刷新实现中，曾出现：

- 成员加入后能同步看到变化
- 但房间页每轮刷新都会切回 loading 或整块重渲染

后续已改为：

- 首次进入时显示 loading
- 轮询刷新时走静默拉取
- 仅在房间核心数据发生变化时才 `setData`

从而缓解了页面闪烁问题。

### 8.8 头像只能自己看到，其他同行者看不到

在多人联调中还出现过：

- 成员自己能看到自己的头像
- 其他同行者房间页里头像不显示

问题根因是：

- 个人页选择头像后，早期实现把临时路径直接存进用户资料

后续已改为：

- 先上传头像
- 再把稳定地址写入 `users.avatarUrl`

从而解决跨端可见性问题。

### 8.9 团队记录进行中退出后无法找回入口

在团队记录页开始后，如果用户中途退出小程序，再次进入时曾经出现：

- 无法直接找到原来的进行中房间入口

后续已通过两层补救能力缓解：

- 历史详情页中的继续入口
- 小程序启动时的进行中房间提醒弹窗

### 8.10 已结束同行记录删除时报“不存在”

在联调过程中，曾出现：

- 单人记录可删除
- 同行记录在详情页点删除时提示“记录不存在”

这暴露出同行详情链路和删除链路需要分别部署、分别校验的事实。后续已补：

- 同行独立删除云函数 `deleteTeamWalk`
- 前端详情页删除入口
- 后端对 `finished` 状态的校验
- 更明确的错误原因映射

### 8.11 团队记录页轮询曾覆盖编辑中的内容

团队记录页为了同步房间状态，会定时轮询 `getTeamRoomDetail`。

早期实现中，轮询结果会直接回填当前任务的 `editorDraft`，导致：

- 输入到一半的文字被刷掉
- 刚选的图片、视频、录音被重置
- 还没点击“同步给团队”，编辑区内容就被服务端旧数据覆盖

后续已通过以下方式修复：

- 按任务建立页面内草稿缓存
- 为草稿增加 dirty 标记
- 如果当前任务存在未提交修改，轮询时优先保留本地草稿
- 真正同步成功后，再把这份草稿标记为已提交

这个问题目前已经从“编辑中会实时丢内容”降级为“退出页面后还缺少持久化恢复”。

### 8.12 团队卡片生成耗时比单人更长

团队打卡卡片需要按成员分别整理文字和图片，再逐个生成对应的小猫记录，因此生成耗时会明显高于单人卡片。

为避免用户误以为页面卡住，团队结果页当前已增加明确提示：

- `66 正在整理卡片...`

## 9. 建议进一步改进的地方

### 9.1 产品层建议

建议下一阶段优先补齐：

1. 团队结果页纪念产物升级，例如团队卡片或团队贴纸。
2. 更明确地区分“我提交了什么”和“团队整体完成了什么”。
3. 补一个更清晰的房间空态、异常态和结束态说明。
4. 将动态流做得更有氛围感，而不只是简单事件列表。
5. 明确多人模式是否保留轨迹，还是只保留地点与任务。
6. 把“进行中房间恢复”做成更短链路入口，而不是只依赖历史页。

### 9.2 前端工程层建议

建议继续做：

1. 为团队记录页增加本地草稿恢复能力。
2. 把团队任务卡、成员列表、动态流拆为独立组件。
3. 对房间状态、团队统计字段做统一的 view model 预处理。
4. 在多人页面中进一步避免模板层逻辑分支膨胀。
5. 把房间页和团队记录页的同步策略进一步统一。
6. 增加更明确的 loading、空态、失败重试 UI。
7. 给团队打卡卡片增加生成缓存、失败态和超时提示，避免多人任务重复触发 AI 生成。

### 9.3 后端与数据层建议

建议优先加强：

1. 给 `teamWalkRooms/teamWalkMembers/teamWalkContributions/teamWalkActivities` 补字段规范文档。
2. 统一用户标识字段，避免前端多字段兜底判断。
3. 给团队记录补操作日志和异常日志。
4. 为团队结束汇总增加更稳定的统计过程。
5. 评估是否需要单独的 `teamWalkSummaries` 汇总表。

## 10. 如何修改得更稳健并适用于生产环境

### 10.1 数据结构层

生产环境建议正式固化多人模式的数据模型。

建议至少明确：

- 房间对象 schema
- 成员对象 schema
- 贡献对象 schema
- 动态对象 schema
- 团队结果对象 schema

并对关键字段做前后端一致约束。

### 10.2 环境初始化层

建议补齐：

1. 云数据库集合初始化文档。
2. 新云函数部署清单。
3. 当前环境名与 `cloudEnvId` 对照说明。
4. 团队模式必需集合的权限建议。

否则很容易出现：

- 代码已更新，但环境未初始化
- 创建房间直接失败
- 删除同行记录时报错，因为 `deleteTeamWalk` 未部署
- 开发与测试环境不一致

### 10.3 状态机与权限层

建议把房间状态机正式收敛为：

- `waiting`
- `active`
- `finished`
- `dissolved`

并对每个状态允许的操作做统一限制。

同时建议：

- 统一房主鉴权
- 统一成员鉴权
- 给重复操作提供更清晰的后端返回语义

### 10.4 媒体与弱网层

当前团队任务提交虽然能用，但生产环境建议补齐：

- 上传失败重试
- 大文件失败兜底
- 批量素材部分成功 / 部分失败提示
- 团队提交过程中断后的恢复机制

同时也建议把头像上传链路纳入同一套上传失败治理中，避免出现：

- 头像保存成功提示已展示
- 实际上传失败导致成员头像为空

### 10.5 历史接入层

当前历史页只是前端列表拼接，生产环境建议考虑：

1. 统一“记录卡片”模型。
2. 统一详情跳转协议。
3. 明确团队记录排序与筛选策略。
4. 如果后续接 Web 后端，保证单人和团队记录接口在分页和排序语义上统一。

### 10.6 体验层

如果要接近正式上线体验，建议继续补：

- 房间邀请态优化
- 团队结束态纪念产物
- 更清晰的错误提示
- 更清晰的权限说明
- 更明确的多人定位边界
- 更直接的进行中房间快捷恢复入口
- 从房间页到团队记录页更清晰的“继续进行中任务”提示

## 11. 如何测试

### 11.1 测试前准备

在测试多人模式前，需要先完成以下准备：

1. 部署新增的 10 个云函数。
1. 部署新增的团队模式云函数，当前至少包括：
   - `createTeamRoom`
   - `getTeamRoomDetail`
   - `joinTeamRoom`
   - `leaveTeamRoom`
   - `startTeamWalk`
   - `submitTeamContribution`
   - `listTeamActivities`
   - `finishTeamWalk`
   - `getTeamWalkDetail`
   - `listMyTeamWalks`
   - `deleteTeamWalk`
2. 创建以下数据库集合：
   - `teamWalkRooms`
   - `teamWalkMembers`
   - `teamWalkContributions`
   - `teamWalkActivities`
3. 确认当前云环境与小程序 `cloudEnvId` 一致。
4. 准备两个以上可登录的小程序账号。
5. 建议至少一轮双设备或双账号真机联调。

### 11.2 主链路测试

建议至少走通以下链路：

1. 首页切换到“同行”模式。
2. 生成主题后发起同行房间。
3. 分享邀请给另一个账号。
4. 第二个账号加入房间。
5. 房主开始同行漫步。
6. 两个账号分别对任务提交素材。
7. 房主结束同行漫步。
8. 进入团队结果页回看。
9. 在历史页确认看到“同行”记录。

建议在这条链路里额外确认：

- 房间页是否自动看到新成员和新动态
- 成员头像是否双向可见
- 团队结果页任务区样式是否已与单人详情页基本一致
- 团队打卡卡片是否按成员逐段从上到下排列

### 11.3 异常分支测试

建议覆盖：

1. 未登录时发起同行。
2. 未登录时加入房间。
3. 房间开始后再尝试加入。
4. 房主解散房间。
5. 普通成员退出房间。
6. 非房主尝试结束同行。
7. 数据库集合未创建时的报错表现。
8. 登录完成后是否自动回到原来的加入房间页。
9. 小程序中途退出后，是否还能通过历史详情页继续进入团队记录页。
10. 待出发的同行记录是否能从详情页进入房间继续组队。
11. 已结束的同行记录是否可以成功删除。
12. 进行中或待出发的同行记录删除时，是否会被正确拦截。
13. 团队结果页在卡片未生成时，是否显示 `66 正在整理卡片...`。
14. 团队记录页输入到一半的文字、图片、视频、录音，是否会被轮询刷新冲掉。

### 11.4 数据正确性测试

建议检查：

1. `teamWalkRooms` 中房间状态是否正确流转。
2. `teamWalkMembers` 中成员数量是否正确。
3. `teamWalkContributions` 中提交内容是否正确落库。
4. `teamWalkActivities` 中动态记录是否齐全。
5. 团队结束后 `teamStats` 与 `teamSummary` 是否正确生成。
6. `users.avatarUrl` 是否为稳定可跨端访问的地址。

### 11.5 回归测试

因为当前多人模式接入了探索页和历史页，所以必须补以下回归：

1. 单人模式仍能正常开始和保存。
2. 原有单人历史记录仍能正常展示。
3. 单人详情页仍能正常打开。
4. 多人记录不会误跳到单人详情。
5. 个人页登录、头像保存与资料修改不受影响。
6. 团队记录页切换任务后再切回，本地未提交草稿仍保留。

### 11.6 真机测试建议

多人模式建议至少有一轮真机测试，重点验证：

- 分享卡打开房间
- 首次登录后是否自动回原房间页
- 图片 / 视频 / 录音上传
- 头像跨端显示
- 录音按钮行为
- 多账号切换
- 团队页面渲染
- 进行中房间恢复入口
- 弱网下的提交反馈

## 12. 当前结论

截至当前版本，可以给出如下判断：

- 同行模式入口：`已实现`
- 房间创建与加入：`已实现`
- 团队记录提交：`已实现`
- 团队结果页：`已实现`
- 历史页接入：`已实现`
- 登录回跳：`已实现`
- 房间自动同步：`已实现`
- 头像跨端显示：`已实现`
- 进行中房间恢复入口：`已实现`
- 社交体验深化：`未完成`
- 生产级稳健性：`未完成`

如果下一阶段目标是“尽快达到可上线试运行”，建议优先顺序如下：

1. 补齐数据库初始化与环境部署说明。
2. 固化团队模式数据结构与状态机。
3. 为团队记录页增加草稿恢复与异常兜底。
4. 把“进行中房间恢复”做成更短链路入口。
5. 补测试与埋点。
6. 再继续加强纪念产物、动态流氛围和多人体验。

一句话总结当前多人模式：

它已经从“设计书”进入了“可跑通的初版实现”，并补齐了登录回跳、房间同步、头像显示和恢复入口等关键体验问题，但当前更适合继续内部联调和小范围验证，还不适合直接当作成熟生产能力上线。
