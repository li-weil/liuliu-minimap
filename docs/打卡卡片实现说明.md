# 打卡卡片实现说明

## 背景

项目里的“打卡卡片”已经不再是早期的静态视觉 demo。它现在是任务记录回看的一部分。

它的目标是把一次漫步中某个任务下的内容：

- 文字记录
- 图片记录
- AI 陪伴文案

整理成一张可查看、可保存、可随记录一起存档的纪念卡片。

目前它已经可以内部试用，但还没达到长期稳定上线的标准。

## 已实现功能

### 1. 记录页负责任务打卡，详情页负责首次生成卡片

记录页的任务面板中，每个任务支持：

- 上传图片
- 上传视频
- 上传录音
- 输入文字
- 点击 `完成这一站`

记录页不再提供 `生成卡片`、卡片弹窗或保存到相册入口。现在的流程是：

1. 用户在记录页为任务添加素材和文字。
2. 用户点击 `完成这一站`，任务立即进入已打卡状态。
3. 前端后台调用 `generateCompanionNote` 补齐 66 陪伴文案。
4. 保存漫步后，用户在足迹详情页按需首次生成打卡卡片。
5. 卡片生成后上传并写回 `missionAssetMap[mission].cardImagePath`，后续详情页直接复用图片资源。

### 2. 新增“总结与补充”特殊任务位

任务列表除了正常任务外，还追加了一个特殊项：

- `总结与补充`

它本质上不是原始主题任务之一，而是为了给整次漫步留一个补充记录入口。

它有自己的：

- `mission key`
- `noteText`
- `photoList`
- `videoList`
- `audioList`
- `companionNote`
- `cardImagePath`

因此，打卡卡片既可以服务标准任务，也可以服务整段漫步的总结记录。

### 3. 卡片改成“我 + 66”的双叙事结构

现在的打卡卡片不再只是“标题 + 文案 + 图片”的信息排版。

现在它的核心结构变成了：

- 顶部主题头图区域
- 用户自己的记录气泡
- AI 陪伴角色 `66` 的回复气泡
- 底部图片区域
- 日期、地点、装饰元素

卡片现在更像一张对话式纪念卡：既有用户记录，也有 66 的回应。

### 4. 已接入 AI 陪伴文案生成

记录页打卡后，会异步调用：

- `generateCompanionNote`

它会为当前任务生成一段 `小猫66 的记录`。详情页生成卡片时优先读取已保存的 `companionNote`，不再现场重新请求 AI 文案。

逻辑特点：

- 打卡后先标记完成，不等待 AI 文案返回
- 如果打卡内容后续被修改，会清空旧 `companionNote` 和 `cardImagePath`，并要求重新打卡
- 如果 AI 文案仍在生成中，详情页会提示 `66 正在记录卡片…` 并自动刷新
- 如果文案生成失败，任务仍可保存，详情页制卡会使用现有文字和默认内容

### 6. 卡片只消费文字与图片，不嵌入视频和录音

卡片只使用：

- `noteText`
- `companionNote`
- `photoList`

不会把下列素材绘制进卡片：

- `videoList`
- `audioList`

视频和录音只保留在任务素材里，不进入卡片。

### 7. 多图支持仍然存在，但当前限制为最多两张

相比上一版“把全部图片一路排下去”的做法，现在改为：

- 读取任务图片列表
- 做 Canvas 兼容处理
- 最终最多取前两张进入卡片

这样可以控制卡片高度，也能保持画面节奏。

这意味着：

- 一张图任务可以完整展示
- 两张图任务可以双图展示
- 三张及以上图片当前不会全部画进卡片

所以更准确的说法是：

- **支持多图输入**
- **当前卡片模板默认展示前两张图**

### 8. 详情页生成后使用弹窗查看，并支持滚动

在历史详情页点击 `生成` 后，会：

1. 打开弹窗
2. 显示“正在生成打卡卡片...”
3. Canvas 生成完成后在弹窗内展示图片
4. 弹窗内容支持滚动查看
5. 用户可直接保存到本地

这套流程已经比较完整，也适合当前小程序页面结构。记录页不再保留卡片弹窗。

### 10. 生成结果会回写到任务资产中

这是当前版本非常重要的变化。

卡片生成完成后，不再只是弹窗里看一下，而是会把结果临时路径写回当前任务资产：

- `cardImagePath`

这意味着：

- 足迹详情页当前任务下会出现一张已生成卡片预览
- 点击预览可再次打开弹窗查看
- 生成完成后立即上传并写回当前历史记录
- 以后再进入详情页时直接读取已保存的 `cardImagePath`

从这个角度看，打卡卡片已经开始具备“记录的一部分”属性，而不是完全临时态效果。

### 11. 保存记录时保存素材与陪伴文案，卡片在详情页生成后回写

当前保存漫步记录时，任务资产上传包含以下字段：

- `noteText`
- `companionNote`
- `photoList`
- `videoList`
- `audioList`
- `cardImagePath`

记录页保存时通常只会带上素材和 `companionNote`。`cardImagePath` 主要由历史详情页首次制卡后上传并回写，因此当前卡片持久化属于“详情页按需生成 + 持久化复用”的链路。

### 12. 已有基础图片格式兼容层

当前图片链路已经补了基础兼容处理，覆盖：

- 选图

- 上传
- Canvas 绘制

主要做法：

- `chooseImage` 优先走压缩图
- 识别 `jpg/jpeg/png/webp/bmp/gif/heic/heif`
- 对不稳定格式尽量转为更兼容的临时 JPEG
- Canvas 绘制失败时再尝试标准化一次
- 上传时推断扩展名，减少无后缀路径失败

如果按项目阶段判断，当前版本已经比最初 demo 更进一步。

更准确地说：

- **阶段：可内部试用的业务前端版本**
- **阶段：已进入“功能整理 + 体验打磨”阶段**
- **阶段：仍未达到稳定生产版**

已经具备：

- 任务级卡片生成
- “总结与补充”独立入口
- AI 陪伴文案
- 历史详情页弹窗查看与滚动

- 记录保存时上传素材与陪伴文案

还没有完全解决：

- 卡片首次生成仍依赖历史详情页临时挂载 `checkin-card`

## 技术说明

### 页面与数据入口

任务卡片的数据入口主要来自：

- `draft.missionAssetMap`

每个任务项已经统一扩展为：

- `photoList`
- `videoList`
- `audioList`
- `noteText`
- `companionNote`
- `cardImagePath`

并通过 `createEmptyMissionAssets` 做默认初始化。

### 总结任务的数据同步

新增了一个特殊的总结任务键：

- `__summary__`

并通过：

- `syncDraftAggregates`

把总结任务的数据同步到页面顶层的：

- `draft.noteText`
- `draft.photoList`
- `draft.videoList`
- `draft.audioList`

所以“整次漫步总结”已经纳入和普通任务一致的资产模型。

### 卡片生成方式

仍采用：

- 小程序端原生 2D Canvas

而不是：

- Painter
- 后端拼图
- 云函数合成成品图

优点：

- 迭代快
- 样式直接改代码即可
- 不依赖额外图像服务

缺点：

- 真机差异明显
- 多图和长图对性能不友好
- 图片解析兼容性需要自己处理

### 陪伴文案生成方式

“小猫66 的记录”由服务层调用：

- `generateCompanionNote`

记录页打卡后先异步生成陪伴文案。历史详情页生成卡片时，会带着：

- 用户文案
- AI 陪伴文案
- 图片

一起生成卡片。

所以现在打卡卡片已经不只是前端排版，而是“记录页 AI 文案补齐 + 详情页前端排版出图”的组合能力。

## 遇到过的问题

### 1. 初版只显示第一张图片

这个问题已经修复过，但当前又做了模板收敛。

现在不是“无限多图纵向拼接”，而是：

- 最多展示两张图

这不是 bug，而是模板策略。

如果未来业务要求“所有图片都必须进卡片”，还要再设计新版版式。

### 2. 卡片过长导致弹窗内容超出屏幕

这个问题已通过弹窗滚动查看解决。

不过因为当前模板已经限制最多两张图，长图问题比上一版弱了一些。

### 3. 部分图片上传失败或无法绘制

这个问题仍然存在，只是已经有基础缓解措施。

风险主要来自：

- `heic/heif`
- 无后缀临时图
- 微信转存图
- 个别机型 Canvas 对图片格式的兼容差异

现在的方案是“尽量转换、尽量容错”，但还不能承诺生产级全兼容。

### 4. AI 陪伴文案存在风格不稳定风险

当前 `generateCompanionNote` 已接上，但这类文案天然存在：

- 风格波动
- 内容重复
- 偶发空结果
- 个别表述不够贴近任务语境

目前代码层已做失败降级，但内容质量层面还没有完全收稳。

### 5. 卡片结果已经开始持久化，但历史链路还不完整

生成结果会写入 `cardImagePath`，保存时也会上传。

但目前整体还没有形成完整的“历史查看卡片”产品链路，例如：

- 历史详情页统一展示卡片
- 卡片列表回放
- 卡片分享页

所以数据层已经前进了一步，展示层还没有完全跟上。

另外，近期已经暴露过一次典型链路问题：

- 记录页直接生成的卡片可以保存
- 历史详情页弹窗里的卡片保存失败

这个问题已经修复。根因是详情页原先把本地 `tempFilePath` 也当成网络地址再次 `downloadFile`，而记录页已经区分了本地路径和网络路径。当前两个页面的保存逻辑已经统一，但这也说明页面间逻辑漂移是当前阶段的真实风险。

## 建议进一步改进的地方

### 1. 明确“卡片展示全部图片”还是“模板限制前两张图”

实现和用户预期之间最容易产生误解的点，就是多图策略。

建议明确做一个产品决策：

- 路线 A：卡片永远只展示前两张图，强调版式稳定
- 路线 B：卡片展示全部图片，强调记录完整

现在代码偏向路线 A，但如果业务更在意“完整收纳”，就要重做版式。

### 2. 给 `companionNote` 建立质量控制

建议补：

- prompt 收敛
- 长度限制
- 风格校正
- 空结果处理
- 重复表达压制

否则 AI 陪伴文案会成为上线后最不稳定的一环。

### 3. 继续强化卡片结果的持久化链路

当前已经实现：

- `cardImagePath`
- 历史详情页正式展示卡片
- 查看与保存已保存卡片
- 首次生成后上传并回写历史记录

推荐下一步继续补：

1. 记录列表可选展示封面卡片或卡片摘要。
2. 为卡片生成失败提供单独重试入口。
3. 增加卡片模板版本号，避免后续样式升级影响历史卡片。
4. 若未来确有运营或用户需求，再设计“重新生成并覆盖旧卡片”的权限和确认流程。

同时建议继续统一“查看卡片”和“保存卡片”的来源，避免某些页面优先使用重绘结果，某些页面优先使用落库结果，减少页面差异导致的兼容问题。

### 4. 把卡片模板参数化

版式仍然大量写在 Canvas 绘图逻辑中。

建议把这些抽成配置：

- header 高度
- 气泡宽度
- 字体
- 装饰元素
- 配色
- 图片数量上限

这样之后才能更稳地扩展不同风格。

### 5. 继续完善图片标准化模块

图片标准化仍属于“轻量兼容层”。

建议未来升级成正式模块，覆盖：

1. 统一检测格式
2. 统一压缩策略
3. 统一缩放策略
4. 统一失败处理
5. 统一缓存和复用

### 6. 补齐完整的失败恢复流程

仍然缺少：

- AI 陪伴文案生成失败后的详细提示
- 卡片绘制失败后的重试入口
- 卡片上传失败后的单独恢复
- 保存中断后的资产一致性处理

这些都是上线前需要补齐的部分。

### 7. 建立真机专项回归清单

这块功能非常依赖真机表现，建议固定回归以下场景：

- iPhone 真机生成并保存卡片
- 安卓真机生成并保存卡片
- 历史详情页保存卡片
- 首次拒绝相册权限后再授权
- 本地 `tempFilePath` 保存
- `cloud://` 卡片保存
- `http/https` 卡片保存
- 多图任务卡片生成
- `heic/webp/无后缀图` 上传与绘制

## 上线前建议

建议分四步推进。

### 第一阶段：先稳定前端流程

优先建议先做这几件事：

- 把历史详情页的卡片查看与保存逻辑抽成统一工具，后续团队详情页也可复用
- 统一本地路径、云文件、远程 URL 三种图片来源的处理方式
- 固化相册权限失败、下载失败、文件失效三类错误提示
- 避免页面各自维护一套相似但不完全一致的保存逻辑

目标：

- 让当前版本在主流场景下尽可能稳定

建议动作：

1. 对 AI 陪伴文案增加超时与降级策略
2. 对图片兼容再做一轮真机验证
3. 对卡片导出失败增加可重试提示
4. 记录卡片生成失败日志

### 第二阶段：完成历史展示闭环

目标：

- 让卡片真正成为记录的一部分，而不只是现场生成效果

建议动作：

1. 继续稳定详情页 `cardImagePath` 展示、查看、保存链路。
2. 为“生成失败 / 上传失败 / 文件失效”补单独重试与提示。
3. 明确历史数据迁移策略。

### 第三阶段：整理模板与内容系统

目标：

- 让卡片功能长期可维护

建议动作：

1. 模板配置化
2. AI 陪伴文案 prompt 配置化
3. 装饰素材目录规范化
4. 不同卡片样式支持扩展

### 第四阶段：再决定是否切换 Painter 或服务端渲染

如果未来遇到这些情况：

- 模板数量快速增加
- 真机兼容问题反复出现
- 长图导出性能不稳定
- 分享场景明显增多

再评估：

- 路线 A：保留前端生成，模板层升级到 Painter
- 路线 B：把最终卡片改为服务端或云端生成

如果优先级是生产稳定性，最终通常还是路线 B 更稳。

## 总结

打卡卡片已经进入新的阶段。

它不再只是：

- 一张前端视觉 demo 图

而已经是：

- 带 AI 陪伴文案
- 带任务资产回写
- 带保存链路
- 带总结任务位
- 可跟随记录入库的前端实现版本

但它依然还不算正式生产版本。

离稳定上线最关键的差距，主要集中在：

1. 图片兼容性还不够彻底
2. AI 陪伴文案质量还需要稳定
3. 历史展示流程还没有完全闭环
4. 失败恢复、重试和监控能力还不足
5. 模板和数据结构还需要进一步标准化

如果下一步继续推进，最值得优先做的不是继续堆装饰，而是先把：

1. 内容稳定性
2. 图片稳定性
3. 历史数据闭环
4. 错误恢复
5. 模板配置化

这五件事补齐。这样打卡卡片才适合稳定上线。
