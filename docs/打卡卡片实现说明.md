# 打卡卡片实现说明

## 背景

当前项目已经在小程序记录页内补齐了“任务级打卡卡片”的前端 demo 能力。

这套能力的目标不是替代原有的普通图文记录，而是把某一个任务项下的：

- 文字记录
- 图片记录

整理成一张更适合预览、保存、分享的“纪念卡片”。

目前这条能力仍处于 **可用 demo / 可继续扩展** 阶段，还没有完全收敛到生产可长期维护的版本。

## 当前已经实现的功能

### 1. 记录页支持按任务生成打卡卡片

当前在记录页的任务展开面板中，已经把原来的 `AI核验并打分` 前端按钮替换为：

- `生成打卡卡片`

用户点击后，会针对当前任务项生成一张卡片。

相关入口：

- [record.wxml](D:/liuliu-minimap/miniprogram/pages/record/record.wxml)
- [mission-list.wxml](D:/liuliu-minimap/miniprogram/components/mission-list/mission-list.wxml)

### 2. 卡片采用前端 Canvas 生成

当前卡片不是后端拼图，也不是图片模板服务，而是由小程序前端 Canvas 直接绘制。

实现组件：

- [checkin-card.js](D:/liuliu-minimap/miniprogram/components/checkin-card/checkin-card.js)

当前绘制内容包括：

- 任务标题
- 任务文字记录
- 任务下的多张图片
- 纸张背景色
- 纸张纹理透明图
- 邮票/邮戳/条码等装饰元素
- 日期与底部信息

### 3. 卡片支持动态高度

当前卡片高度不是固定值，而是根据内容动态计算。

已实现的动态逻辑包括：

- 根据标题与正文行数计算文字区域高度
- 根据每张图片原始宽高比计算缩放后高度
- 多张图片时按顺序竖向排列
- 最终自动累加得到整张卡片高度

这意味着：

- 一张图和三张图的卡片高度不同
- 简短文案和长文案的卡片高度不同

### 4. 当前卡片只嵌入文字和图片

目前已经明确限制卡片内容只包含：

- `noteText`
- `photoList`

不会把以下内容画入卡片：

- `videoList`
- `audioList`

这部分规则已经在当前实现中固定下来，避免出现卡片内容混乱。

### 5. 多图任务已经支持整合到一张卡片

最初 demo 只使用任务的第一张图，后来已经改为：

- 遍历任务下全部图片
- 按顺序依次绘制
- 在卡片中继续往下排列

所以现在同一个任务下上传多张图，卡片会包含全部图片，而不是只显示第一张。

### 6. 生成结果改为弹窗查看

当前卡片不会直接嵌在任务列表里常驻展示，而是改成：

1. 点击 `生成打卡卡片`
2. 弹出卡片弹窗
3. 卡片生成完成后在弹窗中展示
4. 用户可以保存到本地

这比直接把长卡片塞进任务面板更适合当前页面结构。

相关逻辑：

- [record.js](D:/liuliu-minimap/miniprogram/pages/record/record.js)
- [record.wxml](D:/liuliu-minimap/miniprogram/pages/record/record.wxml)
- [record.wxss](D:/liuliu-minimap/miniprogram/pages/record/record.wxss)

### 7. 弹窗已支持滚动查看

由于多图卡片会很长，当前已经把弹窗内容区改成可滚动。

当前体验是：

- 上方展示卡片
- 下方固定操作按钮
- 用户可以在弹窗内上下滑动查看整张卡片

### 8. 支持保存到本地相册

当前弹窗内已经提供：

- `保存到本地`

并处理了相册权限不足时的引导逻辑。

### 9. 已补一层图片格式兼容处理

当前已经对图片链路补了一层基础兼容，覆盖：

- 选图
- 预览
- 上传
- Canvas 绘制

做法包括：

- 选图优先取压缩图
- 对部分不稳定格式尽量转成更兼容的临时 JPEG
- 上传时推断扩展名，避免无后缀路径导致异常
- Canvas 绘制失败时再尝试一次兼容转换

相关文件：

- [media.js](D:/liuliu-minimap/miniprogram/utils/media.js)
- [api.js](D:/liuliu-minimap/miniprogram/services/api.js)
- [checkin-card.js](D:/liuliu-minimap/miniprogram/components/checkin-card/checkin-card.js)

## 当前实现到了什么程度

如果按阶段判断，当前更适合定义为：

- **阶段：高保真前端 demo 已完成**
- **阶段：可内部试用，但未达到稳健生产版本**

当前已经具备的能力：

- 能从任务素材生成卡片
- 能生成多图卡片
- 能弹窗查看
- 能保存到本地
- 有基础图片兼容层
- 有基础视觉风格

当前还没有完全完成的部分：

- 没有把卡片文件作为正式业务数据持久化保存
- 没有统一的卡片模板配置系统
- 没有完整的失败重试与兜底提示体系
- 没有做足够的真机兼容验证
- 没有覆盖复杂图片格式的完整生产级方案
- 没有沉淀成可复用的服务端/前端稳定模块

## 当前技术说明

## 页面与数据入口

当前任务卡片的数据入口来自记录页草稿对象中的：

- `draft.missionAssetMap`

其结构中每个任务项通常包含：

- `photoList`
- `videoList`
- `audioList`
- `noteText`

卡片当前只消费：

- `photoList`
- `noteText`

## 生成方式

当前采用的是：

- 小程序端原生 2D Canvas

而不是：

- 后端拼图
- 云函数图片合成
- Painter 第三方组件

这意味着当前方案的优点是：

- 开发速度快
- 修改样式方便
- 不依赖额外服务

缺点是：

- 端能力差异大
- 图片兼容问题需要自己兜底
- 大图、多图、长图更容易遇到性能问题

## 视觉素材

当前卡片视觉使用了本地资源：

## 当前遇到过的问题

### 1. 初版只显示第一张图片

最初卡片组件只取了任务的第一张图，导致多图任务生成出来不完整。

目前已修复为：

- 读取全部图片
- 顺序绘制
- 动态拉长卡片

### 2. 卡片过长导致弹窗内容超出屏幕

当任务图片较多时，卡片高度很容易超过屏幕。

目前已修复为：

- 卡片弹窗内容区可滚动

### 3. 某些图片上传失败或无法绘制

这是当前最典型的兼容问题，通常出现在：

- `heic/heif`
- 一些转发保存图
- 无明显后缀的临时图
- Canvas 对个别格式解析不一致

当前已经补了基础标准化层，但只能说“兼容性增强”，还不能说“所有图片格式都稳定支持”。

### 4. 贴纸能力与任务卡片能力并存，容易让页面主流程分散

当前项目里同时存在：

- AI 打卡贴纸
- AI 任务核验
- 任务卡片

如果都同时展示，记录页会很重，用户也不容易理解主操作路径。

所以当前策略是：

- 先保留逻辑
- 隐藏贴纸与 AI 核验前端展示
- 集中推进任务卡片链路

## 建议进一步改进的地方

### 1. 建立正式的图片标准化模块

当前只是加了轻量兼容层，还不算真正生产级。

建议后续补成一套统一图片标准化流程：

1. 选图后统一识别格式
2. 对不稳定格式统一转 JPEG 或 PNG
3. 超大图片统一缩放
4. 产出标准化临时文件
5. 再进入预览、上传、绘制

这样可以减少同一张图在不同链路里重复兜底。

### 2. 把卡片生成结果持久化

当前卡片更像即时生成的临时结果。

建议后续支持以下两种策略之一：

- 保存卡片成品图到云存储，并把地址写入记录
- 保存卡片描述数据，在详情页按需重新渲染

如果希望更适合分享与历史查看，建议优先：

- 保存成品图

### 3. 把模板参数化

当前卡片样式大量写死在 Canvas 逻辑里。

建议拆成模板配置，例如：

- 边距
- 字号
- 颜色
- 装饰素材
- 图片圆角
- 标题位置

这样后面就能做：

- 多套模板
- 节日主题
- 城市场景主题

### 4. 评估是否切换到 Painter

当前实现是原生 Canvas。

如果后续你希望：

- 更容易排版
- 模板结构更清晰
- 更少手写绘图代码

可以评估接入 Painter。

但需要注意：

- Painter 并不会自动解决所有图片兼容问题
- 只是让模板描述更结构化
- 仍需处理图片下载、缓存、绘制稳定性

所以建议是：

- 当前 demo 可继续用原生 Canvas推进
- 等模板结构稳定后，再决定是否迁移到 Painter

### 5. 增加失败态和重试机制

目前生成失败时的体验还不够完整。

建议补齐：

- 图片解析失败提示
- 某一张图片跳过提示
- 一键重试
- 生成中状态锁定
- 卡片导出失败提示

### 6. 做真机专项测试

当前最需要的不是继续堆功能，而是做兼容验证。

建议至少覆盖：

- iPhone 相册原图
- iPhone 实况/HEIC 来源图
- 安卓不同品牌相册
- 微信聊天保存图
- 截图
- WebP 图
- 多图长卡片
- 超长文字

## 后续如何修改，才能更稳健适用于生产环境

建议按下面顺序推进。

### 第一阶段：把当前 demo 收稳

目标：

- 先让当前版本在常见场景下稳定可用

建议动作：

1. 给图片选取增加更明确的格式和大小提示
2. 对所有待上传图片统一做标准化
3. 对卡片生成增加超时、失败、重试
4. 做 10 到 20 组真机样例验证

### 第二阶段：把卡片结果纳入正式数据结构

目标：

- 让卡片成为记录的一部分，而不是单次临时效果

建议动作：

1. 新增 `missionCardList` 或 `missionCardMap`
2. 记录每个任务卡片的图片地址或生成参数
3. 详情页支持稳定查看
4. 后续再考虑分享页或海报页

### 第三阶段：沉淀模板系统

目标：

- 让这套卡片能力能够长期演进

建议动作：

1. 把卡片布局参数独立成配置
2. 把素材文件目录规范化
3. 把装饰元素抽象成模板变量
4. 允许不同任务/主题使用不同模板

### 第四阶段：再决定是否引入 Painter 或服务端渲染

如果未来诉求是：

- 模板很多
- 分享很多
- 生成频率很高
- 真机绘制性能不稳定

可以再评估两条路线：

- 路线 A：继续前端绘制，但模板层升级为 Painter
- 路线 B：后端或云函数生成成品图，前端只展示

如果目标是生产稳健性优先，通常路线 B 更稳定。

## 总结

当前打卡卡片能力已经完成了一个比较完整的前端 demo：

- 可以按任务生成
- 可以整合文字与多张图片
- 可以弹窗查看
- 可以保存到本地
- 有基础图片兼容能力

但它目前仍然更适合定义为：

- **可继续推进的内部试用版**

而不是：

- **已经稳定适用于正式生产环境的最终版**

如果下一步要继续做，最值得优先投入的不是继续加新视觉，而是：

1. 图片标准化
2. 错误恢复
3. 真机兼容测试
4. 卡片结果持久化
5. 模板配置化

把这五件事补齐之后，这套能力才更适合走向长期维护和生产落地。
